### F-PRF-001 — `framer-motion` importé statiquement dans tout le bundle client, y compris les composants de layout globaux

- **Axe** : PRF
- **Sévérité** : Medium
- **Description** : `framer-motion` (librairie d'animation lourde) est importé de façon statique dans **32 modules client** (`grep -rl 'from "framer-motion"' src` → 32 fichiers). Plusieurs de ces modules sont des **composants de layout présents sur quasiment toutes les pages authentifiées** (`sidebar-list-item.tsx`, `sidebar-folder-item.tsx`, `sidebar-space-item.tsx`, `mobile-nav.tsx`, `breadcrumbs.tsx`, `account-switcher.tsx`, `quick-action-fab.tsx`). La librairie **n'est pas listée** dans `experimental.optimizePackageImports` de `next.config.mjs` et n'est jamais chargée via `next/dynamic`, ni via l'API allégée `LazyMotion`/`m`. Le runtime d'animation complet est donc inclus dans le bundle partagé chargé au premier rendu.
- **Preuve** : `src/components/layout/mobile-nav.tsx:4`, `src/components/layout/sidebar-list-item.tsx:3`, `src/components/layout/breadcrumbs.tsx:4`, `src/components/layout/account-switcher.tsx:3`, `src/components/layout/quick-action-fab.tsx:4` ; `next.config.mjs:13` (`optimizePackageImports` sans `framer-motion`) ; `commande: grep -rl 'from "framer-motion"' src --include='*.tsx' --include='*.ts' | wc -l` → `32`.
- **Impact** : alourdissement du bundle JavaScript de premier chargement sur l'ensemble des pages de la plateforme (le layout est commun), au détriment du Time-to-Interactive, particulièrement sensible sur mobile (cible principale Capacitor) et sur réseau lent.
- **Recommandation** : adopter l'API allégée de `framer-motion` (`LazyMotion` + composant `m` avec `domAnimation`) pour ne charger que les fonctionnalités utilisées ; et/ou réserver `motion` aux frontières réellement animées en chargeant les composants concernés via `next/dynamic`. Mesurer le gain avec `ANALYZE=true npm run build`. Ne pas implémenter pendant l'audit.

### F-PRF-003 — Primitives de présentation pures (`Card`, `Badge`, `Skeleton`) marquées `"use client"` et largement réimportées

- **Axe** : PRF
- **Sévérité** : Medium
- **Description** : Trois primitives de la couche `src/components/ui/` portent la directive `"use client"` alors qu'elles sont **provablement pures** : aucun hook React, aucun gestionnaire d'événement, aucune API navigateur, aucune primitive Radix cliente. `Card`/`CardHeader`/`CardContent`… ne rendent que des `<div>`/`<h3>`/`<p>` stylés ; `Badge` ne rend qu'un `<div>` ; `Skeleton` ne rend qu'un `<div className="animate-pulse">`. Dans la distribution canonique de shadcn/ui, ces trois fichiers **ne portent pas** `"use client"` (à la différence des primitives Radix comme `dialog`, `select`, `tabs` qui en ont légitimement besoin). Marquées client, elles forcent l'inclusion de leur code dans le bundle client de **tous leurs consommateurs** et — surtout — empêchent ces consommateurs d'être rendus côté serveur dès lors qu'ils n'importent que ces primitives. Distinct de `F-LOG-007` qui visait quatre composants *applicatifs* (`category-badge`, `budget-alert`, `budget-skeleton`, `list-view-custom-fields`), non les primitives de base.
- **Preuve** : `src/components/ui/card.tsx:1` ; `src/components/ui/badge.tsx:1` ; `src/components/ui/skeleton.tsx:1` ; portée d'import — `commande: grep -rln 'from "@/components/ui/card"' src --include='*.tsx' | wc -l` → `26` ; `commande: grep -rln 'from "@/components/ui/badge"' src --include='*.tsx' | wc -l` → `12` ; `commande: grep -rln 'from "@/components/ui/skeleton"' src --include='*.tsx' | wc -l` → `20`.
- **Impact** : code JavaScript de présentation embarqué inutilement dans le bundle client sur l'ensemble des écrans qui consomment ces primitives (cartes de dashboard, badges de statut/priorité, squelettes de chargement). Plus pénalisant encore : un composant parent qui n'utilise que `Card`/`Badge`/`Skeleton` est contraint de basculer côté client par contagion, ce qui contrarie la stratégie « Server Components par défaut » et alourdit le premier chargement, notamment sur la cible mobile Capacitor.
- **Recommandation** : retirer la directive `"use client"` de `card.tsx`, `badge.tsx` et `skeleton.tsx` (les aligner sur la distribution canonique shadcn/ui — ces composants restent utilisables tels quels dans des arbres client). Vérifier ensuite, via `@next/bundle-analyzer`, la réduction du JS partagé. Ne pas implémenter pendant l'audit.

### F-PRF-004 — Frontière client remontée au niveau `page.tsx` sur quasiment tout le groupe `(platform)` (22/24 pages client)

- **Axe** : PRF
- **Sévérité** : Medium
- **Description** : 22 des 24 pages `page.tsx` du groupe authentifié `(platform)` sont déclarées `"use client"`, dont plusieurs pages volumineuses : `dashboard/calendar-settings/page.tsx` (645 lignes), `calendar/page.tsx` (610), `notes/page.tsx` (450), `goals/page.tsx` (319), `search/page.tsx` (313), `time-tracking/page.tsx` (303), `settings/page.tsx` (274), `finance/page.tsx` (263), `my-tasks/page.tsx` (262), `dashboard/page.tsx` (260). Mesurer le First Load JS par route avec `@next/bundle-analyzer` avant/après. Traiter conjointement avec `F-LOG-008`/`F-LOG-009` (même cause racine). Ne pas implémenter pendant l'audit.

### F-PRF-005 — Avatars utilisateurs servis en pleine résolution sans optimisation ni conversion de format (`<AvatarImage>` = `<img>` brut, `images.unoptimized: true` non justifié par un export statique)

- **Axe** : PRF
- **Sévérité** : Medium
- **Description** : L'intégralité des images de contenu de l'application transite par la primitive Radix `<AvatarImage>` (21 sites d'usage), qui rend un `<img>` natif **sans redimensionnement serveur, sans conversion en format moderne (WebP/AVIF) et sans `srcset` responsive**. Aucun `next/image` / `<Image>` n'est utilisé nulle part dans `src/` (`grep` → 0 occurrence). Les sources d'avatars sont soit des **uploads locaux** stockés tels quels dans `public/uploads/avatars/` (jusqu'à **2 Mo**, pleine résolution, types JPEG/PNG/WebP/GIF — cf. `route.ts`), soit des images **distantes** (par ex. `user.image` Google OAuth, autorisées par `remotePatterns: [{ hostname: "**" }]`). Elles sont affichées dans des conteneurs de **24 à 64 px** (`h-6 w-6` … `h-16 w-16`). Par ailleurs, l'optimisation d'image Next est **désactivée globalement** (`images.unoptimized: true`) : comme documenté ci-dessus, cette désactivation **n'est pas justifiée** ici par un export statique (le build cible `output: "standalone"` et l'app Capacitor charge l'URL distante), donc l'API d'optimisation Vercel serait disponible mais reste inexploitée.
- **Preuve** : `next.config.mjs:8` (`unoptimized: true`), `next.config.mjs:5` (`output: "standalone"`), `next.config.mjs:9-10` (`remotePatterns` `hostname: "**"`) ; `src/components/ui/avatar.tsx:27` (`AvatarPrimitive.Image` → `<img>` natif) ; `src/app/api/user/avatar/route.ts:8` (`MAX_FILE_SIZE = 2 * 1024 * 1024`, aucune étape de redimensionnement/compression au stockage) ; `capacitor.config.ts` (`webDir: "public"`, `server.url` distant — pas d'export statique) ; `commande: grep -rn '<AvatarImage' src/ --include='*.tsx' | wc -l` → `21` ; `commande: grep -rn 'next/image' src/ --include='*.tsx' --include='*.ts'` → exit `1` (aucune occurrence).
- **Impact** : un avatar de 2 Mo en pleine résolution est téléchargé intégralement pour être affiché dans un cercle de 24–64 px, gaspillant la bande passante à chaque liste de membres, d'assignés ou d'entrées de temps (vues à forte densité d'avatars). Aucune conversion WebP/AVIF ni redimensionnement côté serveur n'allège la charge. Coût particulièrement sensible sur la cible mobile Capacitor (WebView sur réseau cellulaire) et sur connexion lente.
- **Recommandation** : au choix — (a) redimensionner et compresser les avatars au moment de l'upload dans `/api/user/avatar` (par ex. `sharp` vers une largeur max ~128 px en WebP) ; et/ou (b) activer l'optimisation Next (`unoptimized: false`) et servir les avatars via `next/image` avec `width`/`height` explicites, en documentant que le projet n'utilise pas `output: 'export'` (donc l'optimisation Vercel standalone est disponible). Conserver `unoptimized: true` uniquement si une migration vers `output: 'export'` est ultérieurement décidée. Ne pas implémenter pendant l'audit.

### F-PRF-006 — Recherche plein texte non-sargable : `contains` (sans index trigramme/GIN) provoque un balayage séquentiel `ILIKE '%q%'` quel que soit le `take:`

- **Axe** : PRF
- **Sévérité** : Medium
- **Description** : La recherche globale (`GET /api/search`, `src/app/api/search/route.ts`) et le filtre de recherche du board (`GET /api/tasks`, `src/app/api/tasks/route.ts`) reposent sur des prédicats `{ contains: q }` (`LIKE '%q%'` côté Postgres). Ce patron est **non-sargable** : aucun index B-tree classique ne peut servir une recherche par sous-chaîne avec joker de tête (`%q%`), et le schéma `prisma/schema.prisma` ne déclare **aucun index trigramme (`pg_trgm`/GIN) ni full-text (`tsvector`)** (provider `postgresql`, `previewFeatures = ["driverAdapters"]` sans `fullTextSearch` — `prisma/schema.prisma:5-14`). Conséquence : chaque recherche déclenche un **`Seq Scan` complet** des tables ciblées, suivi d'un filtre ligne à ligne. La route `/api/search` exécute **trois** de ces scans (sur `Task` lignes 44-45, `List` ligne 65, `Space` lignes 80-81), chacun joint à un filtre workspace transverse (`list.space.workspaceId`), et **chaque caractère tapé** déclenche la requête (le hook `useDebounce(query, 300)` de `src/components/search/search-command.tsx:51` et `src/app/(platform)/search/page.tsx:68` ne fait que lisser à 300 ms, mais une frappe continue produit tout de même une rafale de requêtes). Le `take: 20`/`take: 10` présent sur ces `findMany` **ne réduit pas** le coût du scan : le `LIMIT` SQL ne s'applique qu'**après** le balayage et le tri `orderBy: { updatedAt }` ; sur une table `Task` à forte cardinalité, le moteur lit toute la table avant de tronquer. À noter que `/api/budget/[budgetId]/transactions/route.ts:67-68` utilise `mode: "insensitive"`, ce qui **aggrave** encore le cas (le `LOWER()` implicite empêche tout index fonctionnel non prévu à cet effet), alors que `/api/search` et `/api/tasks` omettent `mode` (recherche sensible à la casse — incohérence fonctionnelle, hors périmètre PRF).
- **Preuve** : `src/app/api/search/route.ts:44-45` (`title`/`description` `contains` sur `Task`), `:65` (`name` `contains` sur `List`), `:80-81` (`name`/`description` `contains` sur `Space`) ; `src/app/api/tasks/route.ts:43-44` (`contains` sur board) ; `src/components/search/search-command.tsx:51,79` (requête déclenchée par frappe, debounce 300 ms) ; `prisma/schema.prisma:5-14` (provider postgresql sans `fullTextSearch` ni index trigramme/GIN) ; `commande: grep -rn "contains:" src/app/api --include="*.ts"` (7 sites, consignée en annexe A tâche 12.4). Recoupement : distinct de `F-BDD-031` (qui traite le `take:` manquant) — ici le `take:` est **présent** mais inopérant face au `Seq Scan` de sous-chaîne.
- **Impact** : à mesure que les tables `Task`/`List`/`Space` grossissent, la latence de la recherche globale croît linéairement avec le volume (chaque scan est O(n)). Sur un workspace mature (plusieurs dizaines de milliers de tâches), une recherche multi-mots devient perceptiblement lente, et la rafale de requêtes par frappe (même débouncée) multiplie la charge Postgres/Neon — poste de coût compute non borné et dégradation UX directe sur une fonctionnalité centrale (Cmd+K). Le `mode: "insensitive"` sur les transactions budgétaires accentue le coût unitaire.
- **Recommandation** : (i) activer l'extension `pg_trgm` et créer des index GIN trigramme sur les colonnes recherchées (`Task.title`, `Task.description`, `List.name`, `Space.name`) pour rendre `ILIKE '%q%'` sargable ; ou (ii) adopter la recherche full-text Postgres (`tsvector` + index GIN) via le preview feature Prisma `fullTextSearch`, plus adapté à une recherche multi-mots ; (iii) homogénéiser `mode: "insensitive"` sur tous les sites de recherche une fois l'index trigramme en place. Mesurer via `EXPLAIN ANALYZE` avant/après. Ne pas implémenter pendant l'audit.

### F-PRF-007 — `GET /api/finance/stats` : requêtes indépendantes exécutées en cascade séquentielle (waterfall) + agrégations monétaires en JavaScript au lieu de `groupBy`/`aggregate` SQL

- **Axe** : PRF
- **Sévérité** : Medium
- **Description** : La route `GET /api/finance/stats` (`src/app/api/finance/stats/route.ts`) calcule le tableau de bord financier d'un workspace en enchaînant **cinq requêtes Prisma strictement séquentielles** via des `await` successifs, alors qu'elles sont **mutuellement indépendantes** et pourraient être parallélisées (`Promise.all`) : `financeAccount.findMany` (`:21`), `financeTransaction.findMany` (`:27`), `financeTransaction.groupBy` (`:43`), `financeCategory.findMany` (`:54`), `financeGoal.findMany` (`:64`). Chaque `await` ajoute son temps de round-trip réseau Postgres/Neon au précédent : la latence totale est la **somme** des cinq latences au lieu du **maximum**. Deux problèmes de performance s'y cumulent : (1) **agrégation monétaire en JavaScript** — `monthlyTransactions = financeTransaction.findMany(...)` (`:27`, **sans `select`**, donc toutes les colonnes hydratées) rapatrie **toutes les transactions du mois** en mémoire serveur uniquement pour calculer `monthlyIncome`/`monthlyExpense` par `.filter().reduce()` côté JS (`:35-41`) ; le même résultat s'obtient en une requête `groupBy({ by: ["type"], _sum: { amount: true } })` côté Postgres, sans rapatrier la moindre ligne brute. Le code prouve d'ailleurs maîtriser le patron correct, puisque `expenseByCategory` (`:43`) utilise bien un `groupBy` `_sum`. (2) **`totalBalance` calculé en JS** — `accounts.reduce((sum, a) => sum + a.balance, 0)` (`:25`) au lieu d'un `aggregate({ _sum: { balance: true } })`. La route **n'expose aucun en-tête `Cache-Control`** (contrairement à `/api/dashboard/stats:178`), donc chaque montage de la page Finance recalcule tout.
- **Preuve** : `src/app/api/finance/stats/route.ts:21` (findMany accounts), `:25` (`reduce` totalBalance en JS), `:27-33` (findMany transactions du mois sans `select`), `:35-41` (filter+reduce income/expense en JS), `:43` (groupBy — patron correct), `:54` (findMany categories), `:64` (findMany goals), `:68-69` (reduce goals en JS) — les cinq accès BDD sont chaînés par `await` sans `Promise.all`. Recoupement : à rapprocher de `F-BDD-029` (même pathologie « agrégation JS au lieu de SQL » sur `/api/budget/stats`), mais l'angle **waterfall séquentiel** (sérialisation de requêtes indépendantes) est **propre à cette route** et non relevé en BDD ; le contre-exemple `dashboard/stats` (parallélisé) confirme l'écart.
- **Impact** : latence de l'écran Finance dégradée par construction (somme de 5 round-trips réseau au lieu d'un seul lot parallèle), aggravée par l'hydratation inutile de toutes les transactions du mois (mémoire serveur + bande passante interne Neon→Vercel) pour un calcul réductible à un agrégat SQL. À l'échelle (workspace à fort volume de transactions, ou plusieurs utilisateurs concurrents), le coût compute et le risque de latence P95 augmentent. L'absence de cache HTTP fait répéter ce coût à chaque navigation vers `/finance`.
- **Recommandation** : (i) paralléliser les requêtes indépendantes via `Promise.all` (sur le modèle de `dashboard/stats:51`) ; (ii) remplacer `financeTransaction.findMany` + `filter/reduce` (`:27-41`) par un `groupBy({ by: ["type"], where: { ...mois }, _sum: { amount: true } })` unique ; (iii) remplacer `accounts.reduce` (`:25`) par `financeAccount.aggregate({ _sum: { balance: true } })` (ou conserver le `findMany` si la liste des comptes est par ailleurs nécessaire au rendu) ; (iv) ajouter un en-tête `Cache-Control: private, max-age=30, stale-while-revalidate=300` comme sur `/api/dashboard/stats`. Mesurer le gain de latence via les `Server-Timing` ou `EXPLAIN ANALYZE`. Ne pas implémenter pendant l'audit.

### F-PRF-008 — Stratégie de cache SWR uniforme : les données de référence peu volatiles (statuses, membres, catégories, tags, champs personnalisés) sont revalidées comme des données volatiles (`dedupingInterval` 2 s non relevé + `revalidateOnFocus`/`revalidateOnReconnect` hérités à `true`)

- **Axe** : PRF
- **Sévérité** : Medium
- **Description** : Le provider global `SWRProvider` (`src/lib/swr-config.tsx:36-41`) applique une configuration **unique et non hiérarchisée par volatilité** à l'ensemble des 59 fetchers : `dedupingInterval: 2000` (2 s — la valeur par défaut de SWR), `revalidateOnFocus: true`, `revalidateOnReconnect: true`. Or une part importante des fetchers cible des **données de référence quasi-immuables** à l'échelle d'une session : statuts d'une liste, membres d'un workspace, catégories financières, tags, définitions de champs personnalisés, listes. Pour ces ressources, **aucun hook ne relève `dedupingInterval`** ni ne déroge à `revalidateOnFocus`/`revalidateOnReconnect` (seul `DashboardPage` surcharge `dedupingInterval: 10000`, cf. annexe D #50). Cas le plus marquant : la clé `/api/lists/${listId}/statuses` est **souscrite par 6 emplacements `useSWR` distincts** (`board-view.tsx:38`, `list-view.tsx:39`, `task-row.tsx:42`, `status-badge.tsx:34`, `status-manager.tsx:81`, `status-filter.tsx:34`) ; en vue liste, `task-row.tsx:42` souscrit cette clé **pour chaque ligne de tâche rendue**. SWR déduplique correctement par clé (1 fetch réseau partagé tant que la fenêtre de 2 s est respectée), mais avec `dedupingInterval` à 2 s et `revalidateOnFocus`/`revalidateOnReconnect` à `true`, **chaque retour de focus d'onglet et chaque reconnexion réseau déclenche une revalidation réseau de cette clé** (et de toutes les clés de référence montées), alors que ces données ne changent quasiment jamais en cours de session. Distinct de `F-LOG-014`, qui ne traite que les 5 hooks à `refreshInterval` explicite (objectif : éviter le doublon poll + focus) ; le présent Finding vise les ressources de référence **non pollées**, jamais tunées.
- **Preuve** : `src/lib/swr-config.tsx:37` (`dedupingInterval: 2000`), `:39` (`revalidateOnFocus: true`), `:40` (`revalidateOnReconnect: true`) ; réutilisation de la clé statuses — `src/components/views/board-view.tsx:38`, `src/components/views/list-view.tsx:39`, `src/components/task/task-row.tsx:42`, `src/components/task/status-badge.tsx:34`, `src/components/list/status-manager.tsx:81`, `src/components/filters/status-filter.tsx:34` ; `commande: grep -rn 'statuses' src --include='*.tsx' | grep '/api/lists/'` (6 consommateurs `useSWR`, consignée en annexe A) ; absence de surcharge `dedupingInterval`/`revalidateOnFocus` sur ces hooks (annexe D, colonne « Options non par défaut » = `—`).
- **Impact** : revalidations réseau répétées et inutiles de données quasi-statiques à chaque changement de focus et chaque reconnexion. Sur la cible mobile Capacitor (WebView), les événements de focus/reconnexion sont fréquents (bascule d'application, notification push, retour de veille, réseau cellulaire instable) : chaque retour sur l'app relance un lot de requêtes `/api/lists/{listId}/statuses`, `/api/workspaces/{id}/members`, `/api/finance/categories`, `/api/tags`, etc., consommant bande passante, batterie et temps de fonction serverless Vercel, sans bénéfice de fraîcheur réel. La fenêtre de déduplication de 2 s est par ailleurs trop courte pour amortir les montages/démontages successifs de ces ressources lors de la navigation entre vues d'une même liste.
- **Recommandation** : introduire une **hiérarchisation du cache par volatilité** plutôt qu'une configuration globale unique : (i) pour les ressources de référence peu volatiles, créer un hook/option partagé avec un `dedupingInterval` nettement supérieur (par ex. 30–60 s) et `revalidateOnFocus: false` / `revalidateOnReconnect: false` (la donnée est rafraîchie explicitement via `mutate` lors des mutations, ce que fait déjà `status-manager`) ; (ii) centraliser la clé `/api/lists/${listId}/statuses` dans un hook dédié (`useStatuses(listId)`) portant ces options, au lieu de la réécrire dans 6 composants ; (iii) conserver `revalidateOnFocus: true` uniquement pour les données réellement volatiles (tâches, notifications). Mesurer la réduction du nombre de requêtes au retour de focus via l'onglet Réseau / les logs Vercel avant/après. Ne pas implémenter pendant l'audit.
