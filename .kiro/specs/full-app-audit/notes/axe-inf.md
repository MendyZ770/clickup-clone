## Tâche 9.1 — Audit de `vercel.json` (`functions`, `regions`, `headers`, `rewrites`)

Périmètre inspecté : le fichier `vercel.json` à la racine du dépôt (`/Users/zerbib/clickup-clone/vercel.json`, 22 lignes), confronté aux inventaires `notes/inv-crons.md` (jobs cron et endpoints cibles) et `notes/annexe-C-routes-api.md` (96 routes API). Les sections explicitement contrôlées sont `buildCommand`, `framework`, `crons`, `functions`, `regions`, `headers`, `rewrites`. La revue plus large des en-têtes de sécurité (CSP, HSTS, X-Frame-Options, X-Content-Type-Options) relève de la tâche 11.5 (axe SEC) et n'est pas re-instruite ici. ### F-INF-001 — Cron handlers sans `maxDuration` ni `memory` explicites dans `vercel.json`

- **Axe** : INF
- **Sévérité** : Medium
- **Description** : Le fichier `vercel.json` ne déclare aucune section `functions`. Les quatre cron handlers planifiés (`/api/cron/reminders`, `/api/cron/daily-tasks`, `/api/cron/upcoming-deadlines`, `/api/cron/budget-alerts`) effectuent un `prisma.findMany` initial puis itèrent en série sur les résultats pour réaliser, par itération, un `prisma.notification.findFirst` anti-doublon, des appels `sendNotificationToUsers` (push FCM + Web Push), et pour `reminders` des envois Resend `sendReminderEmail`. Aucun `maxDuration` ni `memory` n'est déclaré côté `vercel.json`, et les handlers n'exportent pas non plus de constante `export const maxDuration` au niveau des Route Handlers Next.js (cf. `src/app/api/cron/reminders/route.ts`, `src/app/api/cron/budget-alerts/route.ts`, `src/app/api/cron/daily-tasks/route.ts`, `src/app/api/cron/upcoming-deadlines/route.ts`). Les handlers s'exécuteront donc avec les valeurs par défaut Vercel (typiquement 10 s sur le plan Hobby, 15 s sur Pro), insuffisantes dès que le volume métier (rappels, budgets, tâches dues) croît au-delà de quelques dizaines d'éléments.
- **Preuve** : `vercel.json:1-22` (absence du champ `functions`) ; cf. boucles `for ... of reminders` dans `src/app/api/cron/reminders/route.ts:34-78` et `for (const budget of budgets)` dans `src/app/api/cron/budget-alerts/route.ts:25-72`. Aucune occurrence de `export const maxDuration` dans les quatre fichiers `route.ts` sous `src/app/api/cron/`.
- **Impact** : Risque de timeouts en production sur les jobs cron lorsque la base atteint un volume nominal (centaines d'utilisateurs, milliers de tâches/budgets). Un timeout interrompt la boucle d'envoi à mi-parcours sans reprise (les handlers ne disposent pas de mécanisme d'idempotence par batch ni de file d'attente). Conséquence métier : rappels manqués, alertes budgétaires non envoyées, deadlines à venir non notifiées, sans alerte explicite côté observabilité (logs Vercel uniquement).
- **Recommandation** : Déclarer une section `functions` dans `vercel.json` ciblant les paths `src/app/api/cron/**/route.ts` (par ex. `"functions": { "src/app/api/cron/**/route.ts": { "maxDuration": 60, "memory": 512 } }`), ou exporter `export const maxDuration = 60` (et `export const dynamic = "force-dynamic"`) dans chaque Route Handler de cron. Confirmer en consultant la console Vercel (External_Console, lecture seule) le plan effectif et la limite supérieure de `maxDuration` autorisée pour ce plan, et ajuster en conséquence.

### F-INF-002 — Aucune `regions` déclarée : risque de désalignement avec la région de la base Neon Postgres

- **Axe** : INF
- **Sévérité** : Medium
- **Description** : Le fichier `vercel.json` ne déclare pas le champ `regions`. Les fonctions serverless (Route Handlers, cron handlers) sont donc déployées dans la région par défaut du projet Vercel (généralement `iad1` — Washington D.C.). L'application accède à une base Postgres hébergée sur Neon via `DATABASE_URL` (`process.env.DATABASE_URL`, cf. `notes/inv-env.md`). Si l'instance Neon n'est pas située dans la même région que les fonctions Vercel, chaque appel Prisma ajoute un aller-retour réseau inter-régions, multiplié par le nombre de requêtes par requête HTTP. Or de nombreuses routes API exécutent plusieurs requêtes successives : `/api/dashboard/stats` lance huit requêtes en `Promise.all` plus un `groupBy` (cf. `notes/annexe-C-routes-api.md`), `/api/favorites` GET réalise un N+1 par favorite, et les cron handlers chaînent un `findMany` puis un `findFirst` par itération. La région effective de l'instance Neon ne peut être lue que via External_Console (Neon).
- **Preuve** : `vercel.json:1-22` (absence du champ `regions`) ; usage de `process.env.DATABASE_URL` documenté dans `notes/inv-env.md` ; routes BDD-intensives cataloguées dans `notes/annexe-C-routes-api.md` (`/api/dashboard/stats`, `/api/favorites`, cron handlers).
- **Impact** : Si la région Vercel diffère de la région Neon, la latence par requête HTTP augmente proportionnellement au nombre de requêtes BDD enchaînées, dégradant les temps de réponse perçus par l'utilisateur (en particulier sur le tableau de bord) et augmentant le coût en compute. Effet amplifié pour les Route Handlers serverless qui paient un cold start par invocation. À confirmer via console Vercel/Neon (External_Console).
- **Recommandation** : Récupérer en lecture seule la région de l'instance Neon depuis la console Neon (External_Console). Si elle diffère de la région par défaut Vercel, déclarer dans `vercel.json` `"regions": ["<region>"]` (single-region, alignée avec Neon) — un déploiement single-region est ici suffisant et évite la complexité d'un schéma multi-régions sans réplica de lecture côté Neon. Documenter le choix de région dans le `README` ou `CLAUDE.md` pour éviter une dérive lors d'un futur déploiement.


## Tâche 9.2 — Audit des crons vs endpoints cibles et `CRON_SECRET`

Périmètre inspecté : les quatre jobs cron déclarés dans `vercel.json` (`/api/cron/reminders`, `/api/cron/daily-tasks`, `/api/cron/upcoming-deadlines`, `/api/cron/budget-alerts`), leurs route handlers cibles sous `src/app/api/cron/**/route.ts`, les méthodes HTTP exportées, et la nature du contrôle d'authentification mis en œuvre, croisés avec l'inventaire `notes/inv-crons.md` (tâche 3.6) et la `Property 7` du design (`cron sans secret = High`). Constat : la bijection `vercel.json ↔ route.ts` est complète (aucun cron orphelin), chaque handler n'exporte que `GET`, et chaque handler effectue bien un contrôle de `CRON_SECRET` — `Property 7` au sens strict (cron totalement non authentifié) **n'est donc pas violée**. Les Findings ci-dessous portent sur la **forme** du contrôle d'authentification (header non standard, secret en query string), pas sur son absence.

### F-INF-003 — Cron handlers ne vérifient pas le header `Authorization: Bearer <CRON_SECRET>` envoyé par Vercel

- **Axe** : INF
- **Sévérité** : High
- **Description** : Les quatre cron handlers planifiés via `vercel.json` (`/api/cron/reminders`, `/api/cron/daily-tasks`, `/api/cron/upcoming-deadlines`, `/api/cron/budget-alerts`) implémentent un contrôle d'authentification qui lit uniquement le query string `?secret=` ou le header non standard `x-vercel-cron-secret`, et **n'inspecte pas** le header officiel `Authorization: Bearer <CRON_SECRET>` que Vercel injecte automatiquement dans les requêtes des Cron Jobs lorsque la variable `CRON_SECRET` est définie dans les variables d'environnement du projet. Or `vercel.json` ne déclare que les champs `path` + `schedule` pour chaque cron (cf. `vercel.json:4-21`) : il n'existe aucun mécanisme natif Vercel permettant d'injecter `?secret=...` dans l'URL invoquée par le scheduler, ni de configurer l'envoi d'un header personnalisé `x-vercel-cron-secret` côté plateforme. Le grep insensible à la casse sur `authorization` dans `src/app/api/cron/**/*.ts` ne retourne **aucune** occurrence — le header standard Vercel n'est lu nulle part. À confirmer via External_Console (logs Vercel des invocations cron récentes en production : status code et payload renvoyés).
- **Preuve** : `src/app/api/cron/reminders/route.ts:9-14`, `src/app/api/cron/daily-tasks/route.ts:9-14`, `src/app/api/cron/upcoming-deadlines/route.ts:9-14`, `src/app/api/cron/budget-alerts/route.ts:9-14` (motif identique : `const secret = searchParams.get("secret") ?? request.headers.get("x-vercel-cron-secret");`). `vercel.json:4-21` (déclaration des 4 entrées `crons` sans paramètre permettant à Vercel d'injecter le secret en query string). `commande: rg -i 'authorization' src/app/api/cron/` → 0 résultat (cf. `notes/annexe-A-commandes.md`).
- **Impact** : Si Vercel envoie effectivement `Authorization: Bearer <CRON_SECRET>` comme vecteur d'authentification standard de ses Cron Jobs (comportement documenté par la plateforme), alors les quatre invocations cron déclenchées par le scheduler Vercel renvoient `401 Unauthorized` en production. Conséquence métier : aucun rappel utilisateur (`reminders`), aucun récapitulatif quotidien (`daily-tasks`), aucune alerte d'échéance proche (`upcoming-deadlines`) ni alerte budgétaire (`budget-alerts`) n'est jamais délivré aux utilisateurs — l'ensemble des notifications planifiées est silencieusement non envoyé, sans alerte côté observabilité applicative (les 401 ne sont visibles que dans les logs HTTP Vercel, non remontés vers une alerte interne). Sévérité High retenue au titre du critère « Panne en production / Blocage utilisateur majeur » du design ; à confirmer via External_Console (Vercel) avant exécution de la remédiation.
- **Recommandation** : Étendre le contrôle d'authentification dans les quatre handlers pour accepter **en priorité** le header standard Vercel : `request.headers.get("authorization") === \`Bearer ${CRON_SECRET}\``. Ne conserver les autres canaux (header `x-vercel-cron-secret`, query string `?secret=`) que comme repli temporaire pour des invocations manuelles de debug, et idéalement les retirer une fois le canal standard validé en production (cf. F-INF-004 pour le retrait du canal query string). Avant toute modification, consulter en External_Console (Vercel) le statut HTTP des dernières exécutions des quatre crons sur 7 jours afin de confirmer ou infirmer le défaut en production.

### F-INF-004 — Cron handlers acceptent le secret en query string `?secret=`, exposition passive aux logs

- **Axe** : INF
- **Sévérité** : Medium
- **Description** : Les quatre cron handlers acceptent le secret partagé `CRON_SECRET` directement en paramètre de query string via `new URL(request.url).searchParams.get("secret")`. Or les query strings sont systématiquement consignés en clair dans les logs d'accès HTTP de la plateforme Vercel et de tout proxy/CDN intermédiaire, et peuvent fuir via le header `Referer` en cas de redirection sortante depuis le handler. Le secret peut donc transiter et persister en clair dans plusieurs systèmes d'observabilité sans qu'un attaquant ait besoin d'un accès direct au code source ou au fichier `.env`.
- **Preuve** : `src/app/api/cron/reminders/route.ts:11`, `src/app/api/cron/daily-tasks/route.ts:11`, `src/app/api/cron/upcoming-deadlines/route.ts:11`, `src/app/api/cron/budget-alerts/route.ts:11` — quatre occurrences de la forme `const secret = searchParams.get("secret") ?? request.headers.get("x-vercel-cron-secret");`.
- **Impact** : Toute personne ayant un accès lecture aux logs Vercel du projet (membres de l'équipe, tiers en cas d'export de logs vers un outil d'observabilité tiers) peut récupérer le `CRON_SECRET` en clair. Une fois le secret compromis, l'attaquant peut invoquer librement les quatre cron handlers et déclencher des envois massifs d'e-mails Resend (`reminders`) et de notifications push FCM/Web Push (les quatre routes), soit pour faire du spam à l'échelle de la base utilisateurs, soit pour saturer les quotas Resend/FCM, soit pour exploiter une éventuelle régression future du handler (lecture d'état non destinée au public). Sévérité Medium retenue au titre des critères « Dette technique significative » et « Défaut de robustesse » du design ; aucune fuite avérée à ce stade en l'absence d'inspection des logs Vercel (à confirmer via External_Console).
- **Recommandation** : Supprimer la branche `searchParams.get("secret")` de l'expression `??` dans les quatre handlers et ne conserver qu'un canal d'authentification basé sur le header standard `Authorization: Bearer <CRON_SECRET>` (cf. F-INF-003) — les headers de requête ne sont pas consignés en clair dans les logs HTTP Vercel par défaut. Si un canal de debug manuel par query string reste opérationnellement nécessaire, le restreindre explicitement à un environnement non-production (`if (process.env.NODE_ENV !== "production")`) et ne jamais l'utiliser pour des invocations automatisées.

### F-INF-005 — Variables d'environnement requises côté serveur absentes de `.env.example` (secrets et config OAuth/FCM/Resend/VAPID/Cron)

- **Axe** : INF
- **Sévérité** : Medium
- **Description** : Sept variables d'environnement référencées dans le code applicatif sous `src/app/api/**` et `src/lib/**` ne figurent **pas** dans `/Users/zerbib/clickup-clone/.env.example` : `FIREBASE_SERVICE_ACCOUNT` (init Firebase Admin SDK pour FCM, jette une erreur explicite si absente), `RESEND_API_KEY` (instanciation conditionnelle du client Resend ; sans elle, aucun e-mail transactionnel d'invitation/de rappel n'est envoyé), `VAPID_PRIVATE_KEY` et `NEXT_PUBLIC_VAPID_PUBLIC_KEY` (paire VAPID Web Push ; sans elles, `initWebPush()` retourne `false` et l'envoi Web Push est désactivé silencieusement), `CRON_SECRET` (secret partagé Vercel Cron ↔ handlers ; sans elle, les quatre cron handlers répondent `401 Unauthorized` — cf. `src/app/api/cron/**/route.ts:9-14`), `GOOGLE_CLIENT_ID` et `GOOGLE_CLIENT_SECRET` (OAuth Google Calendar ; sans elles, les quatre routes `/api/calendar/google/**` ne peuvent pas instancier le client OAuth). Le fichier `.env.example` est par ailleurs explicitement présenté comme la référence de configuration (il commente la section Upstash Redis et la section signature Android) : son rôle de documentation déclarative est donc voulu, mais incomplet.
- **Preuve** : `.env.example:1-13` (déclare 9 variables : `DATABASE_URL`, `NEXTAUTH_SECRET`, `NEXTAUTH_URL`, les 4 `RELEASE_*` Android, `UPSTASH_REDIS_REST_URL`, `UPSTASH_REDIS_REST_TOKEN`) ; `src/lib/firebase-admin.ts:8` (`process.env.FIREBASE_SERVICE_ACCOUNT`) ; `src/lib/email.ts:3` (`process.env.RESEND_API_KEY`) ; `src/lib/push.ts:3-4` (`process.env.NEXT_PUBLIC_VAPID_PUBLIC_KEY`, `process.env.VAPID_PRIVATE_KEY`) ; `src/app/api/cron/budget-alerts/route.ts:6` ; `src/app/api/cron/daily-tasks/route.ts:6` ; `src/app/api/cron/reminders/route.ts:7` ; `src/app/api/cron/upcoming-deadlines/route.ts:6` (`process.env.CRON_SECRET`) ; `src/app/api/calendar/google/auth/route.ts:12-13` ; `src/app/api/calendar/google/callback/route.ts:26-27` ; `src/app/api/calendar/google/clear/route.ts:21-22` ; `src/app/api/calendar/google/sync/route.ts:35-36` (`process.env.GOOGLE_CLIENT_ID`, `process.env.GOOGLE_CLIENT_SECRET`). Inventaire complet et toutes occurrences listées dans `notes/inv-env.md` § i et § iii.a.
- **Impact** : Reproduction de l'environnement d'exécution non documentée pour les pans **FCM** (Firebase), **e-mails transactionnels** (Resend), **Web Push** (VAPID), **Cron jobs** (`CRON_SECRET`) et **OAuth Google Calendar**. Conséquences concrètes : (i) un nouvel environnement (recette, dev local, nouveau déploiement Vercel) provisionné en s'appuyant sur `.env.example` démarre sans Firebase, sans envoi d'e-mails, sans Web Push, sans authentification cron et sans intégration Google Calendar — sans erreur de build, mais avec dégradation silencieuse de fonctionnalités majeures (notifications, invitations e-mail, rappels, alertes budgétaires) ; (ii) la documentation `.env.example` sous-représente la surface de configuration réelle (9 variables déclarées vs 15 effectivement requises hors `NODE_ENV`), augmentant le risque de dérive entre les valeurs réellement configurées dans la console Vercel (External_Console) et la documentation versionnée ; (iii) deux variables exposées côté client (`NEXT_PUBLIC_*`) ne sont pas non plus documentées, ce qui prive les contributeurs de la mise en garde implicite que porte le préfixe (à recouper en 11.4 — axe SEC). Sévérité Medium retenue au titre du critère « Dette technique significative » du design ; **non** classée High car (a) l'absence d'erreur de build, (b) l'absence d'exposition de secret (les variables sont consommées côté serveur ou portent légitimement le préfixe `NEXT_PUBLIC_*` pour celles qui sont publiques), (c) l'environnement de production reste fonctionnel tant que les valeurs sont correctement définies dans la console Vercel (à confirmer via External_Console).
- **Recommandation** : Compléter `.env.example` en ajoutant les sept entrées manquantes, regroupées par section commentée pour préserver la lisibilité du fichier, par exemple :

  ```dotenv
  # Firebase Admin SDK (FCM push notifications) — JSON service account stringifié
  FIREBASE_SERVICE_ACCOUNT=

  # Resend (e-mails transactionnels : invitations, rappels) — optionnel, désactivé si absent
  RESEND_API_KEY=
  FROM_EMAIL=
  NEXT_PUBLIC_APP_URL=

  # VAPID Web Push — publique côté client, privée côté serveur
  NEXT_PUBLIC_VAPID_PUBLIC_KEY=
  VAPID_PRIVATE_KEY=
  VAPID_CONTACT_EMAIL=

  # Vercel Cron jobs — secret partagé avec /api/cron/**
  CRON_SECRET=

  # Google Calendar OAuth (intégration agenda)
  GOOGLE_CLIENT_ID=
  GOOGLE_CLIENT_SECRET=
  ```

  Aucune valeur de secret ne doit être committée — uniquement les **noms** des variables et un commentaire explicatif. Ne pas oublier d'aligner le contenu avec les variables réellement définies en console Vercel (lecture seule via External_Console) afin de prévenir une dérive inverse (variable configurée en prod mais non documentée). À recouper avec la tâche 9.4 (`next.config.*` cohérent avec Vercel) et la tâche 11.4 (axe SEC : aucun secret métier ne porte le préfixe `NEXT_PUBLIC_*`).

### F-INF-006 — Variables de configuration non secrètes (`FROM_EMAIL`, `NEXT_PUBLIC_APP_URL`, `VAPID_CONTACT_EMAIL`) absentes de `.env.example` malgré un fallback hardcodé spécifique au déploiement

- **Axe** : INF
- **Sévérité** : Medium
- **Description** : Trois variables de configuration non secrètes consommées en production ne sont pas documentées dans `.env.example` : `FROM_EMAIL` (`src/lib/email.ts:5`, fallback `"onboarding@resend.dev"` — adresse d'envoi des e-mails transactionnels), `NEXT_PUBLIC_APP_URL` (`src/lib/email.ts:6`, fallback `"https://clickup-clone-three.vercel.app"` — URL canonique injectée dans les liens des e-mails et exposée côté client via le préfixe `NEXT_PUBLIC_*`), et `VAPID_CONTACT_EMAIL` (`src/lib/push.ts:5`, fallback `"contact@done.app"` — adresse de contact obligatoire VAPID). Chaque variable bénéficie d'un fallback codé en dur, mais ces fallbacks sont **liés au déploiement actuel** (`onboarding@resend.dev` est l'adresse de bac à sable Resend non vérifiée — conditionne la délivrabilité ; `clickup-clone-three.vercel.app` est l'URL Vercel actuelle du projet — invalide en cas de domaine personnalisé futur ou de second déploiement ; `contact@done.app` n'est pas garanti d'exister). L'omission empêche un nouvel environnement de comprendre que ces valeurs doivent être paramétrées en production.
- **Preuve** : `.env.example:1-13` (les trois variables n'y figurent pas) ; `src/lib/email.ts:5-6` (`const FROM_EMAIL = process.env.FROM_EMAIL ?? "onboarding@resend.dev";` puis `const APP_URL = process.env.NEXT_PUBLIC_APP_URL ?? "https://clickup-clone-three.vercel.app";`) ; `src/lib/push.ts:5` (`const contactEmail = process.env.VAPID_CONTACT_EMAIL ?? "contact@done.app";`). Inventaire complet : `notes/inv-env.md` § iii.a lignes 3, 4 et 7.
- **Impact** : (i) **`FROM_EMAIL`** : tant que la variable n'est pas définie, le fallback `"onboarding@resend.dev"` est utilisé. Cette adresse est l'adresse d'expéditeur fournie par Resend pour le bac à sable (non vérifiée pour un domaine arbitraire) et est connue pour entraîner un placement en spam plus probable et une non-personnalisation de l'expéditeur perçu. Risque de délivrabilité dégradée pour les invitations workspace. (ii) **`NEXT_PUBLIC_APP_URL`** : sans surcharge, les liens insérés dans les e-mails (acceptation d'invitation, etc.) pointent vers le sous-domaine Vercel par défaut `clickup-clone-three.vercel.app`. Si le projet est déployé sur un domaine personnalisé ou cloné dans un nouvel environnement, les liens redirigent vers la mauvaise URL et les invitations deviennent inutilisables. Variable préfixée `NEXT_PUBLIC_*` → exposée côté client (à recouper en 11.4). (iii) **`VAPID_CONTACT_EMAIL`** : utilisée par `webpush.setVapidDetails(\`mailto:${contactEmail}\`, …)` et envoyée aux push services (Mozilla autopush, FCM Web Push). Une adresse fictive est généralement tolérée mais peut entraîner un blocage en cas d'incident push (Mozilla peut tenter de contacter le contact). Sévérité Medium retenue au titre des critères « Dette technique significative » et « Défaut de robustesse » du design.
- **Recommandation** : Ajouter les trois variables à `.env.example` dans la section « Resend » et la section « VAPID Web Push » du fichier complété (cf. F-INF-005), avec un commentaire explicite indiquant le rôle et le caractère obligatoire en production (même si le code dispose d'un fallback). Optionnellement, réviser les fallbacks dans `src/lib/email.ts:5-6` et `src/lib/push.ts:5` pour qu'ils déclenchent un `console.warn` au démarrage en `process.env.NODE_ENV === "production"` lorsqu'aucune valeur n'a été fournie, afin de signaler activement la configuration manquante dans les logs Vercel. Cette modification de runtime est laissée à un workflow d'exécution ultérieur (hors périmètre lecture seule).

### Note — Variables `.env.example` non référencées dans `src/`/`prisma/`

### F-INF-011 — Cron handlers : `try/catch` global unique englobant la boucle d'envoi, une exception sur une itération abandonne tout le batch

- **Axe** : INF
- **Sévérité** : High
- **Description** : Les quatre cron handlers planifiés (`/api/cron/reminders`, `/api/cron/daily-tasks`, `/api/cron/upcoming-deadlines`, `/api/cron/budget-alerts`) implémentent un **unique** bloc `try { … } catch (error) { console.error(...); return 500; }` qui englobe à la fois la requête `prisma.findMany` initiale **et** la boucle `for (const item of items)` qui envoie les notifications. Aucune des itérations n'est isolée par un `try/catch` interne. Conséquence directe : si **une seule** itération lève une exception (par exemple une erreur Prisma intermittente sur `prisma.notification.findFirst`, une `prisma.reminder.update` qui échoue parce que l'enregistrement a été supprimé en concurrence, une exception synchrone remontée par `sendNotificationToUsers` — improbable mais non garanti), l'exception remonte jusqu'au `catch` outer, le handler renvoie immédiatement 500, et **toutes les itérations restantes ne sont jamais exécutées**. Pour `reminders`, cela signifie que tous les rappels postérieurs au rappel fautif dans l'ordre de `findMany` ne sont jamais envoyés à leur utilisateur, ne sont pas marqués `completed: true`, et la passe de cleanup `prisma.reminder.updateMany({ data: { completed: true } })` placée **après** la boucle (cf. lignes 81-84) n'est elle non plus jamais atteinte. Le seul `console.error` produit (`"[CRON] Reminders error:"`) ne mentionne pas **quel** rappel a échoué — il faut inspecter manuellement la stack trace pour reconstituer le contexte, et le tag conventionnel `[CRON]` n'est pas exploité par un APM (cf. F-INF-010).
- **Preuve** : `src/app/api/cron/reminders/route.ts:20-92` (un seul `try` ouvert ligne 20, fermé ligne 86 ; `catch (error)` lignes 87-90 logue puis renvoie 500 ; la boucle `for (const reminder of reminders)` aux lignes 33-79 est entièrement dans ce try, sans `try/catch` per-item — seul l'envoi e-mail asynchrone ligne 71 (`.catch((err) => console.error(...))`) est isolé) ; `src/app/api/cron/daily-tasks/route.ts:19-80` (motif identique : try ouvert ligne 19, fermé ligne 76 ; boucles `for (const task of tasks)` ligne 35, `for (const a of multiAssignees)` ligne 48, `for (const [userId, userTasks] of userTaskMap)` ligne 57 toutes encapsulées sans isolation) ; `src/app/api/cron/upcoming-deadlines/route.ts:20-87` (try ouvert ligne 20, fermé ligne 83 ; boucle imbriquée `for (const [userId, userTasks] of userTaskMap) { for (const task of userTasks) { … } }` lignes 55-79 sans `try/catch` interne, alors qu'elle effectue un `prisma.notification.findFirst` puis un `sendNotificationToUsers` par task) ; `src/app/api/cron/budget-alerts/route.ts:15-79` (try ouvert ligne 15, fermé ligne 75 ; double boucle `for (const budget of budgets) { for (const threshold of thresholds) { … } }` lignes 29-73 sans isolation, alors que chaque budget effectue un `prisma.notification.findFirst` puis un `sendNotificationToUsers` à plusieurs destinataires).
- **Impact** : (i) **Panne en cascade** : un seul rappel/budget/échéance défaillant peut bloquer la délivrance de toutes les notifications planifiées du même run ; en pratique, les invocations cron étant horaires (`reminders`) ou quotidiennes (`daily-tasks`, `upcoming-deadlines`, `budget-alerts`), la prochaine fenêtre rattrape théoriquement les enregistrements non traités — sauf pour `reminders` où la fenêtre `[now-2min, now+5min]` est étroite et la passe de cleanup placée après la boucle `prisma.reminder.updateMany({ ... completed: true ... })` peut rendre les rappels manqués irrécupérables (l'enregistrement est marqué traité même s'il ne l'a pas été). (ii) **Vercel Cron retry** : le scheduler Vercel **ne réessaie pas automatiquement** un cron qui renvoie 500 (le retry des Cron Jobs Vercel est déclenché par non-2xx mais avec un nombre limité — à confirmer via External_Console : politique exacte de la plateforme côté projet). (iii) **Diagnostic difficile** : le `console.error("[CRON] X error:", error)` ne capture pas l'identifiant de l'enregistrement fautif — un opérateur découvrant un 500 dans les Runtime Logs Vercel doit lire la stack trace, retrouver le rappel concerné via le timestamp, et corréler manuellement avec le contenu de la base. Sans APM (cf. F-INF-010), cette corrélation est laborieuse. Sévérité **High** retenue au titre du critère « Panne en production / Blocage utilisateur majeur » du design (les notifications planifiées — rappels métier, alertes budgétaires — peuvent être silencieusement perdues pour l'ensemble des utilisateurs d'un run).
- **Recommandation** : Encapsuler chaque itération de boucle dans un `try/catch` interne qui : (1) consigne l'erreur avec le contexte minimum (`console.error("[CRON] Reminder failed", { reminderId: reminder.id, userId: reminder.userId, err })`) afin de produire un log structuré exploitable côté APM ; (2) `continue` pour passer à l'itération suivante au lieu de propager. Le `try/catch` outer reste utile pour intercepter les erreurs hors-boucle (échec de `prisma.findMany`, panne de connexion Postgres). Modèle de remédiation typique :
  ```ts
  for (const reminder of reminders) {
    try {
      // …logique existante…
      created++;
    } catch (err) {
      console.error("[CRON] Reminder failed", { reminderId: reminder.id, err });
      // optionnel : ne pas marquer completed: true → laisser la fenêtre suivante retraiter
    }
  }
  ```
  Compléter par un fichier `instrumentation.ts` (cf. F-INF-010) qui envoie ces erreurs structurées à l'APM, et — à confirmer en External_Console — vérifier la configuration de retry du Cron Job Vercel (option `retries` dans la déclaration `crons` si supportée par le plan, ou prévoir une logique de re-traitement côté handler basée sur `completed: false` plutôt qu'un `updateMany` défensif).

### F-INF-012 — Routes API critiques : blocs `catch { … }` silencieux sans `console.error` sur des chemins métier (avatar, Google Calendar)

- **Axe** : INF
- **Sévérité** : Medium
- **Description** : Plusieurs routes API critiques contiennent des blocs `catch { … }` (forme TypeScript moderne sans paramètre d'exception, valide depuis TS 4.0) qui **avalent silencieusement l'erreur** sans aucun `console.error`, `console.warn` ni autre journalisation. Six occurrences ont été identifiées par `grep -rEn "catch\s*\{" src/app/api --include="route.ts"` : `src/app/api/calendar/google/clear/route.ts:55` (`catch { return NextResponse.json({ error: "Token expired" }, { status: 401 }); }` — l'échec de `oauth2Client.refreshAccessToken()` est traduit en 401 sans log : impossible de distinguer un refresh token révoqué d'une panne réseau de l'API Google), `src/app/api/calendar/google/clear/route.ts:85` (`catch { errors++; }` — l'échec d'un `calendar.events.delete()` n'est pas logué, seule la métrique `errors` est incrémentée et retournée au client), `src/app/api/calendar/google/sync/route.ts:214` (`catch { /* ignore cleanup errors */ }` — commentaire explicite mais aucune trace si la suppression d'un orphelin Google Calendar échoue), `src/app/api/user/avatar/route.ts:64` et `src/app/api/user/avatar/route.ts:101` (`catch { /* Ignore cleanup errors */ }` — l'échec d'un `unlink()` sur l'ancien avatar lors d'un upload ou d'une suppression est silencieux ; couplé à `F-INF-008`/`F-INF-013` sur le système de fichiers Vercel éphémère, le `unlink` peut échouer en production sans laisser aucune trace). Le cas de `src/app/api/mobile-me/route.ts:26` (`catch { return NextResponse.json({ error: "Invalid token" }, { status: 401 }); }`) est moins préoccupant car il s'agit d'un échec de validation JWT attendu sur token invalide ; il n'est pas inclus dans ce Finding.
- **Preuve** : `commande: grep -rEn "catch\s*\{" src/app/api --include="route.ts"` → 6 résultats listés ci-dessus (à journaliser dans `notes/annexe-A-commandes.md`) ; vérification ligne par ligne : `src/app/api/calendar/google/clear/route.ts:55-57`, `src/app/api/calendar/google/clear/route.ts:85-87`, `src/app/api/calendar/google/sync/route.ts:214-216`, `src/app/api/user/avatar/route.ts:64-66`, `src/app/api/user/avatar/route.ts:101-103`.
- **Impact** : (i) **Avatar upload** : si l'`unlink()` de l'ancien avatar échoue (système de fichiers en lecture seule sur Vercel — cf. observation associée à F-INF-008 sur le caractère éphémère de `public/uploads/`), l'utilisateur reçoit une réponse 200 succès, le nouveau avatar est bien enregistré en base via `prisma.user.update`, mais l'ancien fichier reste sur le disque éphémère jusqu'au prochain redéploiement — aucun signal côté APM ni Runtime Logs. (ii) **Google Calendar clear** : l'utilisateur clique « Tout supprimer », reçoit `{ deleted: 12, errors: 3 }` sans qu'aucun détail ne soit logué côté serveur — un opérateur de support ne peut pas reproduire l'échec sans demander à l'utilisateur de réessayer pendant que les logs verbeux sont activés. (iii) **Google Calendar sync** (cleanup orphelin) : un échec de suppression d'un événement orphelin n'est jamais signalé — l'événement reste dans le calendrier de l'utilisateur indéfiniment, créant une dérive silencieuse entre les tâches Done et le Google Calendar. (iv) **Refresh token Google** : un échec sur `oauth2Client.refreshAccessToken()` est traité de la même manière qu'un token simplement périmé (`401 "Token expired"`) sans aucun moyen de distinguer une panne d'API Google d'une révocation utilisateur — cas où la recommandation à donner à l'utilisateur diffère (réessayer plus tard vs. reconnecter le compte). Sévérité **Medium** retenue au titre des critères « Défaut de robustesse » et « Dette technique significative » du design.
- **Recommandation** : Pour chacun des cinq sites identifiés (hors `mobile-me`), passer du `catch { ... }` au `catch (err) { console.error("[avatar] cleanup failed", err); ... }` (ou `[calendar-clear]`, `[calendar-sync]` selon le contexte) en conservant la logique de fallback existante (`errors++`, `continue`, ou retour du même statut HTTP). Préfixer chaque message d'un tag conventionnel (`[avatar]`, `[google-calendar]`) facilite le filtrage côté Vercel Runtime Logs et, après installation d'un APM (cf. F-INF-010), le regroupement automatique. Pour les deux cas Google Calendar (`refreshAccessToken`, `events.delete`) : capturer **l'erreur structurée** retournée par `googleapis` (qui contient un code d'erreur Google distinct de l'`Error.message`) pour distinguer `invalid_grant` (révocation), `403`/`404` (permission révoquée ou événement déjà supprimé), `429`/`5xx` (panne transitoire).

### F-INF-013 — `sendNotificationToUsers` : erreurs FCM/Web Push non liées à 410 GONE silencieusement avalées

- **Axe** : INF
- **Sévérité** : Medium
- **Description** : La fonction d'orchestration de notifications `sendNotificationToUsers` dans `src/lib/notifications.ts` lance les envois FCM et Web Push en **fire-and-forget** (sans `await`) avec un `.catch(...)` qui ne réagit qu'aux erreurs spécifiques `"GONE"` ou `"registration-token-not-registered"`, en supprimant la `PushSubscription` correspondante. **Tous les autres types d'erreur sont silencieusement avalés**, sans aucun `console.error` : panne de l'API Firebase Admin (jeton de service invalide, quota dépassé), `getFirebaseAdmin()` qui throw `[FCM] FIREBASE_SERVICE_ACCOUNT env var not set` (erreur synchrone qui devient une rejection du `async`), erreurs réseau intermittentes (`ENOTFOUND`, timeout), payloads FCM rejetés (`messaging/invalid-argument`), erreurs Web Push 5xx du push service Mozilla/FCM, etc. Conséquence : si la variable d'environnement `FIREBASE_SERVICE_ACCOUNT` est mal configurée en production (cas plausible compte tenu de F-INF-005 — variable absente de `.env.example`), **chaque tentative d'envoi FCM échoue silencieusement** ; les utilisateurs Android ne reçoivent aucune notification push, mais aucun signal n'apparaît dans les Runtime Logs. La notification in-app (créée en base via `prisma.notification.createMany` avant les envois push) est bien enregistrée, ce qui masque encore davantage le problème côté UI. Le `.catch` interne fait également un `await prisma.pushSubscription.delete(...)` non protégé : si la subscription a été supprimée entre-temps (race condition), la rejection résultante devient elle-même une `unhandledRejection` non capturée, contournant complètement le pattern.
- **Preuve** : `src/lib/notifications.ts:74-86` (envoi FCM : `sendFcmNotification({...}).catch(async (err: Error) => { if (err.message === "GONE" || err.message?.includes("registration-token-not-registered")) { await prisma.pushSubscription.delete(...); } });` — le `if` ferme la branche sans `else` et sans `console.error`) ; `src/lib/notifications.ts:88-101` (envoi Web Push : motif identique, seul `err.message === "GONE"` est traité, le reste est swallowed) ; `src/lib/firebase-admin.ts:8-11` (`getFirebaseAdmin()` lance `throw new Error("[FCM] FIREBASE_SERVICE_ACCOUNT env var not set")` — exception synchrone à l'intérieur d'une fonction `async` → rejection du promise renvoyé par `sendFcmNotification` → swallowed par le `.catch` du caller) ; `src/lib/push.ts:51-62` (`sendPushNotification` consigne `console.log("[PUSH] Subscription expired, will be cleaned up")` puis `throw new Error("GONE")`, et logue les autres échecs via `console.error("[PUSH] Failed to send:", err)` **avant** de retourner sans throw — donc le `.catch` du caller dans `notifications.ts` n'est en fait jamais déclenché pour les erreurs autres que GONE, mais **pour** Web Push uniquement ; côté FCM via `src/lib/firebase-admin.ts:23-50`, aucun `console.error` n'est posé avant le throw : c'est bien le caller qui doit logger).
- **Impact** : (i) **FCM silencieusement désactivé en cas de mauvaise configuration** : si `FIREBASE_SERVICE_ACCOUNT` n'est pas définie (ou contient un JSON invalide), aucune notification FCM n'est délivrée mais l'application répond 200 partout (les notifications in-app sont créées). Couplé à F-INF-005 (variable absente de `.env.example` → susceptible d'être oubliée lors d'un nouveau provisioning d'environnement), risque de régression silencieuse non détectée pendant des semaines. (ii) **Pas d'agrégation possible** : impossible de produire un dashboard « taux d'échec FCM par jour » sans installer un APM **et** sans logger les erreurs en amont. (iii) **Race condition non gérée** : la branche `await prisma.pushSubscription.delete(...)` à l'intérieur du `.catch` peut elle-même échouer (subscription déjà supprimée par un autre handler concurrent) — la promise rejetée résultante n'est attachée à aucun `await` ni `.catch`, devenant une `UnhandledPromiseRejection` côté Node.js qui sera loguée par le runtime mais sans le contexte original. (iv) **Cron callers concernés** : tous les crons (`reminders`, `daily-tasks`, `upcoming-deadlines`, `budget-alerts`) appellent `sendNotificationToUsers` et héritent donc du problème ; le succès apparent du cron (200, `notified: N`) ne reflète pas la délivrance effective des push. Sévérité **Medium** retenue au titre des critères « Défaut de robustesse » et « Dette technique significative » ; **non** classée High car les notifications in-app continuent de fonctionner (parcours utilisateur principal préservé) et la dégradation reste invisible côté UI tant que l'utilisateur ne compare pas le nombre de notifications in-app avec celles de son téléphone.
- **Recommandation** : Modifier les deux `.catch` callbacks dans `src/lib/notifications.ts:81-86` et `src/lib/notifications.ts:95-100` pour logger systématiquement avant de filtrer le cas GONE, par exemple :
  ```ts
  .catch(async (err: Error) => {
    if (err.message === "GONE" || err.message?.includes("registration-token-not-registered")) {
      await prisma.pushSubscription.delete({ where: { id: sub.id } }).catch((delErr) => {
        console.error("[NOTIF] Failed to cleanup expired subscription", { subId: sub.id, delErr });
      });
      return;
    }
    console.error("[NOTIF] Push send failed", { subId: sub.id, provider: sub.provider, err });
  });
  ```
  Le `.catch` imbriqué sur le `prisma.delete` neutralise la race condition mentionnée ci-dessus. Une fois un APM installé (cf. F-INF-010), remplacer les `console.error` par un appel structuré (`Sentry.captureException(err, { tags: { feature: "push", provider: sub.provider } })`). Considérer également d'ajouter un log explicite côté `src/lib/firebase-admin.ts:11` avant le `throw` pour que la cause racine apparaisse dans les Runtime Logs même si le caller swallow la rejection.
