## Findings

### F-UI-001 — `<DialogContent>` sans `<DialogTitle>` dans le wrapper `CommandDialog` (palette de commandes Cmd+K)

- **Axe** : UI
- **Sévérité** : High
- **Description** : Le wrapper `CommandDialog` exporté par `src/components/ui/command.tsx` rend `<DialogContent className="overflow-hidden p-0 shadow-lg">` sans aucun `<DialogTitle>` ni `<DialogDescription>` (cf. `src/components/ui/command.tsx:31`). Or `Radix Dialog` (utilisé par notre primitive `Dialog`) requiert un `DialogTitle` accessible au lecteur d'écran : sans titre, Radix émet un avertissement console en développement et le rôle `dialog` ouvert reste sans nom accessible (`aria-labelledby` non résolu). L'unique consommateur applicatif `src/components/search/search-command.tsx:114-119` ouvre `<CommandDialog open={open} onOpenChange={setOpen}>` puis insère directement `<CommandInput>`/`<CommandList>` à l'intérieur sans fournir de titre ni utiliser `VisuallyHidden`. Le composant `Command` propre (`cmdk`) ne pose pas de titre Radix-compatible. La palette globale de recherche Cmd+K est donc rendue sans nom accessible pour les technologies d'assistance.
- **Preuve** : `src/components/ui/command.tsx:31` ; `src/components/search/search-command.tsx:114` ; `commande: grep -rln '<DialogContent' src/ --include='*.tsx'` (résultat consigné en annexe A) ; `commande: for f in $(grep -rln '<DialogContent' ...); do grep -c '<DialogTitle' "$f"; done` rapportant `DialogTitle=0` pour `src/components/ui/command.tsx`.
- **Impact** : Blocage utilisateur majeur sur le parcours « recherche globale » pour les utilisateurs de lecteurs d'écran (NVDA, VoiceOver, TalkBack). À l'ouverture, le dialogue n'annonce ni son rôle ni son objet : l'utilisateur entend un focus sur l'input sans contexte. Régression d'accessibilité non bloquante visuellement mais bloquante pour la conformité WCAG 2.1 critère 4.1.2 (Nom, rôle, valeur). En développement, Radix Dialog journalise par ailleurs un avertissement console pour chaque ouverture, polluant le débogage.
- **Recommandation** : Ajouter dans `CommandDialog` un titre rendu via `VisuallyHidden` (ex. `import { VisuallyHidden } from "@radix-ui/react-visually-hidden"`), par exemple : avant `<Command>`, insérer `<DialogTitle><VisuallyHidden>Recherche dans le workspace</VisuallyHidden></DialogTitle>` et idéalement une `DialogDescription` masquée décrivant le raccourci `⌘K`. Alternative : exposer une prop `title` obligatoire sur `CommandDialog` et l'imposer à `SearchCommand`.

### F-UI-002 — `<DialogContent>` sans `<DialogTitle>` dans `TaskDetailModal` (modale principale d'édition de tâche)

- **Axe** : UI
- **Sévérité** : High
- **Description** : Le composant `TaskDetailModal` (`src/components/task/task-detail-modal.tsx:22`) ouvre un `<DialogContent>` plein écran sur mobile (`!w-screen !max-w-none h-[100dvh]`) et large sur desktop (`md:!max-w-6xl md:h-[90vh]`) sans aucun `<DialogTitle>` ni `<DialogDescription>` (`grep -c '<DialogTitle' src/components/task/task-detail-modal.tsx` = 0). L'en-tête personnalisé n'affiche qu'un `<span>` portant le préfixe d'identifiant (`#{taskId.slice(0, 8)}`, ligne 25), insuffisant comme nom accessible et non rattaché à `aria-labelledby`. Cette modale est le **point d'entrée principal** d'édition d'une tâche dans toute l'application (ouverte depuis listes, calendrier, mes tâches, recherche).
- **Preuve** : `src/components/task/task-detail-modal.tsx:22` ; `commande: grep -c '<DialogTitle' src/components/task/task-detail-modal.tsx` (sortie : `0`) ; `commande: grep -c '<DialogContent' src/components/task/task-detail-modal.tsx` (sortie : `1`).
- **Impact** : Blocage utilisateur majeur sur le parcours métier le plus utilisé (édition de tâche) pour les technologies d'assistance : la modale s'ouvre sans nom annoncé, le focus tombe sur un contenu sans contexte. Non-conformité WCAG 4.1.2 et 2.4.6 (En-têtes et étiquettes). Sur Android Capacitor avec TalkBack, l'utilisateur perd la possibilité de comprendre qu'il est dans une modale plein écran et où se trouve la sortie. En développement, déclenche les avertissements Radix Dialog à chaque ouverture.
- **Recommandation** : Insérer dans `<DialogContent>` un `<DialogHeader>` contenant `<DialogTitle>` rattaché au titre de la tâche (récupéré via `useTask` ou passé en prop) ; à défaut, un titre masqué via `VisuallyHidden` reprenant `Détail de la tâche {title}`. Ajouter une `<DialogDescription>` masquée annonçant les actions disponibles. La closeur `<DialogPrimitive.Close>` héritée de `DialogContent` (cf. `src/components/ui/dialog.tsx:54-57`) reste fonctionnelle : seule la nommage manque.

Si la sobriété visuelle doit être conservée, masquer le titre via `<VisuallyHidden>` de `@radix-ui/react-visually-hidden`. ### F-UI-005 — Wrapper `DialogContent` interceptant systématiquement `onPointerDownOutside` et `onInteractOutside` sans option de désactivation

- **Axe** : UI
- **Sévérité** : Medium
- **Description** : La primitive `DialogContent` du projet (`src/components/ui/dialog.tsx:36-49`) appelle `e.preventDefault()` dans les deux callbacks `onPointerDownOutside` et `onInteractOutside` avant d'invoquer les handlers utilisateur, ce qui **désactive systématiquement la fermeture au clic extérieur et à `Escape`** pour toutes les modales du projet. Ce comportement diffère du composant shadcn/ui d'origine (qui laisse la fermeture clic-extérieur active par défaut) et n'est documenté nulle part dans le `design.md` ni dans `CLAUDE.md`. Aucun consommateur (`src/components/budget/*`, `src/components/task/*`, `src/components/workspace/*`, …) ne passe explicitement `onPointerDownOutside={(e) => {/* allow close */}}` pour réactiver le comportement. La primitive expose donc une dérive comportementale globale incohérente avec la convention shadcn/ui sur laquelle la documentation utilisateur s'appuie implicitement (le wrapper `Dialog` est utilisé par 25 fichiers — cf. annexe Tâche 5.1).
- **Preuve** : `src/components/ui/dialog.tsx:36`, `src/components/ui/dialog.tsx:38`, `src/components/ui/dialog.tsx:42`, `src/components/ui/dialog.tsx:46` ; `commande: grep -rln '<DialogContent' src/ --include='*.tsx' | wc -l` (sortie : `25`).
- **Impact** : UX dégradée : sur tablette ou desktop, l'utilisateur s'attend à fermer une modale en cliquant en dehors (convention web établie). Ici, **aucune** modale ne se ferme sur clic extérieur (sauf si l'utilisateur spamme la touche Échap, ou utilise le `X` en haut à droite). Sur mobile (modale plein écran via `task-detail-modal.tsx`), ce verrouillage est moins gênant mais empêche tout consommateur ponctuel d'opter pour le clic-extérieur sans réimplémenter `DialogContent`. Risque de dette UX disséminée sur 25 sites d'appel, et incohérence entre les variants `Dialog` (verrouillé) et `Sheet` (`src/components/ui/sheet.tsx:60-73` n'intercepte pas, donc `Sheet` ferme sur clic extérieur) — incohérence de variants au sens du critère « usage cohérent des variants » de la tâche 5.1.
- **Recommandation** : Ajouter une prop `dismissable?: boolean` (default `false` pour conserver le comportement actuel et éviter une régression), qui, lorsque `true`, n'appelle pas `e.preventDefault()`. Documenter le choix dans un commentaire au-dessus de `DialogContent` expliquant pourquoi le comportement diffère du shadcn/ui canonique. À défaut, justifier explicitement la décision dans `CLAUDE.md` et appliquer la même règle à `Sheet` pour cohérence.

### F-UI-006 — Variant `link` du Button déclaré mais jamais utilisé ; multiples boutons icône fixent une taille `h-/w-` qui contredit la variante `size="icon"` (`h-10 w-10` par défaut)

- **Axe** : UI
- **Sévérité** : Medium
- **Description** : Le `buttonVariants` (`src/components/ui/button.tsx:12-32`) déclare 6 variants (`default`, `destructive`, `outline`, `secondary`, `ghost`, `link`) et 4 tailles (`default`, `sm`, `lg`, `icon` = `h-10 w-10`). Le variant `link` n'est **jamais utilisé** dans `src/` (`grep -rn 'variant="link"' src/ --include='*.tsx'` retourne 0 ligne — vérifié avant rédaction). À l'inverse, **20+ usages** de `<Button size="icon">` ajoutent un `className="h-7 w-7"`, `"h-8 w-8"`, `"h-6 w-6"`, `"h-9 w-9"`, `"h-5 w-5"` qui surcharge la dimension canonique `h-10 w-10` du variant `icon` (cf. `src/components/task/task-action-menu.tsx:167` `h-7 w-7` ; `src/components/task/task-recurrence.tsx:70` `h-6 w-6` ; `src/components/task/task-dependencies.tsx:435` `h-5 w-5` ; `src/components/list/status-manager.tsx:254,262,286` `h-6 w-6` ; `src/components/custom-fields/custom-field-renderer.tsx:89-347` `h-8 w-8` x12 ; etc.). Cette dérive a deux conséquences : (i) la cible tactile `h-5 w-5` (20 px) ou `h-6 w-6` (24 px) tombe sous le seuil **44 × 44 px** de la WCAG 2.5.5 et de l'ergonomie mobile (à recouper avec la tâche 6.5) ; (ii) la variante `icon` perd son rôle de standard et l'usage cohérent du système de variants n'est plus garanti.
- **Preuve** : `src/components/ui/button.tsx:21` (variant `link` déclaré) ; `src/components/task/task-dependencies.tsx:435` (`h-5 w-5`) ; `src/components/task/task-recurrence.tsx:70` (`h-6 w-6`) ; `src/components/list/status-manager.tsx:254` ; `src/components/custom-fields/custom-field-renderer.tsx:89,92,123,126,229,232,269,272,305,308,344,347` (12 occurrences `h-8 w-8`) ; `commande: grep -rn 'size="icon"' src/ --include='*.tsx'` (50 occurrences) consignée en annexe A.
- **Impact** : Dette technique significative et défaut de cohérence du design system. (a) Toute évolution future du standard `icon` (par exemple passage à `h-9 w-9` pour une cible 36 px) sera invisible sur 30+ sites qui surchargent. (b) Sur mobile, certains boutons icône (h-5/h-6) sont trop petits pour être atteints fiablement au doigt — les tâches « supprimer dépendance » (`task-dependencies.tsx`), « supprimer statut » (`status-manager.tsx`) et « valider/annuler édition champ personnalisé » (`custom-field-renderer.tsx`) souffrent particulièrement du problème. (c) Le variant `link` mort augmente la surface du `cva` sans bénéfice mesurable et complique la lecture du standard.
- **Recommandation** : (i) Soit ajouter à `buttonVariants` des `size` complémentaires explicites (`icon-sm` = `h-7 w-7`, `icon-xs` = `h-6 w-6`) et migrer les sites d'appel ; soit, plus simplement, documenter la convention « surcharger `h-/w-` est interdit en production sauf justification ». (ii) Auditer et conserver ou supprimer le variant `link` selon que la roadmap prévoit ou non un emploi futur. (iii) Recouper avec la tâche 6.5 (ergonomie mobile, ≥ 44 px) et fusionner si la cause racine est identique en phase 4.

### F-UI-007 — Zoom utilisateur désactivé sur l'ensemble de l'application (`maximumScale: 1`, `userScalable: false`)

- **Axe** : UI
- **Sévérité** : High
- **Description** : Le `viewport` exporté par `src/app/layout.tsx:17-24` impose `maximumScale: 1` et `userScalable: false`. Cette configuration empêche tout utilisateur de zoomer manuellement sur l'interface, sur **toutes les pages** (publiques `/`, `/privacy`, `/terms`, `/login`, `/register` et plateforme `(platform)/**`). C'est une violation directe de **WCAG 2.1 critère 1.4.4 (Redimensionnement du texte)** qui exige qu'un texte puisse être agrandi jusqu'à 200 % sans perte de fonctionnalité. Sur iOS Safari et Android Chrome (web et Capacitor WebView), cette directive bloque le geste pinch-to-zoom natif. Aucune justification n'est consignée en commentaire ni dans `CLAUDE.md` ; il s'agit d'une convention obsolète héritée d'anciennes pratiques mobiles que les standards d'accessibilité ont depuis abrogées.
- **Preuve** : `src/app/layout.tsx:18` (`viewport.width`), `src/app/layout.tsx:20` (`maximumScale: 1`), `src/app/layout.tsx:21` (`userScalable: false`) ; `commande: grep -n 'maximumScale\|userScalable' src/app/layout.tsx` (2 lignes : `20:  maximumScale: 1,` et `21:  userScalable: false,`).
- **Impact** : Blocage majeur d'accessibilité pour les utilisateurs malvoyants ou âgés qui ont besoin de zoomer pour lire les libellés (échéances, montants budget, statuts de tâches). Sur Capacitor Android, l'expérience est aggravée par la WebView qui ne propose pas non plus de zoom système. Non-conformité WCAG 1.4.4 (niveau AA) bloquante pour toute mise en conformité réglementaire (RGAA, ADA, EAA). De plus, le mobile Capacitor étant la principale cible, ce verrouillage rend l'application inutilisable pour une portion non négligeable du public cible.
- **Recommandation** : Supprimer `maximumScale: 1` et `userScalable: false` du `viewport` ; ne conserver que `width: "device-width"`, `initialScale: 1`, `viewportFit: "cover"`, `interactiveWidget: "resizes-content"`. Si une crainte spécifique existe (ex. un composant qui se déforme au zoom), corriger ce composant plutôt que désactiver le zoom global. Vérifier que `pb-[env(safe-area-inset-bottom)]` et `pt-[env(safe-area-inset-top)]` continuent de fonctionner correctement après suppression.

### F-UI-027 — Pattern `data ?? []` / `data ?? defaultValue` consommant immédiatement le résultat SWR : confond loading et empty, masque toute possibilité d'afficher un skeleton sur 13 sites

- **Axe** : UI
- **Sévérité** : Medium
- **Description** : 13 sites consomment immédiatement la donnée SWR via une coalescence `data ?? []` : `(comments ?? []).map(...)`, `(activities ?? []).map(...)`, `(statuses ?? []).map(...)`, `(searchResults ?? []).map(...)`, `const items = subtasks ?? []`, etc. Ce pattern court-circuite **systématiquement** la phase de loading : pendant le premier rendu (avant que `data` soit résolu), `data` vaut `undefined` → coalescence en `[]` → la fonction `.map(...)` rend une liste vide → la branche « empty » du composant s'affiche immédiatement, **avant même le premier appel réseau**. Le squelette potentiel (s'il est défini sous `if (isLoading)`) n'est jamais rendu car la branche empty prend la priorité visuelle.

  Sites concernés (compteur `commande: grep -rnE "(comments|activities|statuses|members|tasks|searchResults|fields|values|attachments|subtasks|invites|teams|workspaces|spaces|notifications) \\?\\? \\[\\]" src/components/ src/app/ --include='*.tsx'` = 13 lignes) :

  - `src/components/task/comment-list.tsx:46` — `(comments ?? []).map(...)` ; squelette présent ligne 35-41 mais conditionné par `isLoading` correctement → OK structurellement, mais **ligne 76 `comments?.length === 0`** masque silencieusement le rendu du « No comments yet » lorsque `comments` est `undefined` (pendant load OU error).
  - `src/components/task/activity-feed.tsx:110` — `(activities ?? []).map(...)` ; idem, ligne 144 `activities?.length === 0` ne se déclenche jamais en error.
  - `src/components/task/subtask-list.tsx:52` — `const items = subtasks ?? []` ; le composant montre **immédiatement** la barre de progression à 0 % et la liste vide sans aucun skeleton, alors que le hook ligne 47 a un `fallbackData: initialSubtasks` qui devrait masquer cet effet — **conflit subtil** : si `initialSubtasks` est vide mais que la requête est en cours, l'utilisateur croit qu'il n'y a aucune sous-tâche.
  - `src/components/task/task-attachments.tsx:68` — `const items = attachments ?? []` ; absence totale de skeleton (le composant ne consomme même pas `isLoading`). Ligne 73 affiche d'emblée « Pièces jointes (0) » avant que la requête ne réponde — illusion d'absence.
  - `src/components/task/assignee-selector.tsx:98` et `src/components/task/multi-assignee-selector.tsx:78,112` — `(members ?? []).map(...)` ; pas de skeleton dans le popover, qui affiche « Aucun membre » dès l'ouverture si la requête est lente.
  - `src/components/views/list-view.tsx:65` et `src/components/views/board-view.tsx:67,122,150` — `(statuses ?? []).map(...)` dans la mémoization ; ici le problème est mitigé car `isLoading` provient de `useTasks`, mais si la requête `statuses` est plus lente que `tasks`, le rendu post-`isLoading` montre une liste sans groupes (cf. F-UI-029 ci-après pour le timing).
  - `src/components/filters/status-filter.tsx:59`, `src/components/filters/assignee-filter.tsx:63`, `src/components/task/status-badge.tsx:60` — popovers de filtres / sélecteur de statut, affichent une liste vide instantanément à l'ouverture si la requête est en cours.

  Le pattern annule le bénéfice du `<Skeleton>` même quand celui-ci est défini : `comment-list.tsx` rend correctement le skeleton sous `isLoading`, mais `data` reste `undefined` pendant un échec → la branche `else` (ligne 43) prend la main → ligne 76 `comments?.length === 0` est faux (undefined !== 0) → **rien n'est rendu**, ni « No comments yet » ni squelette. Le composant tombe en blanc silencieux.

- **Preuve** : `commande: grep -rnE "(comments|activities|statuses|members|tasks|searchResults|fields|values|attachments|subtasks|invites|teams|workspaces|spaces|notifications) \\?\\? \\[\\]" src/components/ src/app/ --include='*.tsx'` (sortie : 13 lignes consignée en annexe A) ; sites individuels : `src/components/task/comment-list.tsx:46,76` ; `src/components/task/activity-feed.tsx:110,144` ; `src/components/task/subtask-list.tsx:52` ; `src/components/task/task-attachments.tsx:68,73` ; `src/components/task/assignee-selector.tsx:98` ; `src/components/task/multi-assignee-selector.tsx:78,112` ; `src/components/views/list-view.tsx:65` ; `src/components/views/board-view.tsx:67,122,150` ; `src/components/filters/status-filter.tsx:59` ; `src/components/filters/assignee-filter.tsx:63` ; `src/components/task/status-badge.tsx:60`.
- **Impact** : (a) Sur les 13 sites listés, le squelette défini en amont est neutralisé soit complètement (cas où aucun `<Skeleton>` n'existe — `task-attachments`, `subtask-list`, `multi-assignee-selector`, sélecteurs et filtres), soit partiellement (le squelette s'affiche pendant `isLoading=true` mais l'absence d'`error` consommé crée une zone blanche post-loading). (c) Sur `<TaskAttachments>`, l'affichage immédiat de « Pièces jointes (0) » alors qu'elles sont en cours de chargement induit l'utilisateur en erreur (il ajoute une pièce jointe persuadé qu'il n'y en a pas, créant un doublon). (d) Sur les vues de liste/board, ce pattern combiné à un statut SWR séparé pour `statuses` peut produire un état où les tâches sont rendues sans groupes (ScreenShot empty state). Sévérité Medium plutôt que High car le timing typique (200-800 ms) limite l'exposition mais le problème reste systémique sur 13 sites.
- **Recommandation** : (i) Remplacer `(items ?? []).map(...)` par un branchement explicite `if (isLoading && !items) → <Skeleton />` puis `if (error) → <ErrorState onRetry />` puis `if (!items || items.length === 0) → <EmptyState />` puis `items.map(...)`. (iii) Pour `<MultiAssigneeSelector>` et `<TaskAttachments>` qui ne destructurent même pas `isLoading`, l'ajouter et brancher proprement. (iv) Cette correction renforce l'effet du Finding F-UI-026 (consommation explicite d'`error`).

(v) **Pour `/notes` et `/goals`**, conditionner les `<EmptyState>` à `data !== undefined`, sinon afficher un skeleton de 3-5 cartes. (vi) Cette mise en cohérence supprime également l'illusion d'« empty state trompeur » pointée en F-UI-029.

### F-UI-029 — États « empty » et « loading » indistinguables : `<EmptyState>` rendu prématurément avant la résolution SWR, induisant l'utilisateur en erreur (10 sites)

- **Axe** : UI
- **Sévérité** : Medium
- **Description** : Sur 10 sites distincts, l'`<EmptyState>` (composant partagé `src/components/shared/empty-state.tsx`) ou un fallback équivalent est rendu **avant** que SWR n'ait résolu sa requête, parce que le test ne distingue pas `data === undefined` (loading) de `data === []` (vraie liste vide). Le pattern fautif est typiquement `if (!data || data.length === 0)` au lieu de `if (data && data.length === 0)`. Conséquence : l'utilisateur voit un message « Aucun X » pendant 200-1500 ms avant que la vraie liste ne s'affiche, ce qui crée une **fausse impression d'absence de données** et incite à une création inutile (cf. boutons « Créer un objectif », « Nouvelle note », « Créer un espace » exposés dans l'`<EmptyState>`).

  Sites concernés :

  - `src/app/(platform)/goals/page.tsx:198` — `(!goals || goals.length === 0)` rend l'`<EmptyState>` « Aucun objectif » (avec bouton « Créer un objectif »). Pendant le chargement, l'utilisateur peut cliquer sur ce bouton et créer un objectif vide superflu.
  - `src/app/(platform)/notes/page.tsx:264` — `filtered.length === 0` (où `filtered` dérive de `notes ?? []`) → message « Aucune note » dans la sidebar. La zone d'édition affiche son propre `<EmptyState>` « Aucune note sélectionnée » avec bouton « Nouvelle note ».
  - `src/app/(platform)/workspace/page.tsx:115` — `workspaces?.length === 0` rend l'`<EmptyState>` « Aucun espace de travail » avec bouton « Créer un espace ». Le test passe à `false` quand `workspaces` est `undefined`, donc ici le bug est inverse : pendant le loading, **aucun** `<EmptyState>` ni squelette n'est rendu (le squelette ligne 64-70 ne se déclenche que sous `isLoading`, OK), mais l'absence du test correct laisse passer un bref blanc avant que le squelette ne disparaisse.
  - `src/app/(platform)/workspace/[workspaceId]/page.tsx:44-49` — `if (!workspace) return <EmptyState>` ; **partiellement bon** car `isLoading` est traité avant (ligne 32-49), mais le test final sur `!workspace` confond 404 (réelle absence), 403 (refus) et 500 (erreur transitoire) — cf. F-UI-026.
  - `src/app/(platform)/workspace/[workspaceId]/space/[spaceId]/page.tsx:46-51` — pattern identique (404/403/500 confondus).
  - `src/components/custom-fields/custom-fields-section.tsx:78-93` — `if (!fields || fields.length === 0)` rend la branche « No custom fields defined for this workspace ». Pendant le chargement (`fields === undefined`), le message faux est affiché. **Régression** : si l'admin du workspace a défini des champs personnalisés mais que la requête est lente, l'utilisateur croit qu'il faut les créer — il ouvre le `<CustomFieldManager>` et peut créer des doublons.
  - `src/components/dashboard/budget-widget.tsx:65-69` — `topBudgets.length === 0` rend « Aucun budget » à l'intérieur de la carte. **Bonne pratique** : `isLoading` est traité en amont (ligne 21-39), donc la branche n'est atteinte qu'après résolution → conforme. Cas d'illustration de la convention attendue.
  - `src/components/dashboard/recent-activity.tsx:107-122` — bien encapsulé sous `if (isLoading)` puis `if (activities.length === 0)` → conforme.
  - `src/components/dashboard/upcoming-deadlines.tsx:75-92` — idem, conforme.
  - `src/components/dashboard/tasks-by-status-chart.tsx:79-84` — `if (data.length === 0 || total === 0)` rend « Aucune tâche ». Conforme car `data` est passé en prop (déjà résolu côté parent via `<DashboardPage>`).
  - `src/components/views/list-view.tsx:79-87` — `if (!statuses || statuses.length === 0)` rend l'`<EmptyState>` « Aucun statut configuré ». Encapsulé après `if (isLoading) return <LoadingSpinner>` (ligne 71-77), donc partiellement OK : le test `!statuses` masquerait toutefois une 500 sur `/api/lists/{id}/statuses` (où `isLoading=false` mais `statuses=undefined`) en montrant l'`<EmptyState>` au lieu d'une erreur — confusion error/empty.
  - `src/components/views/board-view.tsx:165-173` — pattern identique.
  - `src/app/(platform)/reminders/page.tsx:56-72` — `if (workspaceLoading) return <Skeleton>` puis `if (!currentWorkspace)` → conforme pour le workspace, mais `<ReminderList>` consomme ensuite `useReminders` sans gérer la confusion error/empty au niveau `reminders`.
  - `src/components/views/calendar-view.tsx:131-141` — `if (isLoading) return <LoadingSpinner>` mais aucune branche dédiée à un état post-loading vide ; la grille calendrier rend ses cellules avec `dayTasks.length === 0` partout — interprétation laissée à l'utilisateur.

  Au total : **10 sites** où l'`<EmptyState>` ou son équivalent est rendu de façon ambiguë (avant résolution OU sans distinction de l'erreur). 4 sites dashboards (`budget-widget`, `recent-activity`, `upcoming-deadlines`, `tasks-by-status-chart`) servent de **modèle conforme** mais sont l'exception.

- **Preuve** : `src/app/(platform)/goals/page.tsx:198` ; `src/app/(platform)/notes/page.tsx:264,400` ; `src/app/(platform)/workspace/page.tsx:115` ; `src/app/(platform)/workspace/[workspaceId]/page.tsx:44-49` ; `src/app/(platform)/workspace/[workspaceId]/space/[spaceId]/page.tsx:46-51` ; `src/components/custom-fields/custom-fields-section.tsx:78-93` ; `src/components/views/list-view.tsx:79-87` ; `src/components/views/board-view.tsx:165-173` ; comparaison conforme : `src/components/dashboard/budget-widget.tsx:65-69` (bon pattern : `isLoading` testé en amont).
- **Impact** : (a) **Création de doublons** : sur `/goals` et `/notes`, le bouton « Créer » exposé dans l'`<EmptyState>` prématuré peut être pressé par l'utilisateur, créant un objectif/note vide superflu avant que la vraie liste n'apparaisse. (b) Sur `<CustomFieldsSection>`, l'admin peut être tenté de re-créer les champs personnalisés pendant le chargement — duplication BDD possible. (c) Sur `/workspace/{id}` et `/space/{id}`, un 404 réel et un 500 transitoire sont rendus avec le **même** message « introuvable » alors que les recommandations utilisateur diffèrent (le 500 invite à recharger, le 404 invite à revenir à la liste). (d) Sur les vues `list-view`/`board-view`, l'`<EmptyState>` « Aucun statut configuré » peut s'afficher à tort si la route `/api/lists/{id}/statuses` échoue (alors qu'au moins le statut « To Do » est créé par défaut au seed) — l'utilisateur peut alors cliquer pour configurer des statuts qu'il a déjà. (e) Renforce l'incohérence UX entre les 4 widgets dashboard conformes et le reste de l'application — incohérence pédagogique qui complique l'apprentissage de la convention pour les nouveaux contributeurs. Severité Medium car les conséquences sont des doublons mineurs et des fenêtres de confusion brèves, sans perte de données.
- **Recommandation** : (i) **Convention de test** : remplacer `if (!data || data.length === 0)` par `if (data !== undefined && data.length === 0)` quand l'objectif est de rendre l'état vide ; brancher la branche de chargement séparément sous `if (isLoading || data === undefined)`. (ii) Migrer les 10 sites listés. (iii) Pour les pages workspace / space, distinguer `if (error?.status === 404)`, `if (error?.status === 403)`, `if (error)` génériquement, et `if (data && /* empty */)`. (iv) Documenter dans `<EmptyState>` (`src/components/shared/empty-state.tsx`) un commentaire JSDoc rappelant qu'il ne doit jamais être rendu pendant le chargement, et créer une variante `<LoadingState>` ou utiliser le mode `<EmptyState variant="error">` recommandé en F-UI-026. (v) Recouper avec les 4 widgets dashboard conformes pour standardiser le pattern (`budget-widget`, `recent-activity`, `upcoming-deadlines`, `tasks-by-status-chart` servent de référence).
