## Tâche 7.1 — Audit structure App Router (`route.ts` / `page.tsx` / `layout.tsx`)

### F-LOG-002 — Fichier composant client co-localisé dans un dossier de route

- **Axe** : LOG
- **Sévérité** : Medium
- **Description** : Le fichier `task-detail-page-client.tsx` est placé directement dans le dossier de route `src/app/(platform)/task/[taskId]/`. Bien que Next.js tolère la co-location de fichiers non exportés par défaut dans les dossiers de route, cette pratique mélange composants applicatifs et fichiers de routage. La convention recommandée est de placer les composants client dans `src/components/` ou un sous-dossier dédié `_components/` pour maintenir une séparation claire entre la couche de routage et la couche de présentation.
- **Preuve** : `src/app/(platform)/task/[taskId]/task-detail-page-client.tsx:1`
- **Impact** : Risque de confusion sur le rôle du fichier (est-ce une page ? un composant ?). Si d'autres développeurs reproduisent ce pattern, le dossier `app/` devient un mélange de fichiers de routage et de composants, rendant la navigation dans le projet plus difficile.
- **Recommandation** : Déplacer `task-detail-page-client.tsx` vers `src/components/task/` ou créer un sous-dossier `src/app/(platform)/task/[taskId]/_components/` pour signaler explicitement qu'il ne s'agit pas d'un fichier de routage.

### F-LOG-003 — Page `invite` entièrement client-side avec fetch dans useEffect

- **Axe** : LOG
- **Sévérité** : Medium
- **Description** : La page `src/app/invite/[token]/page.tsx` est déclarée `"use client"` et effectue le chargement des données d'invitation via un `useEffect` + `fetch` côté client. Cette approche contourne les avantages des Server Components (rendu initial côté serveur, pas de waterfall client, meilleur SEO). La page pourrait être un Server Component effectuant le fetch côté serveur, avec un composant client enfant pour les interactions (acceptation, redirection).
- **Preuve** : `src/app/invite/[token]/page.tsx:1` (`"use client"`) ; `src/app/invite/[token]/page.tsx:31-47` (fetch dans useEffect)
- **Impact** : L'utilisateur voit un spinner de chargement au lieu d'un contenu pré-rendu. Le temps de premier affichage significatif (FCP) est dégradé. En cas d'erreur réseau, l'utilisateur n'a aucun contenu initial.
- **Recommandation** : Convertir `page.tsx` en Server Component qui effectue le fetch de l'invitation côté serveur (via Prisma ou un appel interne), puis passer les données à un composant client `InvitePageClient` pour les interactions (boutons accepter/refuser, redirection).

### F-LOG-004 — Pages `terms` et `privacy` hors de tout route group sans layout dédié

- **Axe** : LOG
- **Sévérité** : Medium
- **Description** : Les pages `src/app/terms/page.tsx` et `src/app/privacy/page.tsx` sont placées à la racine de `src/app/` sans route group ni layout intermédiaire. Elles héritent du `RootLayout` qui charge tous les providers applicatifs (`AuthProvider`, `WorkspaceProvider`, `ModalProvider`, `SWRProvider`, etc.) alors qu'elles n'en ont aucun besoin — ce sont des pages statiques de contenu légal. De plus, elles codent en dur le thème sombre (`bg-[#0a0a0f] text-white`) au lieu d'utiliser le système de thème de l'application.
- **Preuve** : `src/app/terms/page.tsx:11` ; `src/app/privacy/page.tsx:11` ; `src/app/layout.tsx:95-108` (providers chargés pour toutes les routes)
- **Impact** : Chargement inutile de JavaScript côté client (providers, hooks SWR, composants de notification) pour des pages purement statiques. Incohérence avec le système de thème : si l'utilisateur est en mode clair, ces pages restent en sombre.
- **Recommandation** : Créer un route group `(static)` avec un layout minimal (sans providers applicatifs) pour les pages `terms` et `privacy`. Remplacer les couleurs codées en dur par les tokens du thème (`bg-background text-foreground`).

### F-LOG-005 — Absence de `not-found.tsx` dans le route group `(platform)`

- **Axe** : LOG
- **Sévérité** : Medium
- **Description** : Le route group `(platform)` ne possède pas de fichier `not-found.tsx` dédié. Lorsque la page `task/[taskId]/page.tsx` appelle `notFound()` (par exemple pour une tâche inexistante ou un accès non autorisé), le fallback est le `not-found.tsx` racine qui affiche un design plein écran avec fond sombre (`bg-[#0a0a0f]`), sans la sidebar ni la navigation de la plateforme. L'utilisateur perd tout contexte de navigation.
- **Preuve** : `src/app/(platform)/task/[taskId]/page.tsx:35` (appel à `notFound()`) ; absence de `src/app/(platform)/not-found.tsx`
- **Impact** : Expérience utilisateur dégradée : l'utilisateur se retrouve sur une page sans navigation et doit manuellement revenir au dashboard. Perte de contexte de l'application.
- **Recommandation** : Créer un fichier `src/app/(platform)/not-found.tsx` qui affiche un message d'erreur 404 dans le cadre du layout plateforme (avec sidebar et navigation), permettant à l'utilisateur de naviguer facilement vers une autre section.

### F-LOG-006 — Absence totale de fichiers `loading.tsx` dans l'arborescence App Router

- **Axe** : LOG
- **Sévérité** : Medium
- **Description** : Aucun fichier `loading.tsx` n'est présent dans l'ensemble de l'arborescence `src/app/`. Les pages Server Components effectuant des requêtes Prisma (comme `task/[taskId]/page.tsx`) n'ont pas de squelette de chargement (skeleton) affiché pendant la résolution des données côté serveur. Next.js utilise `loading.tsx` pour afficher un état de chargement instantané via React Suspense pendant que le Server Component résout ses données.
- **Preuve** : `commande: find src/app -name "loading.tsx"` (aucun résultat)
- **Impact** : Lors de la navigation côté client vers une page Server Component avec accès Prisma, l'utilisateur ne voit aucun feedback visuel pendant le chargement. La page semble figée jusqu'à la résolution complète des données, ce qui dégrade la perception de réactivité.
- **Recommandation** : Ajouter des fichiers `loading.tsx` au minimum dans `src/app/(platform)/` (skeleton global avec sidebar) et dans les routes à chargement potentiellement long (`task/[taskId]/`, `workspace/[workspaceId]/`).

## Tâche 7.2 — Audit frontière server/client (directives `"use client"`)

### F-LOG-007 — Composants avec `"use client"` non justifié (rendu statique pur)

- **Axe** : LOG
- **Sévérité** : Medium
- **Description** : Quatre composants portent la directive `"use client"` alors qu'ils n'utilisent aucun hook, gestionnaire d'événement, API navigateur ni fonctionnalité exclusivement client. `category-badge.tsx` rend un `<span>` avec des props. `budget-alert.tsx` effectue un rendu conditionnel statique. `budget-skeleton.tsx` compose des composants `Skeleton` sans interactivité. `list-view-custom-fields.tsx` est un stub vide retournant `null`. Ces composants pourraient être des Server Components, réduisant le bundle JavaScript envoyé au client.
- **Preuve** : `src/components/budget/category-badge.tsx:1` ; `src/components/budget/budget-alert.tsx:1` ; `src/components/budget/budget-skeleton.tsx:1` ; `src/components/views/list-view-custom-fields.tsx:1`
- **Impact** : JavaScript inutile inclus dans le bundle client. Chaque composant marqué `"use client"` force l'inclusion de son code et de ses dépendances dans le bundle côté client, même s'il pourrait être rendu côté serveur sans coût réseau.
- **Recommandation** : Retirer la directive `"use client"` de ces quatre fichiers. Vérifier que leurs composants parents ne dépendent pas implicitement de cette directive (si un parent est déjà `"use client"`, la directive enfant est redondante mais inoffensive ; si le parent est un Server Component, le retrait permet un rendu serveur effectif).

### F-LOG-008 — Pages de vue utilisant `"use client"` uniquement pour `useParams` (contournable via props `params`)

- **Axe** : LOG
- **Sévérité** : Medium
- **Description** : Quatre pages de vue (`board/page.tsx`, `gantt/page.tsx`, `list-view/page.tsx`, `calendar/page.tsx`) sont déclarées `"use client"` uniquement pour appeler `useParams()` et extraire les paramètres de route. Dans l'App Router de Next.js, les composants `page.tsx` reçoivent `params` en prop côté serveur. Ces pages pourraient être des Server Components asynchrones qui passent les paramètres à leurs composants enfants client (`BoardView`, `GanttView`, `ListView`, `CalendarView`), éliminant le besoin de `"use client"` au niveau page.
- **Preuve** : `src/app/(platform)/workspace/[workspaceId]/space/[spaceId]/list/[listId]/board/page.tsx:1` ; `src/app/(platform)/workspace/[workspaceId]/space/[spaceId]/list/[listId]/gantt/page.tsx:1` ; `src/app/(platform)/workspace/[workspaceId]/space/[spaceId]/list/[listId]/list-view/page.tsx:1` ; `src/app/(platform)/workspace/[workspaceId]/space/[spaceId]/list/[listId]/calendar/page.tsx:1`
- **Impact** : La frontière client est remontée au niveau page, forçant l'intégralité de l'arbre de rendu (y compris le layout de la page avec `ViewSwitcher` et `FilterBar`) à être évalué côté client. Le HTML initial envoyé au navigateur ne contient pas le markup de ces pages, dégradant le FCP et le SEO.
- **Recommandation** : Convertir ces pages en Server Components asynchrones recevant `params` en prop : `export default async function BoardPage({ params }: { params: { workspaceId: string; spaceId: string; listId: string } })`. Passer les paramètres extraits aux composants enfants client. Le `ViewSwitcher` peut calculer `basePath` côté serveur et le passer en prop.

### F-LOG-009 — Page `dashboard/page.tsx` entièrement client-side forçant un arbre volumineux côté client

- **Axe** : LOG
- **Sévérité** : Medium
- **Description** : La page `dashboard/page.tsx` (~200 lignes) est déclarée `"use client"` et constitue un arbre de composants volumineux incluant `StatsCards`, `TasksByStatusChart`, `TasksByPriorityChart`, `RecentActivity`, `UpcomingDeadlines`, `BudgetWidget` et `FinanceWidget`. La page utilise effectivement des hooks (`useSWR`, `useWorkspace`, `useModal`, `useRouter`), mais la structure statique (header, quick links, grille CSS) et les composants de présentation pure pourraient être rendus côté serveur si la page était scindée en un Server Component orchestrateur et des îlots client interactifs.
- **Preuve** : `src/app/(platform)/dashboard/page.tsx:1` (directive `"use client"` sur ~200 lignes de JSX)
- **Impact** : L'intégralité du code de la page dashboard (imports, logique de mémoisation, JSX statique) est envoyée au client. Le navigateur doit hydrater tout l'arbre avant que la page ne soit interactive. Sur mobile ou connexion lente, cela augmente le TTI (Time to Interactive) de la page la plus visitée de l'application.
- **Recommandation** : Extraire la page en un Server Component qui effectue le rendu du layout statique (header, quick links, grille) et délègue les zones interactives à des composants client enfants (`<DashboardStats />`, `<DashboardCharts />`, etc.). Alternativement, utiliser le pattern « Server Component wrapper + Client Component island » pour isoler les hooks SWR dans des composants dédiés tout en gardant le squelette de page côté serveur.

## Tâche 7.3 — Audit hooks SWR : contrôle `res.ok`, gestion clé, revalidation

> **Résultat du contrôle `res.ok`** : Les 59 fetchers SWR inventoriés (annexe D) vérifient tous `res.ok` avant de parser le JSON — soit via un fetcher local (`if (!r.ok) throw …`), soit via le fetcher global du `SWRProvider` (`src/lib/swr-config.tsx:22`). **Aucun fetcher ne contourne cette vérification.** Conformité totale aux CLAUDE_Rules.
>
> Les Findings ci-dessous portent sur la **gestion d'erreur côté consommateur**, la **stabilité des clés SWR** et les **stratégies de revalidation**.

### F-LOG-010 — Gestion d'erreur silencieuse dans 40 hooks SWR sur 59

- **Axe** : LOG
- **Sévérité** : Medium
- **Description** : 40 des 59 appels `useSWR` inventoriés (≈ 68 %) ne destructurent pas la valeur `error` retournée par SWR ou ne l'exposent pas au consommateur. Lorsque le fetcher lève une exception (réseau, 401, 500), l'UI ne reçoit aucun signal d'erreur et affiche soit un état de chargement infini (`isLoading` reste `true` indéfiniment après épuisement des retries), soit un état vide trompeur (`data === undefined` interprété comme « aucun résultat »). Cela concerne l'intégralité des composants de la page de tâche (D.3 #25–#49), les pages App Router (D.4 #50–#59) et 4 hooks sous `src/hooks/` (#5, #6, #7, #20).
- **Preuve** : `src/components/task/task-dependencies.tsx:106` (pas de destructuration `error`) ; `src/components/task/activity-feed.tsx:55` (seul `isLoading` destructuré) ; `src/app/(platform)/goals/page.tsx:53` (idem) ; `src/app/(platform)/notes/page.tsx:56` (idem) ; inventaire complet en annexe D colonne « Gestion erreur explicite ».
- **Impact** : L'utilisateur ne reçoit aucun feedback en cas d'échec réseau ou d'erreur serveur sur ces 40 endpoints. Il ne peut pas distinguer « chargement en cours » de « échec silencieux ». Sur mobile avec connexion instable, l'expérience est particulièrement dégradée : les sections de la page de tâche (dépendances, activité, commentaires, pièces jointes, sous-tâches, champs personnalisés) restent vides sans explication.
- **Recommandation** : Pour chaque appel `useSWR`, destructurer `error` et afficher un composant d'erreur contextuel (message + bouton « Réessayer » appelant `mutate()`). Créer un composant réutilisable `<SWRErrorFallback error={error} onRetry={mutate} />` pour uniformiser le pattern. Prioriser les composants de la page de tâche (D.3) et les pages App Router (D.4) qui constituent les parcours principaux.

### F-LOG-011 — Polling `useRunningTimer` à 5 s actif en permanence dans le layout plateforme

- **Axe** : LOG
- **Sévérité** : Medium
- **Description** : Le hook `useRunningTimer` (`src/hooks/use-time-entries.ts:31`) est configuré avec `refreshInterval: 5000` (polling toutes les 5 secondes). Il est consommé par `TimerButton` (`src/components/time-tracking/timer-button.tsx:22`), lui-même monté dans `TopBar` (`src/components/layout/top-bar.tsx:47`), qui fait partie du layout `(platform)` (`src/app/(platform)/layout.tsx:24`). En conséquence, l'endpoint `/api/time-entries/timer` est sollicité toutes les 5 secondes pour **tous les utilisateurs authentifiés**, qu'un timer soit actif ou non. Cela représente 720 requêtes/heure/utilisateur vers cet endpoint.
- **Preuve** : `src/hooks/use-time-entries.ts:35` (`refreshInterval: 5000`) ; `src/components/layout/top-bar.tsx:47` (`<TimerButton />`) ; `src/app/(platform)/layout.tsx:24` (`<TopBar />`)
- **Impact** : Charge serveur proportionnelle au nombre d'utilisateurs connectés × 12 req/min. Sur Vercel serverless, chaque invocation consomme du temps de fonction. Sur mobile, le polling empêche la mise en veille réseau et consomme de la batterie. Le `dedupingInterval` global de 2 s n'atténue pas ce polling car l'intervalle de 5 s est supérieur.
- **Recommandation** : Conditionner le polling à la présence d'un timer actif : `refreshInterval: runningTimer ? 5000 : 0`. Alternativement, augmenter l'intervalle à 30 s quand aucun timer n'est actif et ne passer à 5 s qu'après détection d'un timer en cours. Envisager un mécanisme push (WebSocket ou Server-Sent Events) pour les mises à jour de timer plutôt qu'un polling agressif.

### F-LOG-012 — Clé SWR toujours active sans garde `null` dans `useCustomFieldsSidebar`

- **Axe** : LOG
- **Sévérité** : Medium
- **Description** : Le hook `useCustomFieldsSidebar` (`src/hooks/use-custom-fields.ts:32-37`) accepte `taskId: string` et `workspaceId: string` en paramètres et construit deux clés SWR sans vérification de valeur : `` `/api/custom-fields?workspaceId=${workspaceId}` `` et `` `/api/tasks/${taskId}/custom-fields` ``. Si `workspaceId` ou `taskId` sont des chaînes vides (ce qui est possible avant l'hydratation ou lors d'un rendu conditionnel), les requêtes partent vers des URLs invalides (`/api/custom-fields?workspaceId=` ou `/api/tasks//custom-fields`), provoquant des erreurs 400/404 silencieuses (l'erreur n'est pas destructurée, cf. F-LOG-010).
- **Preuve** : `src/hooks/use-custom-fields.ts:33` (clé sans garde : `` `/api/custom-fields?workspaceId=${workspaceId}` ``) ; `src/hooks/use-custom-fields.ts:37` (clé sans garde : `` `/api/tasks/${taskId}/custom-fields` ``)
- **Impact** : Requêtes réseau inutiles vers des endpoints qui retourneront une erreur. Pollution du cache SWR avec des entrées à clé invalide. Erreurs silencieuses dans la console réseau du navigateur.
- **Recommandation** : Ajouter des gardes conditionnelles sur les clés : `workspaceId ? \`/api/custom-fields?workspaceId=${workspaceId}\` : null` et `taskId ? \`/api/tasks/${taskId}/custom-fields\` : null`. Ce pattern est déjà utilisé correctement dans la majorité des autres hooks (cf. `useBudgetCategories`, `useFavorites`, `useSpaces`).

### F-LOG-013 — Clé SWR toujours active dans `useTimeEntries` sans `taskId`

- **Axe** : LOG
- **Sévérité** : Medium
- **Description** : Le hook `useTimeEntries` (`src/hooks/use-time-entries.ts:13`) accepte un `taskId` optionnel. Lorsqu'il est appelé sans argument (comme dans `src/app/(platform)/time-tracking/page.tsx:46`), la clé SWR résultante est `` `/api/time-entries?` `` (chaîne avec un `?` terminal et aucun paramètre). Cette clé est toujours active (jamais `null`), ce qui déclenche un fetch immédiat au montage. Le problème est double : (1) la clé avec `?` terminal est une URL non canonique qui pourrait ne pas être dédupliquée correctement si un autre composant utilise `/api/time-entries` sans le `?` ; (2) le fetch se déclenche même si la page n'a pas encore les informations nécessaires pour filtrer les résultats.
- **Preuve** : `src/hooks/use-time-entries.ts:17` (`` `/api/time-entries?${params.toString()}` `` — `params` vide produit une chaîne vide après `?`) ; `src/app/(platform)/time-tracking/page.tsx:46` (`useTimeEntries()` sans argument)
- **Impact** : Requête systématique au montage de la page time-tracking, même si l'utilisateur n'a pas encore sélectionné de filtre. La clé non canonique (`/api/time-entries?` vs `/api/time-entries`) empêche la déduplication avec d'éventuels autres consommateurs utilisant la même route sans query string.
- **Recommandation** : Normaliser la construction de la clé : `const key = params.toString() ? \`/api/time-entries?${params.toString()}\` : "/api/time-entries"`. Alternativement, si le fetch sans filtre est intentionnel (charger toutes les entrées de l'utilisateur), documenter ce comportement et s'assurer que l'API pagine les résultats.

### F-LOG-014 — Absence de `revalidateOnFocus: false` sur les hooks à polling explicite

- **Axe** : LOG
- **Sévérité** : Medium
- **Description** : Cinq hooks configurent un `refreshInterval` explicite pour maintenir les données à jour par polling : `useNotifications` (30 s), `useReminders` (60 s), `useRunningTimer` (5 s), `DashboardPage` (300 s), `MyTasksPage` (30 s). Aucun de ces hooks ne désactive `revalidateOnFocus` (hérité à `true` du provider global). En conséquence, lorsque l'utilisateur revient sur l'onglet du navigateur, SWR déclenche une revalidation immédiate **en plus** du polling déjà actif. Sur mobile, les changements de focus sont fréquents (switch d'application, notification push, clavier virtuel) et provoquent des rafales de requêtes redondantes avec le prochain tick de polling.
- **Preuve** : `src/hooks/use-notifications.ts:27` (`refreshInterval: 30000`, pas de `revalidateOnFocus: false`) ; `src/hooks/use-time-entries.ts:35` (`refreshInterval: 5000`, pas de `revalidateOnFocus: false`) ; `src/app/(platform)/dashboard/page.tsx:79` (`refreshInterval: 300000`, pas de `revalidateOnFocus: false`) ; `src/app/(platform)/my-tasks/page.tsx:57` (`refreshInterval: 30000`, pas de `revalidateOnFocus: false`)
- **Impact** : Requêtes réseau redondantes lors des retours de focus, particulièrement sur mobile. Pour `useRunningTimer` (5 s), la revalidation on-focus est quasi-inutile car le prochain poll arrive dans ≤ 5 s. Pour `useNotifications` et `MyTasksPage` (30 s), le doublon est moins critique mais reste un gaspillage sur connexions lentes.
- **Recommandation** : Ajouter `revalidateOnFocus: false` aux hooks qui définissent déjà un `refreshInterval`. Le polling assure la fraîcheur des données ; la revalidation on-focus est redondante dans ce cas. Pattern recommandé : `{ refreshInterval: 30000, revalidateOnFocus: false }`.

### F-LOG-015 — Couverture Zod minoritaire sur les routes acceptant des entrées (~26 %)

- **Axe** : LOG
- **Sévérité** : Medium
- **Description** : Sur 96 fichiers `route.ts`, seuls 25 (≈ 26 %) appliquent un schéma Zod via `safeParse` ou `.parse()` avant tout accès Prisma. Les autres routes acceptant un body ou des query params reposent soit sur de la validation manuelle (`typeof`, `if (!field)`, regex ad hoc), soit sur aucune validation explicite — le body JSON est destructuré et passé tel quel à `prisma.*.create/update`. ### F-LOG-022 — Code `401 Unauthorized` détourné pour signaler un PIN tâche incorrect (verify-pin)

- **Axe** : LOG
- **Sévérité** : Medium
- **Description** : La route `/api/tasks/[taskId]/verify-pin` (POST) renvoie `{ valid: false, error: "Code incorrect" }` avec `status: 401` lorsque le PIN à 4 chiffres soumis ne correspond pas au PIN de la tâche verrouillée (ligne 53-57). Le code HTTP `401 Unauthorized` est, par RFC 9110 §15.5.2, réservé aux requêtes qui ne fournissent **pas** de credentials d'authentification valides ; il déclenche, dans la convention NextAuth/SWR de l'application, le contournement automatique « si 401 → rediriger vers `/login` » (cf. middleware applicatif et fetchers SWR). Le code adapté à RFC 9110 §15.5.4 est `403 Forbidden` (« le serveur a compris la requête mais refuse de l'autoriser »). De plus, la même route renvoie `{ valid: true }` en `200` lorsque le PIN est correct ou que la tâche n'est pas verrouillée (ligne 50, 60), ce qui est cohérent ; mais elle utilise simultanément deux contrats de réponse (`{ valid: boolean }` 200 vs `{ valid: boolean, error }` 401) pour un même endpoint, ce qui complexifie le typage côté client.
- **Preuve** : `src/app/api/tasks/[taskId]/verify-pin/route.ts:53-57` (status 401 sur PIN incorrect) ; `src/app/api/tasks/[taskId]/verify-pin/route.ts:48-50` (status 200 avec `valid: true` sur tâche non verrouillée) ; `src/app/api/tasks/[taskId]/verify-pin/route.ts:59-60` (status 200 avec `valid: true` sur PIN correct).
- **Impact** : (i) Si un consommateur SWR de cette route (ex. `task-detail-page-client.tsx` lors de la modale de déverrouillage) hérite du fetcher global qui intercepte les 401 pour rediriger vers `/login`, l'utilisateur est éjecté de l'application à la première erreur de saisie PIN, alors qu'il est en réalité connecté ; (ii) un script tiers ou un outil de monitoring qui agrège les 401 par utilisateur croira observer des déconnexions intempestives ; (iii) la sémantique mélangée 200/401 pour un même endpoint complique le typage TypeScript côté client (le type discriminé `{ valid: true } | { valid: false, error: string }` doit être croisé manuellement avec le statut HTTP). À noter que l'application utilise plusieurs fetchers locaux qui ne déclenchent pas la redirection automatique sur 401, ce qui réduit le risque pratique mais ne corrige pas la non-conformité RFC.
- **Recommandation** : Utiliser `status: 403` pour le PIN incorrect (« vous êtes authentifié mais vous n'avez pas le droit d'accéder à cette tâche tant que le PIN n'est pas fourni correctement »). Alternativement, conserver `status: 200` pour tous les cas et ne refléter la réussite/l'échec que via le booléen `valid` du body (le client connaît déjà la sémantique métier du flag). Documenter le contrat retenu dans un commentaire en tête du fichier. Cette modification doit être croisée avec les CLAUDE_Rules concernant les redirections 401 (cf. fetcher global SWR `src/lib/swr-config.tsx`).

### F-LOG-025 — Synchronisation répétée de `setWorkspaceId` via `useEffect` : pattern à externaliser

- **Axe** : LOG
- **Sévérité** : Medium
- **Description** : Cinq composants exposent le même `useEffect` à 3-4 lignes dont l'unique rôle est de pousser le `workspaceId` reçu en prop dans le store global `useModal` via `setWorkspaceId(workspaceId)`. Le pattern est : `useEffect(() => { setWorkspaceId(workspaceId); }, [workspaceId, setWorkspaceId]);`. Cette duplication structurelle est un cas-école d'**externe-sync** (synchronisation d'un état applicatif local vers un store externe) qui pourrait être traité (a) par un `useEffect` unique placé dans le layout `(platform)` qui dérive le `workspaceId` du segment de route, (b) par un sélecteur ou un middleware du store `useModal` qui lit directement le `workspaceId` du contexte React (`useWorkspace`), ou (c) par une prop pull plutôt qu'une prop push (les consommateurs du store récupèrent `workspaceId` via le contexte au lieu que chaque vue le pousse). Le doc React (« You Might Not Need an Effect ») classe explicitement ce pattern parmi ceux à éviter.
- **Preuve** : `src/components/views/board-view.tsx:33-36` ; `src/components/views/list-view.tsx:34-37` ; `src/components/views/calendar-view.tsx:52-55` ; `src/components/views/gantt-view.tsx:39` ; `src/app/(platform)/calendar/page.tsx:117-119` (variante avec garde `if (currentWorkspace?.id)`).
- **Impact** : (i) Duplication de code structurelle : tout changement de l'API de `setWorkspaceId` (renommage, ajout d'argument, déprécation) impose 5 éditions ; (ii) chaque vue ré-exécute un `useEffect` au montage et à chaque changement de `workspaceId`, déclenchant un re-render inutile du store et de tous ses consommateurs ; (iii) la cohérence du store dépend de la discipline du développeur — toute nouvelle vue oubliant ce `useEffect` n'apparaît pas comme un défaut statique mais provoque un état désynchronisé (les modales ouvertes depuis cette vue ne connaissent pas le `workspaceId`) ; (iv) sur navigation client-side entre deux vues d'une même liste, deux composants synchronisent successivement la même valeur, déclenchant deux effets et donc deux notifications du store.
- **Recommandation** : Centraliser la synchronisation : (a) déplacer le `useEffect` dans `src/app/(platform)/layout.tsx` ou dans un composant `<WorkspaceModalSync />` placé dans ce layout, qui dérive le `workspaceId` de l'URL via `useParams()` ou du contexte `useWorkspace` ; (b) supprimer les 5 `useEffect` redondants des vues. Alternative plus radicale : faire en sorte que `useModal` dérive lui-même le `workspaceId` du contexte `useWorkspace` au moment de l'ouverture d'une modale, supprimant le besoin de synchronisation explicite.

### F-LOG-026 — État dérivé synchronisé via `useEffect` au lieu d'un état local au formulaire (anti-pattern « state from props »)

- **Axe** : LOG
- **Sévérité** : Medium
- **Description** : Sept composants utilisent le pattern `useState` + `useEffect` pour initialiser ou réinitialiser un état local de formulaire à partir d'une prop (ou d'un `data` SWR) lorsque la prop change. Ce pattern est l'anti-pattern explicitement décrit dans la documentation React sous « **Adjusting some state when a prop changes** » (à éviter). La forme rencontrée est : `const [x, setX] = useState(initial); useEffect(() => { setX(propX); }, [propX]);`, ce qui force un double rendu (premier rendu avec l'ancien `x`, puis effet, puis second rendu avec le nouveau `x`) au lieu d'un seul rendu cohérent. Cas relevés :
  - **`src/components/task/task-detail-content.tsx:48-53`** : `setTitleValue(task.title)` et `setDescriptionValue(task.description ?? "")` dans un `useEffect([task])`. Provoque un flicker entre les anciennes valeurs et les nouvelles à chaque mutation SWR de la tâche, et écrase silencieusement les modifications en cours si l'utilisateur édite pendant qu'un `mutate()` arrive.
  - **`src/app/(platform)/notes/page.tsx:80-91`** : `setEditTitle/EditContent/EditColor` dans un `useEffect([selectedId])` (avec `eslint-disable-line react-hooks/exhaustive-deps` qui masque la dépendance à `selectedNote`). L'autosave et la lecture initiale partagent le même état, ce qui complexifie la cohérence (cf. F-LOG-027).
  - **`src/app/(platform)/settings/page.tsx:74-76`** : `setName(sessionUser.name)` dans un `useEffect([sessionUser?.name])`. Le champ « Nom » est dérivable directement du `sessionUser` et ne nécessite un état local qu'à partir de la première édition utilisateur.
  - **`src/components/budget/budget-form-dialog.tsx:74-91`** : 5 `setX` dans un `useEffect([budget, open])` pour pré-remplir le formulaire d'édition. La logique « si édition → préremplir, sinon → reset » mélange deux responsabilités (initialisation et reset) dans un seul effet, et exécute les `setX` après le premier rendu, provoquant un flash visuel avec les valeurs vides.
  - **`src/components/task/create-task-dialog.tsx:62-68`** : `setListId(defaultListId)` ou `setListId(allLists[0].id)` selon `open` et la disponibilité des listes. Le calcul peut être effectué pendant le rendu (ou via `useMemo`) : `const initialListId = defaultListId ?? allLists[0]?.id ?? ""`.
  - **`src/components/task/quick-create-task.tsx:44-48`** : idem, `setNewListId(allLists[0].id)` dans un `useEffect([allLists, newListId])`.
- **Preuve** : `src/components/task/task-detail-content.tsx:48-53` ; `src/app/(platform)/notes/page.tsx:80-91` (avec `eslint-disable-line` à la ligne 92) ; `src/app/(platform)/settings/page.tsx:74-76` ; `src/components/budget/budget-form-dialog.tsx:74-91` ; `src/components/task/create-task-dialog.tsx:62-68` ; `src/components/task/quick-create-task.tsx:44-48`.
- **Impact** : (i) Double rendu systématique à chaque changement de la prop synchronisée — sur la page de détail de tâche, chaque mutation SWR provoque un cycle render → effet → re-render qui rend perceptible le flash sur les champs « Titre » et « Description » ; (ii) bug de course : si l'utilisateur édite le champ pendant qu'un `mutate()` arrive (ouverture d'un commentaire, action assignee, etc.), le `useEffect` écrase la saisie en cours par les anciennes valeurs serveur ; (iii) la directive `eslint-disable-line react-hooks/exhaustive-deps` (présente dans `notes/page.tsx:92`) masque structurellement le risque — le linter ne signalera plus si une dépendance manque réellement à l'avenir ; (iv) anti-pattern documenté par React dont la persistance dans 7 fichiers indique l'absence de revue de code centrée sur ces idiomes.
- **Recommandation** : Pour chaque cas, appliquer le contournement recommandé par la documentation React :
  - **Cas avec `key`** : pour les dialogues d'édition (`budget-form-dialog`, `create-task-dialog`), réinitialiser l'état du formulaire en passant une `key={budget?.id ?? "new"}` au composant — le démontage/remontage gère naturellement la réinitialisation, sans `useEffect`.
  - **Calcul direct pendant le rendu** : pour `task-detail-content`, gérer le mode « édition » via un état local séparé (`editingTitle: boolean`) qui ne se synchronise qu'à l'entrée en mode édition (`onClick={() => { setTitleValue(task.title); setEditingTitle(true); }}`). En mode lecture, afficher directement `task.title` sans copie locale.
  - **Dérivation par `useMemo`** : pour `create-task-dialog` et `quick-create-task`, calculer `const initialListId = defaultListId ?? allLists[0]?.id ?? ""` pendant le rendu et utiliser un `useState(initialListId)` (initialisation paresseuse) plutôt qu'un `useEffect`.
  - **Source de vérité unique** : pour `settings/page.tsx`, soit lire directement `sessionUser?.name` en lecture seule, soit n'avoir un état local qu'à partir de la première saisie utilisateur (et pas avant).

F-LOG-026 pour le pattern `key`). ### F-LOG-029 — `useEffect` qui écrit dans `localStorage` une clé jamais relue (code mort) — `runningTimer`

- **Axe** : LOG
- **Sévérité** : Medium
- **Description** : Le `useEffect` de `src/components/time-tracking/timer-button.tsx:32-46` écrit dans `localStorage` la clé `"runningTimer"` à chaque changement de `runningTimer` (objet retourné par `useRunningTimer()`), avec le commentaire « Persist timer state in localStorage for instant display on navigation ». De plus, l'écriture est conditionnée par `runningTimer === null` (strictement) pour la suppression : si SWR retourne `undefined` (premier chargement), aucune branche ne s'exécute, et la clé persiste de la session précédente sans être ni nettoyée ni utilisée.
- **Preuve** : `src/components/time-tracking/timer-button.tsx:32-46` (écriture `localStorage.setItem("runningTimer", ...)`) ; `src/components/time-tracking/timer-button.tsx:107` (autre écriture `localStorage.removeItem` dans `handleStop`) ; `commande: grep -rn "runningTimer" src --include='*.ts' --include='*.tsx' | grep localStorage` ne révèle **aucun** `localStorage.getItem("runningTimer")` dans le codebase ; `src/hooks/use-time-entries.ts:30-43` (`useRunningTimer` lit via SWR, jamais via localStorage).

- **Axe** : LOG
- **Sévérité** : Medium
- **Description** : Deux `useEffect` ont une liste de dépendances incluant un objet retourné par un hook tiers ou une fonction qui peut changer d'identité à chaque rendu, ce qui dégrade la stabilité de l'effet :

  - **`src/lib/swr-config.tsx:29-32`** : le `useEffect(() => { refreshTokenCache(); }, [])` au montage du `SWRProvider` rafraîchit le token mobile, mais avec un tableau de dépendances vide et un appel à `refreshTokenCache` qui peut lever (cf. `import { refreshTokenCache }`) ; aucune capture d'erreur n'entoure l'appel. - **Preuve** : `src/app/(platform)/dashboard/calendar-settings/page.tsx:113` (`useEffect`) et `:140` (`}, [searchParams, toast, fetchGoogleStatus]);`) ; `src/lib/swr-config.tsx:29-32` (`useEffect(() => { refreshTokenCache(); }, []);` sans `try/catch` ni `.catch()`).
- **Impact** : (i) **`calendar-settings/page.tsx`** : `fetchGoogleStatus()` est rappelé à chaque navigation client-side qui change un query param de la page (par exemple si un `?tab=calendar` est ajouté ailleurs), gaspillant un appel HTTP ; le `searchParams` instable provoque potentiellement plusieurs déclenchements en cascade selon la version de Next.js — la garde `if (success === "google_connected")` ne couvre pas ce déclenchement intempestif puisque `fetchGoogleStatus()` est dans la branche `if (success ===)`, mais les conditions d'erreur (`if (error)`) n'ont pas de garde et émettent un toast d'erreur à chaque montage si l'URL contient `?error=...`, **même après que l'utilisateur a déjà vu et fermé le toast** une fois (pas de mécanisme de déduplication ; à chaque retour sur la page avec ces params, un nouveau toast est émis) ; (ii) **`swr-config.tsx`** : un échec silencieux de `refreshTokenCache` empêche le bootstrap de l'authentification mobile sans signal côté UI, dégradant le diagnostic — l'utilisateur voit un état « non authentifié » sans pouvoir distinguer une session expirée d'un crash du refresh.
- **Recommandation** :

F-LOG-028).
  - **`swr-config.tsx`** : ajouter un `.catch((err) => console.error("[SWRProvider] refreshTokenCache failed:", err))` pour rendre l'échec observable côté logs, sans propager d'erreur à l'utilisateur. Évaluer si ce refresh est nécessaire au montage ou si SWR peut le déléguer à la première requête authentifiée (lazy auth).
