### F-UX-001 — Suppression de tâche déclenchée sans aucune confirmation (`<DropdownMenuItem>` direct → `DELETE /api/tasks/{id}`)

- **Axe** : UX
- **Sévérité** : High
- **Description** : Dans le menu contextuel des tâches (`src/components/task/task-action-menu.tsx:226-233`), l'item « Supprimer » exécute directement `handleDelete` (`src/components/task/task-action-menu.tsx:119-130`), qui appelle `deleteTask(taskId)` (hook `src/hooks/use-tasks.ts:117-132`) puis `DELETE /api/tasks/{taskId}`. Aucun `<AlertDialog>` n'enveloppe le déclencheur, aucun `window.confirm` n'est appelé en amont (`grep_search 'confirm\\(' src/components/task/**` = 0), et la route serveur (`src/app/api/tasks/[taskId]/route.ts`, méthode `DELETE`) procède à la suppression cascade en base sans contre-confirmation. Le seul garde-fou existant est le **verrouillage par PIN** via la propriété `task.locked` (`src/components/task/task-detail-content.tsx:87-118`), qui est **opt-in et désactivé par défaut** : tant qu'un utilisateur n'a pas explicitement verrouillé une tâche, un seul clic dans le menu suffit à la supprimer. Le menu déclencheur (`<MoreHorizontal />` à 7×7 px, cf. F-UI-006) est de surcroît affiché par défaut sur **chaque tâche** des vues `list-view`, `board`, `calendar`, `gantt`, `my-tasks`, `dashboard` — la cible du « clic accidentel » est donc maximale.

  À titre comparatif, la même application enveloppe **bien** la suppression de budget dans un `<AlertDialog>` titré « Supprimer le budget ? » avec description « Cette action est irréversible » (`src/components/budget/budget-card.tsx:82-110`). L'incohérence entre le traitement budget (haut niveau de friction) et le traitement tâche (zéro friction) confirme que l'absence de confirmation côté tâche est un **oubli de parcours**, pas un choix produit.

- **Preuve** : `src/components/task/task-action-menu.tsx:119` (`handleDelete`), `src/components/task/task-action-menu.tsx:226` (`<DropdownMenuItem onClick={handleDelete} ...>`) ; `src/hooks/use-tasks.ts:117-132` (`useDeleteTask`) ; `src/app/api/tasks/[taskId]/route.ts` (méthode `DELETE`, cf. annexe C ligne `/api/tasks/[taskId]`) ; `src/components/budget/budget-card.tsx:82-110` (référence comparative) ; `commande: grep_search 'confirm\\(|<AlertDialog' --include='src/components/task/**'` (0 occurrence dans le module task).
- **Impact** : Perte de données utilisateur potentielle sur le parcours métier le plus utilisé (CRUD tâche). Une tâche supprimée par mégarde entraîne en cascade : suppression des sous-tâches, des commentaires, des time entries, du journal d'activité (cf. relations Prisma de `Task`). Aucun mécanisme de corbeille ni d'`undo` n'est observable dans le code (recherche `grep_search 'softDelete|deletedAt|trash' src/` = 0 dans le périmètre tâches). Le toast de confirmation post-suppression (`{ title: "Tâche supprimée" }`, ligne 123) ne propose **pas** d'action « Annuler ». Sur tablette et mobile (TalkBack/VoiceOver), le risque est aggravé par la zone tactile sous-dimensionnée du déclencheur (cf. F-UI-006). Cas typique : un utilisateur tente d'ouvrir le menu (`<MoreHorizontal />`), glisse accidentellement vers le bas et atterrit sur « Supprimer ». Severity_High car la définition retenue dans `design.md` > Data Models > Définitions de sévérité inclut explicitement « Risque de perte de données » comme critère.
- **Recommandation** : Envelopper l'item « Supprimer » dans un `<AlertDialog>` (composant `src/components/ui/alert-dialog.tsx` déjà disponible) avec titre `« Supprimer la tâche ? »`, description rappelant l'irréversibilité et la cascade des données associées (commentaires, sous-tâches, time entries), et bouton d'action en variante destructive. Réutiliser le pattern de `src/components/budget/budget-card.tsx:82-110` pour cohérence. Optionnellement, ajouter un toast post-suppression avec action « Annuler » exploitant un soft-delete (chantier hors audit, à matérialiser dans `tasks-remediation.md`).

### F-UX-002 — Changement de workspace via le sélecteur sidebar laisse l'URL pointant sur l'ancien workspace (perte de contexte de navigation)

- **Axe** : UX
- **Sévérité** : High
- **Description** : Le sélecteur de workspace de la barre latérale (`src/components/layout/sidebar.tsx:148-160`) appelle uniquement `setCurrentWorkspace(ws)` (provider `src/providers/workspace-provider.tsx:60-67`) lors du clic sur un autre espace. **Aucun `router.push`** n'est déclenché (vérifié : `grep_search 'setCurrentWorkspace\\(ws\\)|router\\.push' src/components/layout/sidebar.tsx` rapporte les deux symboles côte à côte mais l'item du dropdown n'invoque que `setCurrentWorkspace`, ligne 152). Conséquence : si l'utilisateur se trouve sur une URL contenant `workspaceId` dans le path — typiquement `/workspace/{ancienId}/space/{spaceId}/list/{listId}/list-view`, `/workspace/{ancienId}` ou `/workspace/{ancienId}/settings` — le clic sur un autre workspace dans la sidebar **change l'état React** (`currentWorkspace`), met à jour `localStorage` et déclenche le re-rendu des composants qui dépendent de `useWorkspace().currentWorkspace`, mais **ne modifie pas l'URL**. L'utilisateur se retrouve dans un état hybride :

  - Le contenu central — par exemple `src/app/(platform)/workspace/[workspaceId]/page.tsx:24-26` qui lit `useParams<{ workspaceId }>` et fait `useSWR('/api/workspaces/{paramsId}')` — continue d'afficher l'ancien workspace.
  - La sidebar (espaces, listes, favoris) — qui consomme `useSpaces(currentWorkspace?.id)` (`src/components/layout/sidebar.tsx:65`) — affiche désormais les espaces du **nouveau** workspace.
  - Les widgets globaux du dashboard / `my-tasks` / `calendar` / `notes` qui se basent sur `currentWorkspace?.id` (cf. `grep_search currentWorkspace` exécuté pendant la tâche, ≥ 50 occurrences) chargent les données du nouveau workspace.
  - Toute action de partage d'URL (ouverture d'un onglet, copier-coller du lien) référence l'ancien workspace.

  La désynchronisation est donc **structurellement durable** : tant que l'utilisateur ne navigue pas via la sidebar (clic sur un favori ou une liste), l'URL et l'état restent décorrélés. Cas particulier : sur la page `/workspace/{ancienId}/settings`, l'utilisateur peut modifier les paramètres d'un workspace dont il a cru sortir (le retour API `PATCH /api/workspaces/{ancienId}` est calculé sur `params.workspaceId`, pas sur `currentWorkspace.id`).

- **Preuve** : `src/components/layout/sidebar.tsx:148-160` (item dropdown ne fait que `setCurrentWorkspace(ws)` sans `router.push`) ; `src/providers/workspace-provider.tsx:60-67` (le `setCurrentWorkspace` ne navigue pas) ; `src/app/(platform)/workspace/[workspaceId]/page.tsx:24-30` (lecture du workspace via `useParams`, déconnectée du provider) ; `src/app/(platform)/workspace/[workspaceId]/settings/page.tsx` (même schéma `useParams`). Comparaison interne : la page `(platform)/search/page.tsx:198-201` utilise au contraire `currentWorkspace.id` pour construire l'URL de redirection, ce qui peut produire un mismatch si l'utilisateur a switché juste avant de cliquer sur un résultat.
- **Impact** : Blocage utilisateur majeur sur le parcours « navigation entre workspaces », qui est un parcours critique de la spec (clause 6.1 de `tasks.md`). L'utilisateur peut : (a) modifier les paramètres ou inviter des membres dans le mauvais workspace ; (b) voir un dashboard qui combine des données de deux workspaces différents (sidebar = nouveau, contenu = ancien) ; (c) partager une URL « cassée » par rapport à son intention. Au-delà de l'inconfort, c'est un risque de **fuite de contexte** : si l'utilisateur croit gérer le workspace B et envoie un message en se basant sur ce que la sidebar affiche, il peut exposer des informations du workspace A. Severity_High car correspond à la définition « Blocage utilisateur majeur sur un parcours critique » du `design.md` > Data Models > Définitions de sévérité.
- **Recommandation** : Dans le `onClick` de l'item dropdown (`src/components/layout/sidebar.tsx:152`), enchaîner `setCurrentWorkspace(ws)` avec une navigation explicite — par exemple `router.push('/dashboard')` ou, si l'on souhaite préserver la vue, détecter la présence de `workspaceId` dans `pathname` (via `usePathname`) et rediriger vers la même vue du nouveau workspace (`/workspace/${ws.id}` ou `/dashboard`). Alternative plus robuste : faire dériver `currentWorkspace` du `workspaceId` présent dans l'URL plutôt que d'un état React indépendant ; le `localStorage` ne sert alors qu'à préselectionner le workspace au premier chargement. Cette deuxième approche aligne le modèle mental « URL = source de vérité » courant en App Router.

### F-UX-003 — `callbackUrl` ignoré par les formulaires de connexion et d'inscription : retour systématique au `/dashboard` après authentification

- **Axe** : UX
- **Sévérité** : High
- **Description** : Le formulaire de connexion (`src/components/auth/login-form.tsx:25-128`) lit `useSearchParams()` mais n'utilise le query param que pour pré-remplir l'email (`src/components/auth/login-form.tsx:30`, `setEmail(searchParams.get("email") ?? "")`). Aucune lecture de `callbackUrl`. À la fin du flow web (ligne 70) comme du flow Capacitor (ligne 119), le code redirige **toujours** vers `/dashboard` via `router.push("/dashboard")`. Le formulaire d'inscription suit exactement le même schéma (`src/components/auth/register-form.tsx:88` et `:118` : `router.push("/dashboard")` en dur). Or :

  - La garde serveur du segment authentifié (`src/app/(platform)/layout.tsx#requireAuth`) appelle `redirect("/login")` (`src/lib/auth-helpers.ts:71`) sans construire de `callbackUrl` à partir de l'URL d'origine. Un utilisateur qui ouvre un lien profond `/workspace/{id}/space/{id}/list/{id}/list-view` sans session est donc redirigé vers `/login` **sans mémoire** du chemin demandé, puis envoyé sur `/dashboard` après authentification.
  - La page d'invitation (`src/app/invite/[token]/page.tsx:127-138`) construit explicitement une URL `signIn(undefined, { callbackUrl: \`/invite/${params.token}\` })` ainsi qu'un lien `register?callbackUrl=...&email=...`. L'intention produit est claire : l'utilisateur invité doit **revenir sur la page d'invitation** après login pour pouvoir cliquer sur « Accepter ». Mais comme `LoginForm` et `RegisterForm` ignorent `callbackUrl`, l'utilisateur atterrit sur `/dashboard` et perd le lien d'invitation. Le seul moyen de rejoindre l'espace de travail est alors de retourner manuellement sur le mail d'invitation et de cliquer à nouveau sur le lien (qui, grâce à la session active, déclenche cette fois le bon parcours `emailMatches`).
  - Le composant `LoginForm` utilise `signIn("credentials", { redirect: false })` (ligne 45-48) puis pilote la redirection lui-même : la valeur `callbackUrl` que NextAuth aurait gérée nativement avec `redirect: true` est donc à reconstruire manuellement par le code, ce qui n'est pas fait.

- **Preuve** : `src/components/auth/login-form.tsx:30` (lecture partielle de `searchParams`), `src/components/auth/login-form.tsx:70` et `:119` (`router.push("/dashboard")` en dur) ; `src/components/auth/register-form.tsx:88` et `:118` (idem) ; `src/lib/auth-helpers.ts:65-72` (`requireAuth` redirige sans `callbackUrl`) ; `src/app/invite/[token]/page.tsx:130-138` et `:147-153` (intention `callbackUrl` qui n'est pas honorée par les formulaires).
- **Impact** : Blocage utilisateur sur deux parcours critiques distincts. (1) **Parcours invitation** (clause 6.1 « navigation workspaces ») : l'utilisateur invité ne peut pas accepter l'invitation après une connexion via le formulaire ; le parcours « cliquer sur le lien e-mail → se connecter → accepter » se brise au milieu et l'utilisateur se retrouve sur le dashboard de son **propre** workspace par défaut, alors qu'il pensait rejoindre celui de l'invitant. (2) **Parcours liens profonds** : tout partage de lien `clickup-clone.app/workspace/X/space/Y/list/Z` à un collègue désauthentifié devient inopérant ; après login, le collègue perd le lien et doit naviguer manuellement. Sur mobile Capacitor, le coût de re-navigation est multiplié. Severity_High en application de la définition « Blocage utilisateur majeur sur un parcours critique » : un utilisateur invité ne peut accomplir le parcours « rejoindre un workspace » de manière fluide.
- **Recommandation** : (i) Dans `LoginForm.onSubmit`, lire `searchParams.get("callbackUrl")` et `decodeURIComponent` la valeur ; valider l'origine relative (commencer par `/`, refus des URLs absolues pour éviter l'open-redirect) ; remplacer `router.push("/dashboard")` par `router.push(safeCallback ?? "/dashboard")`. (ii) Faire de même dans `RegisterForm.onSubmit`. (iii) Enrichir `requireAuth` (`src/lib/auth-helpers.ts:65-73`) pour propager le `pathname` courant au formulaire de connexion (par exemple via `redirect(\`/login?callbackUrl=${encodeURIComponent(pathname)}\`)`). (iv) Optionnellement, basculer `signIn("credentials", { redirect: true, callbackUrl })` côté web pour déléguer la redirection à NextAuth quand un `callbackUrl` est présent.

### F-UX-004 — Confirmations destructives bâties sur `window.confirm()` natif (5 sites) — incohérence avec le reste de l'application et inutilisable sur Capacitor / mobile

- **Axe** : UX
- **Sévérité** : Medium
- **Description** : Cinq parcours destructifs utilisent `window.confirm()` natif au lieu d'un `<AlertDialog>` du design system :

  - `src/components/finance/account-list.tsx:53` — suppression d'un compte financier et de **toutes** ses transactions (`"Supprimer ce compte et toutes ses transactions ?"`).
  - `src/components/finance/transaction-list.tsx:38` — suppression d'une transaction (`"Supprimer cette transaction ?"`).
  - `src/components/finance/category-manager.tsx:59` — suppression d'une catégorie de finance.
  - `src/components/finance/goal-list.tsx:27` — suppression d'un objectif financier.
  - `src/app/(platform)/calendar/page.tsx:458` — suppression d'une tâche depuis la vue calendrier (`"Supprimer cette tâche ?"`) — qui contourne par ailleurs le menu d'action standard et fait `fetch('/api/tasks/...')` directement, hors du hook `useDeleteTask`.

  Or l'application embarque un `<AlertDialog>` shadcn/ui (`src/components/ui/alert-dialog.tsx`) déjà utilisé pour la suppression de budget (`src/components/budget/budget-card.tsx:82-110`) et qui fournit un nom accessible, des animations, le focus trap, et la cohérence visuelle. Le recours à `window.confirm()` introduit trois problèmes simultanés :

  1. **Incohérence visuelle et de variants** au sens de la convention « usage cohérent des variants » de la tâche 5.1 : selon le module, l'utilisateur voit tantôt une modale Radix custom, tantôt une popup système blanche / brutaliste qui rompt la charte sombre.
  2. **Inutilisable / dégradé sur Capacitor Android** : la WebView Android peut afficher `confirm()` natif, mais le style et le comportement de focus diffèrent de ceux d'une AlertDialog Radix (pas de safe-area, pas d'animation, message tronqué sur petits écrans). Sur certaines configurations WebView, `confirm()` peut être bloqué ou mal restitué.
  3. **Texte de confirmation insuffisant** : `confirm("Supprimer cette transaction ?")` ne décrit ni l'irréversibilité, ni les effets en cascade (par exemple sur `account-list.tsx`, la suppression du compte cascade sur toutes les transactions, ce qui mériterait d'être visuellement amplifié).

- **Preuve** : `src/components/finance/account-list.tsx:53` ; `src/components/finance/transaction-list.tsx:38` ; `src/components/finance/category-manager.tsx:59` ; `src/components/finance/goal-list.tsx:27` ; `src/app/(platform)/calendar/page.tsx:458` ; comparatif `src/components/budget/budget-card.tsx:82-110` (pattern de référence à appliquer) ; `commande: grep_search 'confirm\\(' --include='src/**/*.tsx'` (5 sites listés ci-dessus).
- **Impact** : Défaut de robustesse et problème UX notable sur cinq parcours destructifs : incohérence visuelle entre modules (finance vs budget vs calendar vs task), expérience dégradée sur la cible mobile principale (Capacitor Android), libellés insuffisamment informatifs sur l'irréversibilité. Severity_Medium en application de la définition « Problème UX notable » et « défaut de robustesse » du `design.md` > Data Models > Définitions de sévérité.
- **Recommandation** : Migrer les cinq sites vers `<AlertDialog>` (`src/components/ui/alert-dialog.tsx`) en réutilisant le pattern de `budget-card.tsx`. Ajouter une description précisant les conséquences (cascade sur les transactions pour la suppression de compte, etc.). Pour la suppression de tâche dans `calendar/page.tsx:458`, préférer le hook `useDeleteTask` du module `task` afin d'unifier les comportements (et profiter de la confirmation à ajouter via F-UX-001). Optionnellement, ajouter une règle ESLint `no-restricted-globals` pour `confirm` afin d'empêcher la régression.

### F-UX-005 — Module finance : sept mutations avalent les erreurs serveur (`catch (error) { console.error(error); }`) sans aucun retour utilisateur

- **Axe** : UX
- **Sévérité** : Medium
- **Description** : L'ensemble des composants du module finance (`src/components/finance/*`) suit un pattern identique : la mutation est encadrée par un `try/catch`, les erreurs API sont propagées via `throw new Error(err.error || …)`, mais le `catch` se contente de `console.error(error)` **sans appeler `toast()` ni positionner d'état d'erreur visible**. Le composant ferme ensuite la modale ou réinitialise le formulaire dans le `finally` — l'utilisateur perçoit donc une apparente réussite alors que l'opération a été refusée par le serveur (`res.ok === false`). Les sept sites concernés :

  1. `src/components/finance/add-transaction-dialog.tsx:79-94` — création de transaction.
  2. `src/components/finance/add-account-dialog.tsx:69-84` — création de compte.
  3. `src/components/finance/add-goal-dialog.tsx:67-82` — création d'objectif.
  4. `src/components/finance/account-list.tsx:54-64` (DELETE compte cascade) et `:166-182` (PATCH compte) — suppression et édition.
  5. `src/components/finance/transaction-list.tsx:39-49` (DELETE) et `:171-185` (PATCH) — suppression et édition de transaction.
  6. `src/components/finance/goal-list.tsx:28-38` (DELETE), `:46-60` (POST contribution) et `:256-271` (PATCH édition) — trois mutations.
  7. `src/components/finance/category-manager.tsx:43-55` (POST), `:60-70` (DELETE), `:78-91` (PATCH édition) — trois mutations.

Optionnellement, ajouter un toast positif (`title: "Transaction créée"`, etc.) pour cohérence avec le module task et le module workspace, qui le font déjà.

### F-UX-007 — Autosave de notes (`notes/page.tsx`) avale silencieusement les échecs réseau, exposant à la perte de contenu utilisateur sans avertissement

- **Axe** : UX
- **Sévérité** : Medium
- **Description** : La page `/notes` (`src/app/(platform)/notes/page.tsx:99-119`) implémente un autosave 800 ms après la dernière frappe (`scheduleAutosave`, ligne 122-131). En cas d'échec réseau (réponse non-OK, `fetch` rejeté), le `catch` est explicitement silencieux avec le commentaire `// silently retry on next change` (ligne 110). Or :

  - Le **retry n'est pas garanti** : si l'utilisateur cesse de modifier (typique : il quitte la page après avoir tapé), il n'y a pas de prochain trigger de save.
  - L'indicateur de statut visible en haut à droite (`{saving ? <Loader2 .../> : <span>Modifié … {format(...)}</span>}`, lignes 296-303) affiche le timestamp issu de **`selectedNote.updatedAt`** côté SWR, qui ne reflète **pas** la dernière tentative locale. - **Preuve** : `src/app/(platform)/notes/page.tsx:104-115` (corps de `saveNote`, `} catch { /* silently retry on next change */ }`) ; `src/app/(platform)/notes/page.tsx:135-144` (`flushSave` réutilise `saveNote`) ; `src/app/(platform)/notes/page.tsx:296-303` (indicateur visuel basé sur `updatedAt` côté serveur) ; `src/app/(platform)/notes/page.tsx:78-89` (`useEffect` qui écrase `editTitle/editContent` au changement de note).

L'absence de notification ne permet pas à l'utilisateur de relancer manuellement la sauvegarde ni de copier le contenu avant qu'il ne disparaisse à la sélection d'une autre note. Severity_Medium : le critère « Risque de perte de données » de Severity_High vise plutôt des problèmes structurels (migration manquante, requête destructive non protégée), tandis qu'ici la perte est conditionnelle (échec réseau + non-relance) ; en revanche les critères Medium « Problème UX notable / feedback utilisateur absent » et « Défaut de robustesse / absence de gestion d'erreur » sont tous deux satisfaits. Le finding pourrait être réévalué à High si une analyse complémentaire (axe LOG ou MOB) confirmait que le scénario WebView Capacitor déclenche fréquemment des `fetch` rejetés sans event utilisateur subséquent.
- **Recommandation** : (i) Dans le `catch` de `saveNote` (ligne 110), positionner un état `saveError` et émettre un toast destructif `{ title: "Sauvegarde échouée", description: "Vos modifications ne sont pas enregistrées. Reconnectez-vous au réseau et réessayez.", variant: "destructive" }`. (ii) Faire dépendre l'indicateur visuel ligne 296-303 d'une troisième valeur (`saved | saving | error`) plutôt que du seul booléen `saving`, et afficher l'horodatage de la **dernière sauvegarde réussie** (issu de `lastSaved.current`, pas de `selectedNote.updatedAt`). (iii) Ajouter une garde de type `beforeunload` qui prévient l'utilisateur si `editTitle/editContent` diffèrent de `lastSaved.current` (cf. pattern courant Web Workers).

### F-UX-008 — Aucun fallback global sur les erreurs SWR : les échecs de chargement laissent l'UI en état « vide » sans message ni toast

- **Axe** : UX
- **Sévérité** : Medium
- **Description** : Le `SWRConfig` global (`src/lib/swr-config.tsx:34-45`) configure `errorRetryCount: 2` mais **n'expose aucun `onError` global** vers le `Toaster`. Le fetcher partagé fait `if (!res.ok) throw new Error("Failed to fetch")` (ligne 24-26), sans aucune surface utilisateur. La quasi-totalité des hooks SWR du projet (`src/hooks/use-tasks.ts`, `use-spaces.ts`, `use-budgets.ts`, `use-time-entries.ts`, `use-favorites.ts`, `use-reminders.ts`, etc., cf. `notes/annexe-D-hooks-swr.md`) renvoient `data ?? []` ou `data ?? null`. Côté consommateurs, l'analyse statique des pages clés montre qu'**aucune ne lit `isError`** de manière différenciée — elles affichent directement le résultat ou un état `EmptyState`. Conséquence : si l'API renvoie 401/403/500, ou si la WebView Capacitor passe offline :

  - `src/app/(platform)/dashboard/page.tsx`, `src/app/(platform)/my-tasks/page.tsx`, `src/app/(platform)/calendar/page.tsx`, `src/app/(platform)/notes/page.tsx`, `src/app/(platform)/finance/page.tsx`, `src/app/(platform)/budget/page.tsx` et toute la chaîne `workspace/[workspaceId]/space/[spaceId]/list/[listId]/list-view|board|gantt|calendar/page.tsx` affichent un `EmptyState` (« Aucune tâche », « Aucun budget », etc.) — visuellement indistinguable d'un workspace vide.
  - L'utilisateur ne sait pas distinguer **« aucune donnée »** de **« la requête a échoué »**, et ne reçoit aucune indication d'action corrective (réessayer, vérifier la connexion, se réauthentifier).
  - L'`errorRetryCount: 2` n'apporte qu'une atténuation : après deux échecs, SWR cesse silencieusement.
  - Aucun `error.tsx` segment-level n'a été détecté dans le périmètre `src/app/(platform)/**` (`grep_search 'export default function Error|^export const ErrorBoundary' --include='src/app/(platform)/**'` = 0 — sera audité plus exhaustivement en tâche 6.3 sur la gestion d'erreur, mais l'absence est notée ici comme corollaire de l'absence de feedback de fetch).

  Quelques rares hooks **propagent** `isError` (ex. `src/hooks/use-tasks.ts:30`, `useTasks().isError`) mais aucun consommateur identifié ne le câble à l'UI ; la propagation est purement vestigiale.

- **Preuve** : `src/lib/swr-config.tsx:34-45` (configuration sans `onError`) ; `src/lib/swr-config.tsx:18-26` (fetcher unique qui throw `"Failed to fetch"`) ; `src/hooks/use-tasks.ts:18-30` (`isError` exposé mais non câblé dans les consommateurs) ; `src/app/(platform)/dashboard/page.tsx`, `src/app/(platform)/my-tasks/page.tsx` (consomment `useSWR` sans lire `error`) ; `commande: grep_search 'isError\\b|\\.error\\b' --include='src/app/(platform)/**/*.tsx'` (occurrences cantonnées aux composants finance et budget, jamais traduites en toast). Comparatif positif : la convention `toast({ ..., variant: "destructive" })` est utilisée pour les erreurs de **mutation** (`task-action-menu`, `team-list`, `member-list`, etc.) mais jamais pour les erreurs de **lecture**.
- **Impact** : Toute panne backend partielle (latence, 5xx ponctuel, expiration de session NextAuth, désynchronisation Capacitor) se traduit pour l'utilisateur par un état « rien ne se passe » ou « il n'y a rien ici » — qui peut conduire à des actions correctives erronées (recréer une tâche déjà existante, supposer la perte de données). Particulièrement préoccupant sur Capacitor Android où la WebView peut perdre la connectivité par à-coups : sans onError global, l'utilisateur ne sait pas qu'il consulte un état stale ou un état vide trompeur. Severity_Medium en application des critères « Problème UX notable / feedback utilisateur absent » et « Défaut de robustesse / état loading/empty manquant sur un parcours non critique » du `design.md` > Définitions de sévérité ; non éligible à High car le périmètre n'est pas un blocage de parcours critique unique mais une dégradation transverse.
- **Recommandation** : (i) Étendre `SWRConfig` (`src/lib/swr-config.tsx:34-45`) avec un `onError(error, key)` qui distingue (a) les 401 (déclencher un refresh de session ou un redirect login) et (b) les autres erreurs (émettre un `toast({ title: "Erreur de chargement", description: "Vérifiez votre connexion ou réessayez.", variant: "destructive" })`). Filtrer par clé pour éviter la cascade de toasts (un seul toast par clé/fenêtre temporelle). (ii) Standardiser les composants `EmptyState` pour distinguer visuellement « aucune donnée » de « erreur de chargement » (variant `error` avec bouton « Réessayer » qui appelle `mutate(key)`). (iii) Préparer la tâche 6.3 (gestion d'erreur, `error.tsx`/`not-found.tsx`) en s'appuyant sur les mêmes primitives de feedback.

### F-UX-009 — Absence d'un `global-error.tsx` racine : toute erreur survenant dans `app/layout.tsx` (providers, hydratation, polices) tombe sur la page d'erreur générique de Next.js

- **Axe** : UX
- **Sévérité** : Medium
- **Description** : Le routeur App Router de Next.js distingue deux fichiers d'erreur racine. (a) `app/error.tsx` enveloppe les pages mais **ne capture pas** les erreurs jetées **dans le layout racine lui-même** (cf. doc Next.js « Error Handling » : `error.js` operates at the page boundary, not at the root layout level). (b) `app/global-error.tsx` est précisément destiné à capturer ces erreurs, qu'elles proviennent du `RootLayout`, des providers (`ThemeProvider`, `SWRProvider`, `AuthProvider`, `WorkspaceProvider`, `ModalProvider` empilés ligne 110-122 de `src/app/layout.tsx`), des effets d'hydratation, ou des scripts injectés (`themeInitScript`, `fetchInterceptorScript`, `swRegisterScript`). Or **`src/app/global-error.tsx` est absent** (`find src/app -name "global-error.tsx"` = 0). Conséquence : si l'un des cinq providers throw au montage (par exemple `WorkspaceProvider` qui charge `localStorage` et appelle `useSession`), ou si un hook serveur lève dans `app/layout.tsx` (i.e. avant que `app/error.tsx` puisse s'instancier), l'utilisateur aboutit sur l'écran d'erreur **par défaut** de Next.js — un texte anglais brut « Application error: a client-side exception has occurred » (en production) ou la stack trace complète (en développement), sans la charte sombre du produit, sans bouton `Réessayer` localisé, sans lien retour vers `/dashboard`. Le contraste avec `app/error.tsx` (qui affiche pourtant un visuel soigné avec animations Framer Motion, ligne 22-66 de `src/app/error.tsx`) est d'autant plus visible que l'écran de fallback Next.js casse la continuité de marque.

  Le risque est concret : la chaîne d'initialisation client `src/app/layout.tsx:90-122` empile cinq providers et trois scripts inline ; un seul throw dans cette chaîne (par exemple lors d'une migration future de NextAuth ou d'un changement de `localStorage` Capacitor) basculera tous les utilisateurs sur le fallback brut en attendant le déploiement d'un correctif.

- **Preuve** : `commande: find src/app -name "global-error.tsx"` (0 ligne) ; `src/app/layout.tsx:90-122` (chaîne `RootLayout` + cinq providers, sans boundary englobante) ; `src/app/error.tsx:1-72` (boundary qui ne couvre **pas** ce périmètre) ; documentation Next.js App Router « Error Handling » (référence externe, non auditée mais norme du framework).
- **Impact** : Défaut de robustesse sur le parcours d'initialisation : toute erreur précoce (provider, hydratation, polices, script `themeInitScript`) prive l'utilisateur de la branding du produit et de toute action localisée. Sur Capacitor Android, le rendu par défaut de Next.js est encore plus problématique car la WebView ne bénéficie ni du cache PWA (Service Worker désactivé sur Capacitor — cf. F-UX-011) ni de l'option « Réessayer » localisée. Severity_Medium en application des critères « Défaut de robustesse / absence de gestion d'erreur » et « Problème UX notable / feedback utilisateur absent » du `design.md` > Définitions de sévérité ; non éligible à High car les conditions de déclenchement sont rares (les providers déjà déployés sont stables) — le risque est prospectif et touche la résilience face aux régressions futures.
- **Recommandation** : Créer `src/app/global-error.tsx` (Client Component, `"use client"`) suivant la signature standard `({ error, reset }: { error: Error & { digest?: string }; reset: () => void; })`. Le composant doit retourner un **document HTML complet** (`<html>` et `<body>` inclus, car il remplace le `RootLayout` qui a échoué). Reproduire la charte sombre minimale (fond `#0a0a0f`, texte blanc) pour la cohérence visuelle. Inclure : un titre (« Une erreur critique est survenue »), un message court, un bouton `Réessayer` qui appelle `reset()`, et un lien `<a href="/">` (sans dépendre de `next/link` qui requiert le routeur). **Ne pas afficher** `error.message` brut (cf. F-UX-012). Logger l'erreur côté client via `useEffect(() => console.error("[global-error]", error), [error])` ou — à terme — via un service externe (Sentry, etc.) si le projet en adopte un.

### F-UX-010 — Granularité insuffisante des fichiers `not-found.tsx` et absence de `(auth)/error.tsx` : `notFound()` et erreurs sous `(auth)` perdent le layout du segment, sortent du contexte applicatif et désorientent l'utilisateur

- **Axe** : UX
- **Sévérité** : Medium
- **Description** : Le routeur expose un seul `not-found.tsx` à la racine `src/app/not-found.tsx` et **aucun** au niveau des segments `(platform)`, `(auth)`, ou plus profond (`workspace/[workspaceId]`, `task/[taskId]`, etc.). Or :

C'est le seul `notFound()` détecté dans le code (`grep_search 'notFound\\(\\)' src/app/` = 1 occurrence), mais il vise précisément le parcours « ouvrir une tâche invalide » qui peut survenir sur (a) tâche supprimée par un autre membre, (b) lien de notification stale, (c) lien partagé invalide.

Un utilisateur non authentifié arrivant sur une URL invalide (par exemple `/workspace/inexistant` partagé sur les réseaux) atterrit sur cette page, clique sur le seul bouton, et est immédiatement redirigé par `requireAuth` vers `/login` sans `callbackUrl`. Cette transition est jarrante et redondante avec F-UX-003 (callbackUrl ignoré).

  4. **Aucun `loading.tsx`** à aucun niveau du routeur (`find src/app -name "loading.tsx"` = 0). Bien que les états `isLoading` SWR existent côté client (cf. annexe D), Next.js n'utilise donc pas le mécanisme de `<Suspense>` automatique entre navigations server-side. Sur Capacitor Android où le RTT peut être élevé, l'utilisateur observe un écran figé pendant la résolution du server component (par exemple `(platform)/task/[taskId]/page.tsx` qui exécute un `prisma.task.findUnique` ligne 14-32 avant tout rendu) sans aucune indication visuelle — cf. - **Preuve** : `commande: find src/app -name "not-found.tsx"` (1 ligne : `src/app/not-found.tsx`) ; `commande: find src/app -name "loading.tsx"` (0 ligne) ; `src/app/(platform)/task/[taskId]/page.tsx:33-35` (`if (!task || ...members.length === 0) { notFound(); }`) ; `src/app/not-found.tsx:1-30` (page racine hors `(platform)` layout, bouton unique vers `/dashboard`) ; `src/app/(auth)/layout.tsx` (présent et nommé `AuthLayout`, pas de boundary error.tsx adjacente — `find src/app/(auth) -name "error.tsx"` = 0) ; `src/app/error.tsx:1-72` (boundary racine qui propose `Retour au dashboard` même en contexte non authentifié, cf. ligne 60-65).

Severity_Medium en application des critères « Problème UX notable / feedback utilisateur absent / navigation cassée sur cas particulier » et « Défaut de robustesse / état loading/empty manquant sur un parcours non critique » du `design.md` > Définitions de sévérité. Non éligible à High car aucun parcours critique n'est totalement bloqué (l'utilisateur peut toujours naviguer manuellement), mais le caractère systémique (toute erreur ou 404 sous un layout group bascule hors de ce layout) justifie d'aller au-delà du seuil cosmétique.
- **Recommandation** : (i) Créer `src/app/(platform)/not-found.tsx` qui réutilise le visuel de la racine **dans le contexte du `PlatformLayout`** — l'utilisateur reste dans la sidebar/top-bar et peut continuer à naviguer. (ii) Créer `src/app/(auth)/error.tsx` et `src/app/(auth)/not-found.tsx` adaptés au visuel d'authentification (fond, glassmorphism actuels du `(auth)/layout.tsx`), avec un bouton « Retour à la connexion » (`/login`) plutôt que `/dashboard`. (iii) Optionnel mais recommandé : créer un `src/app/(platform)/task/[taskId]/not-found.tsx` dédié qui affiche « Cette tâche n'existe plus ou vous n'y avez plus accès » et propose `Voir mes tâches` (`/my-tasks`) plutôt qu'un dashboard générique. (iv) Adapter le bouton CTA de `src/app/not-found.tsx` (racine) pour pointer vers `/login` lorsque l'utilisateur n'est pas authentifié (passer côté server component avec `getServerSession` pour décider, à mettre en cohérence avec F-UX-003). (v) Ajouter `src/app/(platform)/loading.tsx` minimal (skeleton ou spinner) afin que les transitions server-side fournissent un feedback visuel — surtout sur la WebView Capacitor.

### F-UX-011 — Aucune détection ni fallback réseau côté Capacitor Android : Service Worker explicitement désactivé sur la WebView native, aucun plugin `@capacitor/network`, aucune bannière offline

- **Axe** : UX
- **Sévérité** : Medium
- **Description** : La cible mobile principale du projet (Capacitor Android, cible primaire selon `BUILD_APK.md` et `INSTALL_MOBILE.md`) est privée de **tout** mécanisme de détection ou de fallback en cas de perte de réseau. Trois constats convergents :

  1. **Service Worker désactivé sur Capacitor** : `src/app/layout.tsx:79-100` enregistre `public/sw.js` au `load` window **uniquement si non Capacitor** (`if (window.Capacitor && window.Capacitor.isNativePlatform && window.Capacitor.isNativePlatform()) { return; }`, ligne 84-87, avec commentaire explicite ligne 80 : `« désactivé dans Capacitor — casse la WebView »`). Le SW est donc actif uniquement pour les utilisateurs PWA web. Sur Android natif, la WebView ne bénéficie d'aucun cache offline ni d'aucune réponse `{ offline: true }` (qui est par ailleurs renvoyée par le SW dans `public/sw.js:80-91` mais non lue par les consommateurs — cf. F-UX-008).

  2. **Plugin `@capacitor/network` non installé** (`grep -E "@capacitor/" package.json` : 5 plugins `@capacitor/{android,cli,core,preferences,push-notifications}`, **aucun** `@capacitor/network` ni `@capacitor-community/network`). Aucune écoute `Network.addListener('networkStatusChange', ...)` dans le code (`grep -rn -E "Network\\.addListener" src/` = 0). L'application n'a donc **aucun moyen natif** de détecter le passage online ↔ offline sur Android.

  3. **Aucune bannière offline ni écoute `navigator.onLine`** : `grep -rn -E "navigator\\.onLine|window\\.addEventListener\\(.(online|offline)" src/` = 0 occurrence applicative. Aucun composant `<OfflineBanner />`, `<NetworkStatus />` ou équivalent n'est défini ou rendu dans `src/components/` ou `src/providers/`. Conséquence : sur perte de connectivité, **rien** ne change visuellement à l'écran ; les requêtes SWR (cf. annexe D) échouent silencieusement, l'utilisateur voit des `EmptyState` trompeurs (cf. F-UX-008) et n'a aucun moyen de comprendre pourquoi ses tâches ne se chargent pas.

Le `POST /api/tasks` est rejeté par la WebView (Capacitor expose pourtant un `cors` permissif via le `fetchInterceptor` ligne 60-78 de `src/app/layout.tsx`, mais cela ne change rien à l'absence de réseau) ; le hook `useCreateTask` (`src/hooks/use-tasks.ts`) émet un toast d'erreur via le pattern observé en module task — donc le feedback **mutation** existe ponctuellement (cf. F-UX-005 a contrario pour le module finance). Mais la **lecture** des tâches (`useTasks`) renvoie un `EmptyState` ; l'utilisateur peut croire à une perte de données ou à une bug. Aucun écran « Vous êtes hors ligne. Reconnectez-vous pour continuer » n'apparaît jamais.

- **Preuve** : `src/app/layout.tsx:79-100` (registre SW conditionné à `!isNativePlatform`, commentaire explicite « désactivé dans Capacitor — casse la WebView » ligne 80) ; `commande: grep -E "@capacitor/" package.json` (5 plugins, aucun `network`) ; `commande: grep -rn -E "@capacitor/network|navigator\\.onLine|window\\.addEventListener\\(.online|Network\\.addListener" src/ public/` (0 occurrence applicative hors registre SW dans `layout.tsx`) ; `public/sw.js:79-91` (réponse `{ offline: true }` jamais consommée) ; cf. également `notes/inv-plugins-capacitor.md` (inventaire de la tâche 3.4).
- **Impact** : La cible mobile principale du projet est privée de tout indicateur de connectivité. Sur Android — où la perte de réseau est fréquente (métro, ascenseur, zones rurales, bascule WiFi/4G) — l'utilisateur observe une dégradation silencieuse : tâches qui ne se chargent pas, sauvegardes qui échouent (notes, tâches, finances), notifications qui ne partent pas, sans qu'un message ne lui indique la cause. Le risque est aggravé pour les parcours d'écriture (création de tâche, édition de note via `notes/page.tsx` cf. F-UX-007 sur la perte silencieuse) où l'utilisateur peut **perdre du contenu** en croyant qu'il a été sauvegardé. Severity_Medium en application des critères « Défaut de robustesse » et « Problème UX notable / feedback utilisateur absent » du `design.md` > Définitions de sévérité. Non éligible à High car (a) le périmètre n'est pas un parcours bloquant unique mais une dégradation transverse, (b) il existe partiellement un feedback mutation (toasts d'erreur SWR sur certains modules) qui atténue le risque de perte sur les parcours canoniques. Le passage à High se justifierait si une analyse complémentaire (axe MOB) confirmait des cas concrets de corruption d'état applicatif (par exemple split-brain entre cache local et serveur).

(ii) Créer un composant `<OfflineBanner />` (sticky en haut, sous la `top-bar`) qui s'affiche dès `connected === false` avec un message « Vous êtes hors ligne — vos modifications seront synchronisées à la reconnexion » ; positionner ce composant dans `src/app/(platform)/layout.tsx` pour qu'il couvre toute l'aire applicative. (iii) Étendre `SWRConfig` (`src/lib/swr-config.tsx`) avec un `isOnline()` provider qui désactive les revalidations en offline (`isOnline: () => navigator.onLine` ou via le provider Capacitor). (iv) Documenter en clair, dans `BUILD_APK.md` ou un `docs/offline.md`, la stratégie d'absence de cache offline sur Capacitor (le SW restera désactivé tant qu'il casse la WebView) et le comportement attendu (lecture seule des données déjà chargées, mutations bloquées avec toast).

### F-UX-012 — Les pages d'erreur racine et `(platform)` exposent `error.message` brut à l'utilisateur final, fuitant des détails techniques (stack hint, message de Prisma, message d'API)

- **Axe** : UX
- **Sévérité** : Medium
- **Description** : Les deux boundary d'erreur du projet rendent **le message brut de l'erreur** capturée :

  - `src/app/error.tsx:46` : `{error.message || "Quelque chose s'est mal passé. Veuillez réessayer."}` (avec un fallback localisé seulement si `error.message` est vide).
  - `src/app/(platform)/error.tsx:18-22` : `{error.message && (<p className="mb-2 text-xs text-red-500 font-mono break-all">{error.message}</p>)}` — affichage **explicitement en monospace** et en rouge, ce qui revient à une présentation pseudo-développeur.

  Or, en App Router Next.js, les erreurs jetées côté **server component** (par exemple `prisma.task.findUnique` dans `src/app/(platform)/task/[taskId]/page.tsx:14-32`, ou `requireAuth` lui-même) sont sérialisées vers le client avec leur `message` complet **en mode développement** ; en production, Next.js remplace `error.message` par une chaîne générique et expose un `error.digest`. Mais cette protection ne couvre que les erreurs **server-thrown** ; toute erreur **client-thrown** (par exemple un `throw new Error(\`Workspace ${id} introuvable\`)` dans un hook SWR ou un composant React) garde son `message` complet et sera affichée telle quelle à l'utilisateur, y compris en production. La chaîne du fetcher SWR global (`src/lib/swr-config.tsx:24-26`) lance précisément `throw new Error("Failed to fetch")` — un message court mais qui ne renseigne pas l'utilisateur ; d'autres throws applicatifs sont moins anodins (par exemple `task-action-menu.tsx:96-104` propage `error.message` qui peut contenir le `error` JSON renvoyé par l'API, lequel inclut parfois des références techniques telles que `« Foreign key constraint failed on the field: \`task.listId\` »`).

  Le risque est double : (a) **UX dégradée** — l'utilisateur lit un message technique en anglais qui n'a pas été pensé pour lui (« Failed to fetch », « Network error », « ENOTFOUND », « Prisma error » fragmentaire) ; (b) **fuite informationnelle légère** — le rendu peut révéler des éléments d'architecture (noms de modèles Prisma, URL d'endpoint, structure de la stack) qui aident un attaquant ciblé à mapper le système. Pas de fuite de données utilisateur (les `error.message` ne contiennent pas de PII), mais une fuite d'infrastructure qui contredit la stratégie « pas de fuite de détails techniques » mentionnée à la tâche 7.5 du même audit (axe LOG côté API).

  Le `(platform)/error.tsx` ajoute en outre l'affichage du `error.digest` (ligne 24-28). Ce comportement est **bienvenu** côté support (un identifiant court permet de retrouver l'erreur dans les logs serveur), mais la combinaison `error.message` (rouge, monospace) **+** `error.digest` (gris) crée un visuel pseudo-IDE qui suggère à l'utilisateur que le produit est en panne plutôt que de l'orienter vers une action.

- **Preuve** : `src/app/error.tsx:46` (`{error.message || "..."}`) ; `src/app/(platform)/error.tsx:18-22` (`<p className="mb-2 text-xs text-red-500 font-mono break-all">{error.message}</p>`) ; `src/app/(platform)/error.tsx:24-28` (affichage `error.digest`) ; `src/lib/swr-config.tsx:24-26` (source d'un `error.message = "Failed to fetch"` propagé jusqu'au boundary) ; `src/components/task/task-action-menu.tsx:96-104` (référence : ce module utilise un toast destructif avec `description: error.message` — pattern à transposer ici).
- **Impact** : Toute erreur capturée par les deux boundary affiche un texte technique non localisé à l'utilisateur, sapant la qualité perçue (« le produit a un bug »). En cas d'erreur server-thrown (Prisma, NextAuth, validation Zod), le message peut révéler des détails de l'infrastructure (nom de table, FK, message d'authentification) qui ne sont pas pertinents pour l'utilisateur et qui constituent une fuite légère d'information. Severity_Medium en application des critères « Problème UX notable / feedback utilisateur absent ou inadéquat » et « Défaut de robustesse » du `design.md` > Définitions de sévérité. Le finding pourrait être réévalué à High si l'audit SEC (tâche 11) confirmait que des messages d'erreur Prisma incluent des données utilisateur d'autres tenants ou des secrets — ce n'est pas observé statiquement ici, mais la classe de risque le justifierait.
- **Recommandation** : (i) Dans `src/app/error.tsx:46`, remplacer `{error.message || "..."}` par un message strictement utilisateur (« Quelque chose s'est mal passé. Veuillez réessayer ou revenir au dashboard. ») et déplacer `error.message` dans un `useEffect(() => console.error(...))` (ligne 18) pour conserver l'info en console développeur sans la rendre. (ii) Dans `src/app/(platform)/error.tsx:18-22`, supprimer le bloc `{error.message && (...)}` ; conserver l'affichage du `error.digest` (qui est un identifiant opaque conçu pour cet usage) en l'introduisant par « Référence support : » plutôt que « Code : » pour clarifier l'intention. (iii) Optionnel : ajouter un mécanisme de copie du `digest` (`navigator.clipboard.writeText(digest)` au clic) pour faciliter le rapport de bug. (iv) Aligner avec F-UX-008 : si un service de logging externe (Sentry, etc.) est introduit ultérieurement, transmettre `error` complet (message + stack) au service depuis le `useEffect` plutôt qu'à l'utilisateur.

### F-UX-013 — Lien cassé `/tasks/${id}` (pluriel) dans la palette de recherche `<SearchCommand>` : tout résultat « Tâche » mène à un 404 racine hors `(platform)` layout

### F-UX-017 — Sidebar, mobile-nav, top-bar et quick-action FAB pilotent toute la navigation par `router.push()` plutôt que par `<Link>` : pas de prefetch, pas de clic-milieu, pas de sémantique HTML, incohérence avec les widgets dashboard

- **Axe** : UX
- **Sévérité** : Medium
- **Description** : Sur les composants de navigation primaire de l'application (sidebar, mobile-nav, top-bar, quick-action FAB, sidebar-list-item, view-switcher), **toutes** les transitions de route sont implémentées via le pattern `<button onClick={() => router.push(href)}>` plutôt que via `<Link href={href}>` (`next/link`). L'usage est asymétrique avec d'autres surfaces où `<Link>` est correctement employé :

  - **Sites en `<Link>` (corrects)** :
    - `src/components/layout/breadcrumbs.tsx:75-96` (`<Link href="/dashboard">`, `<Link href={segment.href}>`).
    - `src/app/(platform)/dashboard/page.tsx:194-225` (3 quick-links Mes tâches/Calendrier/Rappels).
    - `src/components/dashboard/budget-widget.tsx:56,92` et `src/components/dashboard/finance-widget.tsx:87` (CTA « Voir tous les budgets / Voir tout »).
    - `src/components/budget/budget-card.tsx:60` (cartes de budget).
    - `src/app/(platform)/workspace/page.tsx`, `src/app/(platform)/workspace/[workspaceId]/page.tsx`, `src/app/(platform)/workspace/[workspaceId]/space/[spaceId]/page.tsx` (cartes de navigation hiérarchique).
    - `src/components/auth/login-form.tsx:221`, `register-form.tsx:250` (renvois croisés login ↔ register), `src/app/page.tsx:32-40,76-82,288`, `src/app/error.tsx:63`, `src/app/not-found.tsx:19` (pages publiques et boundary).

  - **Sites en `router.push()` (cible de ce finding)** :
    - `src/components/layout/sidebar.tsx:193` (10 items principaux), `:276,278,280` (favoris), `:391,397` (dropdown utilisateur).
    - `src/components/layout/mobile-nav.tsx:36` (5 items barre du bas).
    - `src/components/layout/top-bar.tsx:101` (cloche notifications).
    - `src/components/layout/quick-action-fab.tsx:84,102` (FAB → /reminders et /notes).
    - `src/components/layout/sidebar-list-item.tsx:23` (chaque liste de la hiérarchie sidebar).
    - `src/components/layout/view-switcher.tsx:42,47` (commutateur de vue list/board/calendar/gantt).

  Conséquences fonctionnelles, accessibilité et performance :

  1. **Pas de `prefetch`** — `<Link>` déclenche en production un prefetch de la route quand le lien entre dans le viewport (comportement par défaut de Next.js App Router, configurable via `prefetch={false}`). `router.push()` ne préfetche jamais. Sur mobile Capacitor (cible primaire) où chaque navigation déclenche un round-trip serveur (`requireAuth` + Prisma queries dans le server component), l'absence de prefetch ajoute systématiquement la latence réseau au temps de transition. Pour la sidebar (10 items toujours visibles côté desktop) et la mobile-nav (5 items toujours visibles côté mobile), le prefetch aurait précisément l'effet maximal.

  2. **Pas de clic-milieu / Cmd+clic / clic droit** — Un `<button>` n'est pas cliquable à la molette, ni en `Cmd/Ctrl+clic` pour ouvrir dans un nouvel onglet, ni en clic droit pour copier l'URL. L'utilisateur power-user web qui souhaite ouvrir « Mes tâches » dans un nouvel onglet pour comparer avec le dashboard est obligé de copier l'URL manuellement depuis la barre d'adresse.

  3. **Sémantique HTML incorrecte** — `<a href>` est annoncé par les lecteurs d'écran comme « lien » avec destination ; `<button onClick>` est annoncé comme « bouton ». Pour des items de navigation (intention sémantique : naviguer vers une URL), la convention WAI-ARIA et WCAG 2.4.4 (Link Purpose, In Context) prescrit le `<a href>` avec un nom accessible explicite (cf. `notes/axe-ui.md#F-UI-014`). Les utilisateurs naviguant en lecture par liens (`Insert+F7` sur NVDA, rotor sur VoiceOver) **ne voient pas du tout** les items de la sidebar et de la mobile-nav puisqu'ils ne sont pas des liens.

  4. **Status bar du navigateur vide au survol** — Sur desktop, le survol d'un `<a href>` affiche la cible dans la barre d'état du navigateur (en bas à gauche). C'est un repère de confiance (l'utilisateur sait où il va atterrir). `<button>` n'affiche rien.

  5. **Cohérence interne brisée** — L'application utilise correctement `<Link>` sur les widgets dashboard (qui copient pourtant les mêmes destinations que la sidebar : « Mes tâches », « Calendrier », « Rappels »). Les utilisateurs reçoivent donc deux UX différentes pour la même destination selon qu'ils cliquent dans le widget ou dans la sidebar. La coexistence des deux patterns n'est justifiée par aucun blocage technique (le seul cas où `router.push` est requis serait un onClick complexe avec side-effects préalables — mais les sites cités ne font qu'enchaîner `setOpen(false); router.push(href)`, ce qui est trivial à composer avec `<Link onClick={() => setOpen(false)}>`).

  Le cas particulier de `sidebar.tsx:181-194` mérite mention : l'item « Rechercher » dispatche un `KeyboardEvent` au lieu de naviguer (ouverture de la palette `⌘K`). Pour cet item-là, `<button>` est légitime (action, pas navigation). Mais la même boucle `navItems.map` traite **uniformément** les 10 items en `<button>` ; il faudrait scinder le rendu entre items-navigation (`<Link>`) et items-action (`<button>`).

- **Preuve** : `src/components/layout/sidebar.tsx:172-244` (corps de la boucle `navItems.map` rendant 10 `<button onClick={() => router.push(item.href)}>`) ; `src/components/layout/sidebar.tsx:265-285` (favoris en `<button onClick={router.push(...)}>`) ; `src/components/layout/mobile-nav.tsx:34-58` (5 `motion.button` avec `router.push(item.href)`) ; `src/components/layout/top-bar.tsx:97-117` (cloche en `<Button onClick={() => router.push("/notifications")}>`) ; `src/components/layout/quick-action-fab.tsx:81-110` (deux `motion.button` avec `router.push("/reminders")` et `router.push("/notes")`) ; `src/components/layout/sidebar-list-item.tsx:23` (`onClick={() => router.push(listPath)}`) ; `src/components/layout/view-switcher.tsx:42,47` (commutateur de vue) ; `commande: grep_search 'router\\.push' --include='src/components/layout/**'` (5 fichiers, 11 sites de navigation) ; comparatifs corrects en `<Link>` : `src/components/layout/breadcrumbs.tsx:75,87`, `src/app/(platform)/dashboard/page.tsx:195,207,219`, `src/components/dashboard/{budget,finance}-widget.tsx`. `commande: grep_search 'import Link from "next/link"' --include='src/**'` (15 fichiers utilisent correctement `<Link>` — la layout en est exclue).
- **Impact** : Performance (latence supplémentaire systématique sur chaque transition mobile, là où le prefetch aurait l'impact le plus fort), accessibilité (les utilisateurs de lecteurs d'écran en mode « liste de liens » ne voient pas la sidebar ni la mobile-nav, contradiction avec WCAG 2.4.4), ergonomie power-user (impossible d'ouvrir une page de la sidebar dans un nouvel onglet desktop). L'incohérence interne avec les widgets dashboard sape la cohérence du produit. Severity_Medium en application des critères « Problème UX notable / surface ergonomique » + « Défaut de robustesse / convention sémantique HTML non respectée sur la primitive de navigation ». Non éligible à High car aucun parcours n'est bloqué et la navigation fonctionne fonctionnellement — c'est une dégradation transverse de qualité, pas un blocage.
- **Recommandation** : (i) **Refactorer** `sidebar.tsx`, `mobile-nav.tsx`, `top-bar.tsx`, `quick-action-fab.tsx`, `sidebar-list-item.tsx`, `view-switcher.tsx` pour utiliser `<Link href={href}>` partout où l'`onClick` se réduit à `router.push(href)`. Pour `view-switcher.tsx` qui calcule dynamiquement le chemin, calculer le `href` au moment du rendu (`const targetHref = ...`) puis poser `<Link href={targetHref}>`. (ii) Pour les sites composés (ex. `MobileSidebar` qui doit fermer le sheet **et** naviguer), utiliser `<Link href={href} onClick={() => setOpen(false)}>` — la composition `<Link>` + `onClick` est officiellement supportée par Next.js et préserve le prefetch. (iii) Scinder dans `sidebar.tsx` les items-action (« Rechercher » qui dispatche un keyboard event) des items-navigation, pour pouvoir rendre les seconds en `<a>` proprement. (iv) Aligner avec F-UI-014 en posant `aria-current="page"` sur le `<a>` actif (Next.js `<Link>` supporte la propagation d'attributs), ce qui résout aussi la dimension a11y. (v) Documenter la règle « toute navigation = `<Link>`, jamais `router.push` sauf side-effect asynchrone obligatoire » dans `CLAUDE.md` ou un `docs/conventions.md`, et envisager une règle ESLint personnalisée bloquant `router.push` dans `src/components/layout/**`.
