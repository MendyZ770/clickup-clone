# Axe BDD — Findings

> Notes de Findings pour l'axe **BDD** (Base de données / schéma Prisma vs usages applicatifs effectifs). Chaque Finding suit le schéma normalisé du `design.md` (Property 1 : huit champs requis non vides, identifiant `F-BDD-NNN`, preuve de type `path:line`, `path#symbole` ou `commande:`).
>
> **Périmètre couvert par cette tâche 8.1** : confrontation du `prisma/schema.prisma` (49 modèles, cf. `notes/annexe-E-prisma.md`) à ses **usages effectifs** dans `src/` et `prisma/`. Les axes index manquants (8.2), cascades `onDelete` (8.3), cohérence migrations (8.4), N+1 / scans complets (8.5) et SQL brut (8.6) sont traités dans les sous-tâches dédiées et exclus ici pour éviter les doublons.
>
> **Méthodologie** : (i) énumération exhaustive des accès `prisma.<model>` via `grep -roh -E "prisma\.[a-zA-Z]+"` (43 modèles distincts détectés sur 49 déclarés) ; (ii) recherche des champs scalaires suspects (FK logiques non-relationnelles, champs nullable jamais lus) ; (iii) recherche des relations `@relation` jamais traversées via `include`/`select` ; (iv) recherche inverse (références code à des modèles/champs absents du schéma) ; (v) confrontation aux types réexportés par `src/types/index.ts`.
>
> **Critères de sévérité retenus** : la dette technique d'un modèle ou d'un champ entièrement mort est classée Severity_Medium par défaut (clause 7.1 + Property 3) — elle pollue le schéma, induit des migrations futures à risque et trompe la maintenance. Le passage en Severity_High est réservé aux cas où la dette crée une **double vérité** ou un **risque d'intégrité** (deux mécanismes contradictoires sur la même donnée, FK polymorphe non contrainte sur ressource sensible).

## Findings

### F-BDD-001 — Modèle `BudgetAlert` déclaré mais jamais persisté ni lu (table morte avec FK vers `User` et `Budget`)

- **Axe** : BDD
- **Sévérité** : Medium
- **Description** : Le modèle `BudgetAlert` (`prisma/schema.prisma#BudgetAlert`, lignes 643-657 de l'inventaire annexe E §1.44) déclare 7 champs (`id`, `type`, `threshold`, `isActive`, `budgetId`, `userId`, `createdAt`), 2 relations cascadées (`Budget` et `User`, `onDelete: Cascade`), 2 `@@index` (`[budgetId]`, `[userId]`) et la relation inverse `Budget.alerts` (`prisma/schema.prisma#Budget`, ligne 537). Or **aucun appel `prisma.budgetAlert.*`** n'existe dans `src/` ni `prisma/seed.ts` (commande `grep -rn "prisma\.budgetAlert" src/ prisma/` → exit code 1, aucune ligne ramenée). La relation inverse `Budget.alerts` n'est jamais consommée non plus (commande `grep -rn -E "(alerts:|alerts\b)" src/app/api/budget/` ne retourne aucun `include`/`select`). Le seul rappel est typologique (`src/types/index.ts:28` `BudgetAlert,` importé depuis `@prisma/client` puis `src/types/index.ts:229` exporté comme `BudgetAlertWithBudget` — type lui-même non consommé, voir commande consignée). Le **composant React `BudgetAlert`** (`src/components/budget/budget-alert.tsx:9`, importé par `src/app/(platform)/budget/[budgetId]/page.tsx:14`) est un **homonyme** : il calcule un pourcentage côté client à partir de `spentPercent` et n'écrit rien en base. Le cron `src/app/api/cron/budget-alerts/route.ts:42-72` envoie des `prisma.notification` (modèle `Notification`) avec `type: "budgetAlert"` et un déduplicage 24 h via `prisma.notification.findFirst` (lignes 50-58) — il n'écrit **jamais** dans la table `BudgetAlert`. Les seuils d'alerte (80 %, 100 %) sont **codés en dur dans le cron** (lignes 38-41), pas lus depuis `BudgetAlert.threshold`.
- **Preuve** : `prisma/schema.prisma#BudgetAlert` (déclaration) ; `commande: grep -rn "prisma\.budgetAlert" src/ prisma/` (exit 1, aucune ligne) ; `commande: grep -rn -E "(financeGoalContribution|budgetAlert|budgetTemplate|budgetTransactionTag)\." src/` (1 seule ligne ramenée et concernant `financeGoalContribution`) ; `src/types/index.ts:28` ; `src/types/index.ts:229` ; `src/components/budget/budget-alert.tsx:9` (homonyme client) ; `src/app/api/cron/budget-alerts/route.ts:38-72` (logique alternative via `prisma.notification`).
- **Impact** : Dette technique significative. (a) La table `BudgetAlert` existe en base de production (cf. F-BDD-006 ci-dessous : aucune migration versionnée — le déploiement a nécessairement créé la table) mais **n'est jamais peuplée** : ligne morte qui occupe l'espace de schéma et complexifie le diagramme E-R. (b) L'absence d'utilisation invalide les `@@index([budgetId])` et `@@index([userId])` : ils consomment du stockage et des écritures inutilement. (c) Les seuils d'alerte étant codés en dur dans le cron, **la fonctionnalité métier d'alertes paramétrables est inopérante** : un utilisateur ne peut pas définir un seuil custom (40 %, 50 %) bien que le schéma le suggère (`type: "threshold"`, `threshold: Float?`). (d) Risque pour le mainteneur : un développeur futur lisant le schéma supposera l'existence d'un système d'alertes paramétrables et pourra construire dessus, créant des bugs en production. (e) Le typage `BudgetAlertWithBudget` (`src/types/index.ts:229`) ouvre une porte de duplication.
- **Recommandation** : Trancher entre deux options et documenter le choix. **Option A — Activer le modèle** : implémenter la chaîne `BudgetAlert` (CRUD API sous `src/app/api/budget/[budgetId]/alerts/`, lecture dans le cron `cron/budget-alerts/route.ts` pour remplacer les seuils en dur, UI de configuration dans la page budget). **Option B — Supprimer** : retirer le modèle de `prisma/schema.prisma`, retirer la relation inverse `Budget.alerts`, retirer les types `BudgetAlert` et `BudgetAlertWithBudget` de `src/types/index.ts`, planifier la migration de suppression de la table. Option B recommandée à court terme si la roadmap n'inclut pas d'alertes paramétrables. Toute action effective relève d'une tâche de remédiation post-audit (audit en lecture seule).

À cause commune avec F-BDD-001 — fusionnable en une seule tâche de remédiation « nettoyer les modèles Budget non utilisés ».

### F-BDD-003 — Modèle `BudgetTransactionTag` peuplé mais jamais lu en standalone (uniquement via la relation `tags` de `BudgetTransaction`)

- **Axe** : BDD
- **Sévérité** : Medium
- **Description** : Le modèle `BudgetTransactionTag` (`prisma/schema.prisma#BudgetTransactionTag`, cf. annexe E §1.42) déclare 3 champs (`id`, `name`, `transactionId`), une relation cascadée vers `BudgetTransaction`, et 2 contraintes `@@unique([transactionId, name])` + `@@index([transactionId])`. Le modèle est **peuplé** via la création nestée dans `src/app/api/budget/[budgetId]/transactions/route.ts:138-141` (`tags: tags && tags.length > 0 ? { create: tags.map((name) => ({ name })) } : undefined`) puis **lu** uniquement via `include: { tags: true }` (lignes 96 et 146). Or **aucun appel `prisma.budgetTransactionTag.*` direct** n'existe (commande `grep -rn "prisma\.budgetTransactionTag" src/ prisma/` → exit 1). En particulier, **aucune route ne permet la mise à jour, le renommage ou la suppression d'un tag** : (a) la mise à jour d'une transaction (`src/app/api/budget/[budgetId]/transactions/[transactionId]/route.ts` PATCH/PUT — à vérifier en 7.5) ne réécrit pas la collection `tags` ; les tags initiaux sont donc figés à la création, sans possibilité de les modifier. (b) Aucune API `tags` au niveau workspace pour lister, fusionner ou supprimer un tag à l'échelle. (c) Aucune route filtrant les transactions par tag (`where: { tags: { some: { name: ... } } }`). Le composant `src/components/budget/add-transaction-dialog.tsx:225` permet d'**ajouter** un tag mais aucun composant ne permet de l'**éditer** ou de le **filtrer**.
- **Preuve** : `prisma/schema.prisma#BudgetTransactionTag` ; `src/app/api/budget/[budgetId]/transactions/route.ts:96`, `:138`, `:146` (création nestée, lecture via include) ; `commande: grep -rn "prisma\.budgetTransactionTag" src/ prisma/` (exit 1) ; `src/types/index.ts:26`, `:193` (`tags: BudgetTransactionTag[]`).
- **Impact** : Dette technique. Le modèle est partiellement actif (création + lecture en include), mais l'absence d'**édition** et de **filtrage** rend la fonctionnalité « tags » incomplète : un utilisateur qui ajoute un tag « cousre » par erreur ne peut pas le corriger sans recréer la transaction. À l'échelle, l'absence de listing des tags d'un workspace empêche la cohérence (« courses » vs « course » vs « Courses » coexistent silencieusement, malgré `@@unique([transactionId, name])` qui n'opère qu'à l'échelle d'une transaction). Le `@@index([transactionId])` est correctement utilisé mais aucun index sur `name` ne soutient un filtrage `where: { name: { contains } }` futur.
- **Recommandation** : Compléter la chaîne CRUD ou consolider. **Option A** : ajouter une route `PATCH src/app/api/budget/[budgetId]/transactions/[transactionId]/route.ts` qui synchronise `tags` (delete + create dans une transaction Prisma), une route `GET src/app/api/budget/tags/route.ts` listant les tags distincts d'un workspace, et un filtre `?tag=` sur le `GET /api/budget/[budgetId]/transactions`. **Option B** : remplacer la table par un champ scalaire `tags String[]` (Postgres array) sur `BudgetTransaction`, plus simple si la fonctionnalité reste minimale. À recouper avec la tâche 8.4 si une migration future est planifiée.

### F-BDD-004 — Champs `BudgetTransaction.sourceBudgetId` et `BudgetTransaction.targetBudgetId` jamais lus ni écrits (FK logiques mortes pour la fonctionnalité « transfert entre budgets »)

- **Axe** : BDD
- **Sévérité** : Medium
- **Description** : Le modèle `BudgetTransaction` (`prisma/schema.prisma#BudgetTransaction`, cf. annexe E §1.40) déclare un drapeau `isTransfer Boolean @default(false)` et **deux champs scalaires `sourceBudgetId String?` et `targetBudgetId String?`** destinés à matérialiser les transferts entre budgets distincts. Ces deux champs **ne sont jamais référencés** dans le code applicatif (commande `grep -rn "sourceBudgetId\|targetBudgetId" src/ prisma/` → exit 1, aucune ligne ramenée). Aucun appel ne lit, ne filtre, ni n'écrit ces champs ; aucun schéma Zod ne les valide. Le drapeau `isTransfer` est par ailleurs déclaré dans `src/app/api/budget/[budgetId]/transactions/route.ts:16` (Zod `z.boolean().optional()`) mais sa **valeur n'est jamais persistée** : la création de transaction (lignes 132-143) déstructure `recurrenceEnd` et étale `...data` sans renvoyer `sourceBudgetId`/`targetBudgetId` (et le schéma Zod ne les expose pas). À noter : la fonctionnalité de transfert **est implémentée pour `FinanceTransaction`** (`src/app/api/finance/transactions/route.ts:56-99`, qui utilise correctement `targetAccountId`, voir F-BDD-005) — `BudgetTransaction` reste la moitié inerte du dispositif.
- **Preuve** : `prisma/schema.prisma#BudgetTransaction` (déclaration) ; `commande: grep -rn "sourceBudgetId\|targetBudgetId" src/ prisma/` (exit 1) ; `src/app/api/budget/[budgetId]/transactions/route.ts:16` (Zod `isTransfer`) ; absence d'écriture dans le bloc `prisma.budgetTransaction.create` (`:132-143`).
- **Impact** : Dette technique notable et risque d'incohérence métier. (a) Les champs `sourceBudgetId` / `targetBudgetId` consomment de l'espace en base et la documentation laisse penser que la fonctionnalité « transferts inter-budgets » existe, alors qu'elle n'est jamais persistée. (b) Si un utilisateur active `isTransfer: true` côté client, l'enregistrement est créé sans contrepartie : aucun mouvement n'est réellement enregistré sur le budget cible, créant une incohérence comptable silencieuse (la transaction apparaît côté source mais le total cible n'est pas affecté). (c) Les champs n'étant pas matérialisés en `@relation` Prisma, la cascade `Cascade` qui supprime un `Budget` source ou cible **n'est pas appliquée** : si une suppression manuelle survenait, on aurait des FK orphelines.
- **Recommandation** : Soit (i) **finaliser la fonctionnalité** : ajouter les `@relation` Prisma pour matérialiser les FK avec `onDelete: SetNull` (similaire à `BudgetTransaction.category`), étendre le schéma Zod du POST, écrire dans `prisma.budgetTransaction.create` les deux champs, et idéalement créer la transaction inverse côté budget cible dans une `prisma.$transaction` ; soit (ii) **supprimer les champs** du schéma et le drapeau `isTransfer` côté `BudgetTransaction` puisque la fonctionnalité est portée par `FinanceTransaction`. À recouper avec F-BDD-005 (`FinanceTransaction.targetAccountId` dans le même état de FK non contrainte).

### F-BDD-005 — Champ `FinanceTransaction.targetAccountId` lu et écrit mais sans relation Prisma (FK logique non contrainte sur ressource financière sensible)

- **Axe** : BDD
- **Sévérité** : High
- **Description** : Le modèle `FinanceTransaction` (`prisma/schema.prisma#FinanceTransaction`, cf. annexe E §1.47) déclare un champ scalaire `targetAccountId String?` mais **aucune `@relation`** ne matérialise la FK vers `FinanceAccount`. Le champ est **activement utilisé** dans le code applicatif : `src/app/api/finance/transactions/route.ts:56` extrait `targetAccountId` du body, ligne 73 le persiste dans `prisma.financeTransaction.create`, lignes 87-99 effectue la chaîne « transfert » qui (a) crée la transaction d'origine, (b) **résout manuellement** le compte cible via `prisma.financeAccount.findUnique({ where: { id: targetAccountId }, … })`, (c) crée une seconde `financeTransaction` symétrique sur ce compte. Côté client, `src/components/finance/add-transaction-dialog.tsx:38,77,134` envoie cette FK. **Trois problèmes** se cumulent : (1) **Aucune contrainte d'intégrité référentielle** côté base : `targetAccountId` peut pointer vers un `FinanceAccount` inexistant, vers un compte d'un autre workspace, ou vers une chaîne arbitraire — la base accepte ; le contrôle ne se fait qu'au moment du POST si l'utilisateur a accès au compte cible (la route ne vérifie pas explicitement l'**isolation workspace** entre `accountId` source et `targetAccountId` cible — à recouper en 11.2 axe SEC). (2) **Aucune cascade `onDelete`** : la suppression d'un `FinanceAccount` cible (cascade depuis `Workspace` ou `User`) **laisse des `FinanceTransaction` avec `targetAccountId` orphelin** sans rendre le champ `null` ; les listes de transferts continueront de référencer un compte fantôme. (3) **Aucune contrainte logique vérifiée en base** que `accountId !== targetAccountId` (un transfert d'un compte vers lui-même est syntaxiquement permis et générerait un double mouvement absurde — la vérification doit donc être faite côté API, ce que le code actuel ne fait pas explicitement). La fonctionnalité de transfert est centrale à la couche Finance (cf. UI `add-transaction-dialog.tsx`) et touche à des montants monétaires.
- **Preuve** : `prisma/schema.prisma#FinanceTransaction` (champ scalaire sans `@relation`, cf. annexe E §1.47 « `targetAccountId` est une FK logique non matérialisée par une `@relation` Prisma ») ; `src/app/api/finance/transactions/route.ts:56` (extraction), `:73` (persistance), `:87-99` (transfert composé manuel) ; `src/components/finance/add-transaction-dialog.tsx:77,134` (envoi côté client) ; `commande: grep -rn -E "(sourceBudgetId|targetBudgetId|targetAccountId)" src/ --include="*.ts" --include="*.tsx"` (résultat consigné en annexe A).
- **Impact** : Risque d'intégrité de données financières — **donc Severity_High** (clause 7.1 + Property 7 : intégrité de données). (a) La base peut héberger des transactions de transfert avec `targetAccountId` orphelin après suppression de comptes ; les soldes affichés deviennent incohérents. (b) En l'absence de `@relation`, Prisma ne typecheck pas l'usage : un développeur peut écrire `targetAccountId: someTaskId` par erreur sans erreur compile-time. (c) L'absence de contrainte `accountId !== targetAccountId` ouvre la voie à des transactions auto-référencées qui dupliqueraient les mouvements. (d) Couplé à F-BDD-004, l'application affiche un patron incohérent : `BudgetTransaction` a deux champs FK morts, `FinanceTransaction` a un champ FK actif mais non contraint.
- **Recommandation** : Ajouter une `@relation` Prisma sur `targetAccountId` (`targetAccount FinanceAccount? @relation("FinanceTransferTarget", fields: [targetAccountId], references: [id], onDelete: SetNull)`), avec relation inverse côté `FinanceAccount.transferTargetTransactions FinanceTransaction[] @relation("FinanceTransferTarget")`, accompagner d'une contrainte `CHECK (account_id <> target_account_id)` (via SQL natif Postgres) et durcir le contrôle workspace dans la route POST. À recouper avec la tâche 8.3 (cascades) et 11.2 (ownership cross-workspace).

### F-BDD-006 — Champ `Favorite.targetId` polymorphe non contraint (référence implicite à 4 modèles distincts sans intégrité référentielle)

- **Axe** : BDD
- **Sévérité** : High
- **Description** : Le modèle `Favorite` (`prisma/schema.prisma#Favorite`, cf. annexe E §1.30) implémente un patron **polymorphe** : un champ `type String` valant `"task" | "list" | "space" | "folder"` (cf. commentaire schéma) et un champ scalaire `targetId String` qui pointe **selon le type** vers `Task.id`, `List.id`, `Space.id` ou `Folder.id`. Aucune `@relation` Prisma ne matérialise cette FK, et `@@unique([userId, type, targetId])` ne vérifie que l'unicité du couple (pas l'existence). Conséquences : (1) **Aucune intégrité référentielle** côté base : `targetId` peut pointer vers une ressource inexistante ou supprimée. (2) **Aucune cascade** : la suppression d'une `Task`, `List`, `Space` ou `Folder` (toutes cascadées depuis `Workspace`) **laisse des `Favorite` orphelins** ; les écrans « Mes favoris » afficheront des entrées pointant vers des ressources fantômes (ou échoueront en 404 silencieux). (3) **Aucune isolation workspace** : un utilisateur peut techniquement marquer comme favori l'`id` d'une tâche d'un workspace dont il n'est pas membre, si l'API ne vérifie pas l'ownership avant l'INSERT. Le contrôle est entièrement délégué à l'API (à recouper en 11.2 SEC). (4) Les requêtes d'enrichissement (« charger les détails du favori ») sont obligatoirement faites en **deux temps** côté serveur (d'abord lire `Favorite`, puis dispatcher selon `type` vers `Task`, `List`, `Space`, `Folder`), ce qui exclut tout `include` Prisma natif et complique la prévention des N+1 (cf. tâche 8.5).
- **Preuve** : `prisma/schema.prisma#Favorite` ; annexe E §1.30 (« `targetId` est une FK polymorphe non contrainte (le type cible dépend de `type`). À recouper en phase 3 (axe BDD). ») ; `commande: grep -rn "prisma\.favorite" src/` (consignée en annexe A).
- **Impact** : Risque d'intégrité référentielle élevé — **Severity_High** (clause 7.1, Property 7). À chaque suppression cascadée d'une `Task`, `List`, `Space` ou `Folder` (ce qui est une opération courante : on supprime des tâches en masse, on archive des listes), des `Favorite` deviennent orphelins. Sans nettoyage, la table grossit silencieusement et les écrans favoris dégradent l'UX. Sur le long terme, l'absence d'isolation workspace crée un risque que des `targetId` cross-workspace coexistent (par bug ou par évolution future) sans qu'aucune contrainte ne les détecte. Le pattern polymorphe rend par ailleurs toute migration future risquée.
- **Recommandation** : Refactoriser pour matérialiser les relations explicitement. **Option A — Quatre champs FK exclusifs** : remplacer `targetId` par `taskId String?`, `listId String?`, `spaceId String?`, `folderId String?` (mutuellement exclusifs via `CHECK` SQL `(num_nonnulls(task_id, list_id, space_id, folder_id) = 1)`) et 4 `@relation` avec `onDelete: Cascade`. **Option B — Quatre tables séparées** : `FavoriteTask`, `FavoriteList`, `FavoriteSpace`, `FavoriteFolder` (plus verbeux, mais plus simple à requêter). **Option C — Conserver le pattern actuel** mais ajouter (i) un job de purge des favoris orphelins (à brancher sur les routes DELETE de `Task`/`List`/`Space`/`Folder`), (ii) une vérification d'ownership systématique côté API avant INSERT, (iii) un test d'intégration qui crée puis supprime une tâche favorisée et vérifie l'absence d'orphelin. À fusionner avec F-BDD-005 et F-BDD-004 sur une tâche de remédiation « FK logiques non contraintes ».

### F-BDD-007 — Cohabitation contradictoire entre `Task.assigneeId` (FK simple) et `TaskAssignee` (table d'association multi-assignation)

- **Axe** : BDD
- **Sévérité** : High
- **Description** : Le modèle `Task` (`prisma/schema.prisma#Task`, cf. annexe E §1.14) déclare **simultanément** deux mécanismes d'assignation : (1) un champ scalaire `assigneeId String?` avec `@relation("TaskAssignee", fields: [assigneeId], references: [id], onDelete: SetNull)` — modèle « un seul assigné » ; (2) une relation inverse `assignees TaskAssignee[]` vers la table d'association `TaskAssignee` (`prisma/schema.prisma#TaskAssignee`, cf. annexe E §1.29) — modèle « multi-assignation ». Les deux mécanismes sont **activement utilisés en parallèle** dans le code, sans synchronisation explicite : (a) les routes de création/édition de tâche écrivent uniquement dans `assigneeId` (`src/app/api/tasks/route.ts:115,172`, `src/app/api/tasks/[taskId]/route.ts:209-241`, `src/app/api/tasks/[taskId]/subtasks/route.ts:92,121`, `src/app/api/tasks/[taskId]/duplicate/route.ts:61`) ; (b) les routes dédiées multi-assignation lisent et écrivent dans `TaskAssignee` (`src/app/api/tasks/[taskId]/assignees/route.ts:10,29,49`) **sans toucher à `assigneeId`** ; (c) les notifications de commentaire combinent les deux (`src/app/api/tasks/[taskId]/comments/route.ts:102,104` : `if (task.assigneeId && …) notifyIds.add(task.assigneeId)` puis `prisma.taskAssignee.findMany({ where: { taskId } })`). Conséquences directes : un utilisateur peut être (i) assigné via `assigneeId` mais absent de `TaskAssignee`, (ii) présent dans `TaskAssignee` mais pas dans `assigneeId`, (iii) dupliqué dans les deux (notifications doublées : `notifyIds.add` est un `Set`, donc les comments sont OK, mais d'autres flux peuvent dupliquer). Aucun trigger SQL, aucune `@@check` Prisma, aucun service applicatif central ne maintient la cohérence des deux sources.
- **Preuve** : `prisma/schema.prisma#Task` (déclaration `assigneeId` ligne ~239 et `assignees TaskAssignee[]` ligne ~256) ; `prisma/schema.prisma#TaskAssignee` ; annexe E §1.29 (note explicite : « ce modèle ouvre la voie aux multi-assignations en parallèle de la FK simple `Task.assigneeId` (cf. modèle `Task`). Cohabitation à recouper en phase 3 (axe BDD/LOG). ») ; `src/app/api/tasks/route.ts:115`, `:172` (création écrit uniquement `assigneeId`) ; `src/app/api/tasks/[taskId]/route.ts:209-241` (édition écrit uniquement `assigneeId`, déclenche notification) ; `src/app/api/tasks/[taskId]/assignees/route.ts:10,29,49` (CRUD multi-assignation indépendant) ; `src/app/api/tasks/[taskId]/comments/route.ts:100-105` (combinaison ad hoc) ; `src/types/index.ts:39,50,62,65` (types `assignee` (singulier) et `assignees?` (pluriel) coexistent).
- **Impact** : Risque d'intégrité métier et de double vérité — **Severity_High** (clause 7.1 + Property 7 sur double mécanisme contradictoire). (a) **Notifications incohérentes** : selon que l'on regarde `task.assigneeId` ou `task.assignees`, la liste des destinataires varie ; certains flux n'utilisent que l'un (cron `daily-tasks/route.ts` filtre `assigneeId: user.id`) tandis que d'autres combinent (`comments/route.ts`). Un utilisateur retiré de `assigneeId` mais oublié dans `TaskAssignee` continuera de recevoir des commentaires. (b) **Affichage incohérent** : les vues qui lisent `task.assignee` (singulier, pris depuis `assigneeId`) affichent une seule personne ; les vues qui lisent `task.assignees` (pluriel) affichent la liste — sans qu'aucune règle ne définisse laquelle fait foi. (c) **Filtres incohérents** : `src/app/api/tasks/route.ts:40` filtre par `where.assigneeId = assigneeId`, ce qui exclut les tâches où l'utilisateur n'est présent que dans `TaskAssignee` ; `src/app/api/my-tasks/route.ts:20` est dans le même cas. (d) **Duplication des données utilisateur** : si un utilisateur est dans les deux, `include: { assignee: true, assignees: { include: { user: true } } }` charge ses informations deux fois. (e) **Cascade incohérente** : `assigneeId` est `SetNull` à la suppression de l'utilisateur, alors que `TaskAssignee.user` est `Cascade` — la même suppression d'utilisateur produit une trace partielle (la tâche conserve sa valeur historique d'assigné nul, mais perd définitivement la trace dans `TaskAssignee`).
- **Recommandation** : Choisir une source de vérité unique et migrer. **Option A recommandée — Tout vers `TaskAssignee`** : retirer `Task.assigneeId` du schéma, migrer les valeurs existantes vers `TaskAssignee` (script de migration unique : `INSERT INTO "TaskAssignee" (task_id, user_id) SELECT id, assignee_id FROM "Task" WHERE assignee_id IS NOT NULL ON CONFLICT DO NOTHING;`), retirer `task.assignee` (singulier) des types, mettre à jour toutes les routes de création/édition/duplication pour écrire dans `TaskAssignee`, mettre à jour les filtres `where.assigneeId` en `where.assignees: { some: { userId } }`, mettre à jour l'UI pour afficher exclusivement `assignees`. **Option B — Tout vers `assigneeId`** : abandon de la multi-assignation, suppression de `TaskAssignee`. Option A préférable si la multi-assignation est valorisée par le produit (cas par défaut pour un clone ClickUp). À traiter en priorité dans la tâche de remédiation. À fusionner avec une analyse 7.3 (axe LOG) pour la cohérence des notifications.



## Tâche 8.2 — Index manquants sur champs filtrés / joints / triés

> **Périmètre** : confrontation des `@@index`, `@unique` et `@id` déclarés dans `prisma/schema.prisma` (cf. `notes/annexe-E-prisma.md`) aux usages effectifs dans `src/` (clauses `where`, `orderBy`, `include`, `groupBy`, `count`, `updateMany`, `deleteMany`, et requêtes d'enrichissement par jointure logique côté Prisma).
>
> **Méthodologie** : (i) énumération exhaustive des `findMany`/`findFirst`/`count`/`groupBy`/`aggregate`/`updateMany`/`deleteMany` par modèle via ripgrep ; (ii) extraction des champs présents en `where`, `orderBy`, `groupBy` et des jointures (`include`, `where: { relation: { ... } }`) ; (iii) confrontation à la liste des index Prisma de l'annexe E ; (iv) un champ est considéré « couvert » s'il est `@id`, `@unique`, ou colonne **de tête** d'un `@@index([a, b, ...])` ou `@@unique([a, b, ...])` (Postgres ne sert pas un index multi-colonnes pour une recherche sur une colonne non-leading) ; (v) sont exclus les modèles dont les indexes existants couvrent toutes les requêtes (`Budget`, `BudgetTransaction`, `FinanceTransaction`, `FinanceAccount`, `FinanceGoal`, `FinanceGoalContribution`, `FinanceCategory`, `Note`, `PushSubscription`, `BudgetTransactionTag`, `WorkspaceInvite`, `Tag`, `TaskTag`, `TaskAssignee`, `Status`, `Team`, `TeamMember`, `BudgetCategory`).
>
> **Critère de sévérité** : par défaut **Severity_Medium** pour toute table dont l'usage est avéré (clause 7.4 + Property 7 « requête N+1 ou balayage complet de table SHALL be ≥ Medium »). L'absence d'index sur des FK fortement filtrées équivaut, à mesure que la table grossit, à un balayage complet (`Seq Scan` Postgres) à chaque requête. Les passages en High sont réservés aux scans avérés en production (non observable en analyse statique pure).
>
> **Note importante sur les colonnes de tête** : par exemple, le `@@unique([taskId, tagId])` de `TaskTag` couvre la recherche `where: { taskId }` (taskId est en tête) mais NE couvre PAS `where: { tagId }` seul. De même, `WorkspaceMember.@@unique([workspaceId, userId])` couvre la lookup `findUnique({ where: { workspaceId_userId } })` mais une requête `members: { some: { userId } }` doit traverser l'index sur `userId`, qui n'existe pas — Postgres se rabat sur un `Seq Scan` ou utilise l'index unique en mode `Index Scan` partiel non optimal.

(a) Pour chaque ouverture d'une liste de tâches (vue principale de l'application, équivalent ClickUp Board/List view), Postgres effectue un **`Seq Scan`** sur toute la table `Task` filtrée par `listId` — coût linéaire en nombre total de tâches du workspace, acceptable jusqu'à quelques milliers, mais qui se dégrade rapidement. Tant que la base reste petite, l'impact reste contenu ; mais le pattern est généralisé et le modèle `Task` est destiné à grossir indéfiniment.
- **Recommandation** : Ajouter au minimum les `@@index` suivants sur le modèle `Task` (par ordre de priorité) : `@@index([listId, parentId])` (sert simultanément `where: { listId }` et `where: { listId, parentId: null }` de la vue Liste), `@@index([assigneeId, dueDate])` (sert « Mes tâches » et le rangement par dueDate), `@@index([dueDate])` (sert les crons et la vue calendrier sur l'ensemble des tâches), `@@index([statusId])` (sert le `groupBy` du dashboard et le `count` lors de la suppression de statut), `@@index([parentId])` (sert le listing des subtasks), `@@index([listId, statusId, position])` (sert le calcul de position en fin de liste lors de la création — colonne de tête `listId` + tri `position` ; gain énorme sur la création). À recouper avec la tâche 12.4 (axe PRF) pour confirmation par EXPLAIN ANALYZE en production. Toute action effective relève d'une tâche de remédiation post-audit.

### F-BDD-010 — Modèle `Notification` : aucun index alors que `userId`, `(userId, read)` et `(userId, type, createdAt)` sont les filtres permanents de l'UI bell-icon et des crons

À recouper avec une éventuelle politique de rétention (suppression des notifications de plus de 90 jours) qui réduirait également le besoin d'indexation.

Alternative plus économe : un seul index `@@index([completed, remindAt])` couvre les deux crons et le cas `upcoming` de l'UI. À traiter conjointement avec la tâche 12.4 (axe PRF) si une métrique de durée des crons est disponible côté Vercel.

### F-BDD-014 — Modèle `WorkspaceMember` : ordre des colonnes de `@@unique([workspaceId, userId])` non optimal — les requêtes de résolution d'ownership `members: { some: { userId } }` ne bénéficient pas de l'index

- **Axe** : BDD
- **Sévérité** : Medium
- **Description** : Le modèle `WorkspaceMember` (`prisma/schema.prisma#WorkspaceMember`, annexe E §1.6) déclare la contrainte `@@unique([workspaceId, userId])`. Dans l'**ordre Postgres**, l'index B-tree créé est utilisable pour : (a) recherche `(workspaceId, userId)` exacte (contrainte d'unicité, sert le `findUnique({ where: { workspaceId_userId } })` — utilisation très répandue, **OK**), (b) recherche `workspaceId` seul (préfixe leading column, **OK**). Il **N'EST PAS** utilisable efficacement pour : (c) recherche `userId` seul. Or **15+ routes API** font le pattern Prisma `where: { workspace: { members: { some: { userId } } } }` ou directement `where: { workspaceMember: { userId } }` pour matérialiser le contrôle d'ownership. Exemples : `src/app/api/budget/[budgetId]/transactions/route.ts:24` (`workspace: { members: { some: { userId } } }`), `src/app/api/budget/stats/route.ts:9`, `src/app/api/budget/route.ts:21` (helper `checkBudgetAccess`), `src/app/api/calendar/google/sync/route.ts:135-141`, `src/app/api/calendar/feed/[token]/route.ts:36-42`, `src/app/api/calendar/export/route.ts:18`, `src/app/api/finance/transactions/route.ts:23`, `src/app/api/finance/stats/route.ts:31,40`, `src/app/api/tasks/route.ts:131` (création de tâche), `src/app/api/tasks/[taskId]/comments/route.ts:60`, `src/app/api/tasks/[taskId]/checklists/route.ts:60`, `src/app/api/tasks/[taskId]/dependencies/route.ts:30,127`, `src/app/api/tasks/[taskId]/subtasks/route.ts:62`, `src/app/api/spaces/route.ts:78`, `src/app/api/folders/route.ts:30,77`, `src/app/api/lists/route.ts:78`, `src/app/api/lists/[listId]/statuses/route.ts:27` (helper `verifyListAccess`). Le `prisma.workspace.findMany({ where: { members: { some: { userId } } } })` de `src/app/api/workspaces/route.ts:13` (page d'accueil de l'application — premier appel après auth) est aussi concerné.
- **Preuve** : `prisma/schema.prisma#WorkspaceMember` (annexe E §1.6 : `@@unique([workspaceId, userId])`) ; `src/app/api/workspaces/route.ts:13` (probable `findMany` filtré par `members.some.userId`) ; `src/app/api/budget/route.ts:13-22` (helper `checkBudgetAccess`) ; `src/app/api/budget/[budgetId]/transactions/route.ts:21-29` ; `src/app/api/calendar/google/sync/route.ts:130-142` ; `src/app/api/calendar/feed/[token]/route.ts:34-46` ; `src/app/api/finance/transactions/route.ts:21-30` ; `src/app/api/tasks/route.ts:128-141` ; `src/app/api/spaces/route.ts:75-87` ; `src/app/api/folders/route.ts:25-36,73-86` ; `src/app/api/lists/[listId]/statuses/route.ts:25-39` ; `commande: grep -rn "members: { some: { userId" src/ --include="*.ts" \| wc -l` (≥ 15 occurrences attendues, à recouper en annexe A).
- **Impact** : À chaque requête API contenant un contrôle d'ownership via `members: { some: { userId } }`, Postgres traduit la sous-requête en `EXISTS (SELECT 1 FROM "WorkspaceMember" WHERE "workspaceId" = ... AND "userId" = ?)` — qui est sert par l'unique composite — **mais** lorsque le `workspaceId` n'est pas connu (`prisma.workspace.findMany({ where: { members: { some: { userId } } } })`, page d'accueil), Postgres doit chercher tous les `WorkspaceMember` d'un utilisateur sans index sur `userId` seul → **`Seq Scan` sur la table `WorkspaceMember` à chaque chargement de la page workspaces**. Cette table reste de taille modeste (typiquement quelques centaines à milliers de lignes pour une instance moyenne), donc l'impact opérationnel reste limité, mais le principe est doublement gênant : (a) c'est l'endpoint le plus chargé après auth, (b) le pattern `some: { userId }` revient sur **chaque vérification d'ownership** d'une route API tierce — multipliée par le volume d'appels global.
- **Recommandation** : Ajouter `@@index([userId])` séparément de l'unique composite : `@@index([userId])` sur `WorkspaceMember`. Postgres choisira automatiquement le meilleur index entre l'unique composite et le nouveau (selon que `workspaceId` est ou non dans le `WHERE`). Coût : un index B-tree supplémentaire (quelques Mo). Bénéfice : `prisma.workspace.findMany({ where: { members: { some: { userId } } } })` passe d'un `Seq Scan` à un `Index Scan`. À recouper avec la tâche 12.4 (axe PRF) sur la requête `/api/workspaces` exécutée à chaque chargement.

## Tâche 8.3 — Cascades et relations

> **Périmètre** : revue exhaustive des clauses `onDelete` / `onUpdate` des `@relation` du `prisma/schema.prisma` (49 modèles, ~70 relations matérialisées, cf. `notes/annexe-E-prisma.md`) au regard des flux métier de suppression effectivement déclenchés par l'application (`prisma.X.delete` dans `src/app/api/`, suppression NextAuth/PrismaAdapter, désinscription utilisateur, suppression de workspace/list/folder/space/task/budget/finance-account/finance-goal).
>
> **Méthodologie** : (i) énumération exhaustive des clauses `onDelete` via `grep -n "@relation\|onDelete" prisma/schema.prisma` (cf. annexe A) ; (ii) confrontation aux routes DELETE applicatives (`grep -rn "prisma\.<model>\.delete"`) ; (iii) identification des défauts Prisma implicites (absence de `onDelete` ⇒ `Restrict` côté FK) ; (iv) vérification de l'existence de garde-fous applicatifs avant suppression (par exemple `prisma.task.count` avant `prisma.status.delete`) ; (v) recoupement avec les Findings d'axe BDD préexistants (F-BDD-004, F-BDD-005, F-BDD-006 sur les FK logiques non matérialisées — non re-traités ici pour éviter les doublons).
>
> **Convention `onUpdate`** : Prisma applique par défaut `onUpdate: Cascade` côté Postgres ; aucune relation du schéma ne déclare `onUpdate` explicitement. Les `id` étant des `cuid()` immutables (jamais modifiés en pratique), `onUpdate: Cascade` est inopérant et n'a pas d'incidence sur les flux métier — non traité.
>
> **Critère de sévérité retenu** :
> - **Severity_High** quand la cascade (a) provoque une **destruction massive et silencieuse** de données métier sans confirmation (par exemple supprimer un compte financier détruit toutes ses transactions historiques sans avertissement), (b) introduit une **incohérence multi-source** (la même opération produit deux états différents selon le chemin de suppression emprunté), ou (c) déclenche un **hard-fail bloquant** sur un flux d'authentification ou de cycle de vie utilisateur (suppression de compte impossible).
> - **Severity_Medium** par défaut pour les inconsistances de cascade qui dégradent la maintenance ou créent des orphelins limités sans risque d'intégrité majeur (clause 7.1 + Property 3).

### F-BDD-016 — Quatre relations `creator: User` déclarées sans `onDelete` explicite (`Goal`, `Budget`, `BudgetTransaction`, `BudgetTemplate`) — fallback `Restrict` qui empêche toute suppression d'utilisateur

- **Preuve** : `prisma/schema.prisma:265` (`status Status @relation(fields: [statusId], references: [id], onDelete: Cascade)`) ; `src/app/api/lists/[listId]/statuses/route.ts:206-213` (garde applicative `prisma.task.count` puis 409) ; `src/app/api/lists/[listId]/statuses/route.ts:218` (`prisma.status.delete`) ; `commande: grep -rn "prisma\.status\.delete" /Users/zerbib/clickup-clone/src/ --include="*.ts"` (1 seule ligne, cette route — la garde couvre donc 100 % des suppressions actuelles).

### F-BDD-018 — Cascade `Folder → List` et `List → Folder` non cohérente avec l'attente UX (suppression d'un dossier détruit l'arborescence sans confirmation explicite côté API)

- **Axe** : BDD
- **Sévérité** : Medium

Action de remédiation à ouvrir séparément (post-audit). À recouper avec l'axe UX 6.x (confirmation des destructions massives) si le sujet y est ouvert.

### F-BDD-019 — Cascade `FinanceAccount → FinanceTransaction` et `Budget → BudgetTransaction` détruisent l'historique financier sans archivage (perte de données comptables)

- **Axe** : BDD
- **Sévérité** : High
- **Description** : Le modèle `FinanceTransaction` (`prisma/schema.prisma:835`) déclare `account FinanceAccount @relation(fields: [accountId], references: [id], onDelete: Cascade)` ; le modèle `BudgetTransaction` (`prisma/schema.prisma:694`) déclare `budget Budget @relation(fields: [budgetId], references: [id], onDelete: Cascade)`. Conséquence : supprimer un `FinanceAccount` (route `DELETE /api/finance/accounts/[accountId]`, `src/app/api/finance/accounts/[accountId]/route.ts:53-67`) **détruit irrémédiablement toutes les transactions historiques du compte** ; supprimer un `Budget` (route `DELETE /api/budget/[budgetId]`, `src/app/api/budget/[budgetId]/route.ts:94-111`) détruit toutes les transactions de ce budget. Couplé à F-BDD-005 (`FinanceTransaction.targetAccountId` sans `@relation`), le risque s'aggrave : les **transferts dont le compte cible est supprimé** continueront d'exister dans la base avec un `targetAccountId` orphelin (cf. Le contraste est notable avec `FinanceGoal` (`prisma/schema.prisma:866`) qui déclare `account FinanceAccount? @relation(... onDelete: SetNull)` : la suppression d'un compte préserve les goals (en perdant leur lien). La logique appliquée à `FinanceTransaction` aurait pu être identique (via un soft-delete `accountIsDeleted boolean` ou un `accountId String?` SetNull), mais l'option choisie est destructive. À noter que les `FinanceGoalContribution` sont en `Cascade` côté `goal` (`:883`) — comportement souhaitable cette fois (les contributions n'ont plus de sens sans goal).
- **Preuve** : `prisma/schema.prisma:694` (`Budget → BudgetTransaction Cascade`) ; `prisma/schema.prisma:835` (`FinanceAccount → FinanceTransaction Cascade`) ; `prisma/schema.prisma:866` (`FinanceAccount → FinanceGoal SetNull` — incohérent avec le précédent) ; `src/app/api/finance/accounts/[accountId]/route.ts:53-67` (DELETE direct, aucun `_count`, aucune confirmation) ; `src/app/api/budget/[budgetId]/route.ts:94-111` (idem) ; recoupement avec F-BDD-005 (orphelins `targetAccountId`).
- **Impact** : Severity_High. (a) **Perte de données financières historiques** : un utilisateur supprimant son compte « BNP » par erreur perd définitivement plusieurs années de transactions. Aucun mécanisme de récupération ; pas de table d'archive ; pas de soft-delete. Pour une application de gestion budgétaire personnelle (cf. README, INSTALL_MOBILE.md décrivant l'app comme un clone ClickUp + budgétisation perso), c'est une régression UX majeure par rapport aux apps comparables (Bankin', Linxo, YNAB préservent l'historique). (b) **Inconsistance avec `FinanceGoal`** : le même évènement « suppression d'un FinanceAccount » a deux comportements distincts (Cascade des transactions, SetNull des goals), trahissant une décision de design ad hoc plutôt qu'un choix systématique. (c) **Risque de fraude / litige** : en cas de litige fiscal ou de contestation de mouvement, l'historique a disparu — sans trace d'auditabilité. (d) **Renforcement de F-BDD-005** : les transferts entre comptes (où `targetAccountId` n'est pas une `@relation`) finissent dans un état partiellement orphelin, partiellement cascadé selon la direction.
- **Recommandation** : Choix de design à arbitrer avec le produit. **Option A (recommandée pour la finance personnelle)** : passer `BudgetTransaction.budget` et `FinanceTransaction.account` à `onDelete: Restrict` ; obliger l'utilisateur à archiver/transférer ses transactions avant de supprimer le compte/budget ; ajouter une route `POST /api/finance/accounts/[accountId]/archive` pour soft-delete (`isArchived Boolean @default(false)`). **Option B (cohérent mais destructif)** : conserver `Cascade` mais (i) ajouter un `_count.transactions` côté API DELETE, (ii) exiger un paramètre `?confirm=true` ou `?force=true`, (iii) afficher une dialog de confirmation explicite côté UI mentionnant le nombre de transactions à supprimer. Dans tous les cas, harmoniser avec `FinanceGoal` (qui est en `SetNull`) — l'incohérence actuelle est intenable. À traiter en priorité si la finance personnelle est dans la roadmap.

À fusionner avec une éventuelle remédiation « auditabilité » (axe LOG ou SEC). Ne pas confondre avec F-BDD-007 (cohabitation `assigneeId` / `TaskAssignee`) qui est de nature différente.

### F-BDD-021 — Asymétrie `Task.assignee → SetNull` versus `TaskAssignee.user → Cascade` : la même suppression d'utilisateur produit deux états distincts sur la même tâche

- **Axe** : BDD
- **Sévérité** : Medium
- **Description** : Le modèle `Task` déclare `assignee User? @relation("TaskAssignee", fields: [assigneeId], references: [id], onDelete: SetNull)` (`prisma/schema.prisma:266`) — la suppression d'un User remet `Task.assigneeId` à NULL. Le modèle `TaskAssignee` (table d'association multi-assignation) déclare `user User @relation("TaskAssignees", fields: [userId], references: [id], onDelete: Cascade)` (`prisma/schema.prisma:485`) — la suppression du même User détruit l'entrée de la table d'association. Conséquence directe : sur une **même tâche** où l'utilisateur supprimé était à la fois `assigneeId` (assignation principale) et présent dans `TaskAssignee` (multi-assignation), la suppression aboutit à un **état hybride** : (a) `Task.assigneeId = NULL` (trace conservée du fait qu'il y a eu un assigné, mais identité perdue) ; (b) ligne `TaskAssignee` correspondante détruite (trace disparaît totalement). Si on lit la tâche après suppression : (i) `task.assignee` est `null`, (ii) `task.assignees` est appauvri d'une entrée. Aucun chemin de récupération ne permet de reconstituer qui était assigné. Cette asymétrie aggrave F-BDD-007 (cohabitation `assigneeId`/`TaskAssignee` produisant deux sources de vérité parallèles) : non seulement les deux mécanismes ne sont pas synchronisés en lecture/écriture, mais leur **cycle de vie** sur la suppression d'utilisateur diverge, créant un troisième axe de désynchronisation. Le même effet existe pour `Task.creator … Cascade` (`:267`) versus `Task.assignee … SetNull` : supprimer le créateur d'une tâche **détruit la tâche entière** (et donc TaskAssignee, Comments, etc. en cascade transitive), alors que supprimer un assigné préserve la tâche mais perd l'identité — choix défendable mais qui mérite documentation.
- **Preuve** : `prisma/schema.prisma:266` (`Task.assignee … onDelete: SetNull`) ; `prisma/schema.prisma:485` (`TaskAssignee.user … onDelete: Cascade`) ; F-BDD-007 (preuves détaillées de la cohabitation des deux mécanismes) ; `prisma/schema.prisma:267` (`Task.creator … onDelete: Cascade` — comportement asymétrique avec assignee).
- **Impact** : Severity_Medium. (a) **Perte d'information asymétrique** : un audit a posteriori (« Qui était assigné à cette tâche avant la suppression du compte ? ») est partiellement répondable via `Task.assigneeId` historique (mais nul) et impossible via `TaskAssignee` (lignes effacées). (b) **Renforcement de la dette F-BDD-007** : les développeurs lisant les deux relations sans connaître ce détail produiront des bugs subtils (« Je vois que la tâche a 3 assignees mais l'`assigneeId` est null — pourquoi ? »). (c) Côté UI, les composants utilisant `task.assignees.length` afficheront une décrémentation silencieuse à chaque suppression d'utilisateur, sans message « X (compte supprimé) ».
- **Recommandation** : Décider d'une politique cohérente parallèlement à la résolution de F-BDD-007. **Option A (recommandée)** : si F-BDD-007 est résolu en supprimant `Task.assigneeId` au profit de `TaskAssignee` exclusif, la question disparaît mécaniquement. **Option B (palliative)** : aligner les deux comportements en passant `TaskAssignee.user` à `onDelete: SetNull` avec `userId String?` nullable — préserve l'historique des assignations dans la table d'association. **Option C** : passer les deux à `Cascade` — choix radical, supprime systématiquement les traces de l'utilisateur, à pondérer avec la perte d'historique. À recouper avec F-BDD-007 et F-BDD-016 (cohérence générale des `User` relations).

### F-BDD-023 — `vercel.json#buildCommand` exécute `prisma generate` mais omet `prisma migrate deploy` : régénération du client sans application des migrations en production

- **Axe** : BDD
- **Sévérité** : High
- **Description** : Le fichier `vercel.json:2` déclare `"buildCommand": "npx prisma generate && next build"`. Cette commande **régénère le client Prisma** (typescript bindings) à chaque build Vercel pour s'aligner sur l'état du `schema.prisma` du commit en cours, **mais elle n'exécute aucune migration en base** (`prisma migrate deploy` est absent, `prisma db push` également). Cette asymétrie crée un patron de défaillance silencieuse : si un développeur (i) modifie `schema.prisma` (ajoute un champ `Task.archivedAt DateTime?`), (ii) met à jour le code applicatif pour lire/écrire ce champ, (iii) commite et push, alors **le déploiement Vercel réussit** (`prisma generate` produit un client TypeScript qui connaît la nouvelle colonne, `next build` compile sans erreur), **mais la base Neon de production ne contient pas la colonne `archivedAt`** — au premier appel runtime à `prisma.task.findMany({ where: { archivedAt: null } })` ou à un `prisma.task.create({ data: { archivedAt: ... } })`, Postgres lèvera une erreur « column "archivedAt" does not exist » qui se manifestera en HTTP 500 côté API. Le développeur n'a aucun signal de pré-production : ni le build Vercel, ni l'inspection visuelle, ni les tests TypeScript ne le détectent. Couplé à F-BDD-022 (absence de dossier `prisma/migrations/` versionné), le problème est doublement aigu : il n'y a même pas de migration locale pour rappeler au développeur qu'il doit la déployer manuellement avant de push. Cette procédure manuelle est : (a) non documentée dans `CLAUDE.md` ni `README.md` ni dans aucun fichier `.md` du dépôt ; (b) sujette à des conditions de course (le commit git arrive en prod avant le `db push` → erreurs 500 pendant N minutes ; ou inversement, le `db push` arrive avant le commit → la base contient des colonnes que le code ne connaît pas) ; (c) impossible à reproduire sur staging/preview puisque les preview deployments Vercel pointent **par défaut sur la base de production** (à recouper en 9.x axe INF) sauf configuration explicite de `DATABASE_URL` par environnement.
- **Preuve** : `vercel.json:2` (`"buildCommand": "npx prisma generate && next build"` — pas de `migrate deploy`) ; `package.json:5-19` (`scripts` n'expose aucune commande `prisma migrate`/`db push`) ; `package.json:74-76` (`"prisma": { "seed": "..." }` — pas de `migrate.options`) ; `notes/inv-crons.md:14-15` (« `buildCommand` : `"npx prisma generate && next build"` — confirme l'exécution préalable de `prisma generate` à chaque build Vercel, cohérent avec la configuration Prisma observée en tâche 1.3 ») ; `notes/00-stack.md:26` (« Aucun script `prisma` dédié n'est exposé dans `scripts`. - **Impact** : Severity_High. (a) **Risque opérationnel** : tout déploiement modifiant le schéma a une **fenêtre d'erreur 500** entre le moment où Vercel sert le nouveau code et le moment où le développeur exécute manuellement `prisma db push` contre la base. Pour une app utilisée 24/7, c'est inacceptable même sur quelques secondes ; pour une app à charge faible, le risque reste qu'un développeur **oublie** d'exécuter `prisma db push` après un push git, et que les premières requêtes sur la nouvelle colonne échouent en silence (erreur 500 reportée à Sentry/Vercel logs mais pas bloquante au déploiement). (c) **Incohérence avec les preview deployments** : Vercel crée un preview deployment par PR ; chaque preview exécute `prisma generate && next build` puis sert le code, mais sans `migrate deploy`, les previews **lisent la base de production** (sauf configuration explicite de `DATABASE_URL` par environnement Vercel — non observable depuis le dépôt, à confirmer en 9.x axe INF) — donc une PR qui ajoute un champ déclenche les mêmes erreurs 500 sur la base de production avant même la merge. ### F-BDD-024 — Patron N+1 polymorphe sur `Favorite` : `Promise.all([fav.map → prisma.X.findUnique])` exécute 1 requête par favori (lookup polymorphe non batchable)

En période de stress (volume de budgets élevé, par exemple lors d'une bascule en début de mois où plusieurs budgets dépassent simultanément le seuil 100%), ce cron peut exécuter plusieurs centaines de requêtes Prisma. À noter par ailleurs que le `select` ligne 17 (`{ type: true, amount: true }`) limite intelligemment la projection sur `transactions`, mais cela **n'évite pas** le scan de toutes les transactions de tous les budgets.
- **Preuve** : `src/app/api/cron/budget-alerts/route.ts:16-25` (findMany sans `where:` ni `take:`, include profond avec sous-relations) ; `src/app/api/cron/budget-alerts/route.ts:29-72` (boucle imbriquée `for budget of budgets` × `for threshold of thresholds`) ; `src/app/api/cron/budget-alerts/route.ts:49-58` (`findFirst` notification par budget) ; `vercel.json:14-17` (cron programmé `0 10 * * *`) ; recoupement avec F-BDD-001 (modèle `BudgetAlert` mort, seuils codés en dur ligne 38-41) ; recoupement avec F-BDD-010 (table `Notification` sans index sur `(userId, type, createdAt)`, donc le `findFirst` de déduplication scanne aussi).
- **Impact** : Severity_High. (a) Le cron est **quotidien** et son coût compute Vercel croît avec le carré du nombre de budgets (chargement initial × nombre de budgets dépassant un seuil). (b) Le pattern « charger tout puis itérer » est l'anti-pattern N+1 canonique : la même opération aurait pu être réalisée en SQL pur via une seule requête de type `SELECT b.*, COALESCE(SUM(...), 0) FROM Budget b LEFT JOIN BudgetTransaction t ... (c) Couplé à F-BDD-019 (cascade `Budget → BudgetTransaction Cascade`, perte de données financières) et F-BDD-010 (Notification sans index), le cron est un poste de coût non négligeable et **non observé** (aucun log de durée n'est exporté). (d) Si la base contient des budgets de workspaces partagés ou abandonnés (typique d'un environnement multi-tenant qui accumule), le cron continue de les scanner indéfiniment.
- **Recommandation** : Refactoriser en deux temps. **Étape 1 (rapide)** : ajouter un `where:` sur `prisma.budget.findMany` pour ne traiter que les budgets actifs (par exemple `where: { workspace: { members: { some: {} } } }` pour exclure les workspaces vidés) et un agrégat côté SQL via `prisma.budgetTransaction.groupBy({ by: ["budgetId"], where: { type: "expense" }, _sum: { amount: true } })` qui retourne directement la dépense par budget en **une seule** requête. **Étape 2 (long terme)** : matérialiser un `Budget.spentAmount Float @default(0)` mis à jour atomiquement à chaque écriture/suppression de `BudgetTransaction` (via trigger Postgres ou middleware Prisma), ce qui transforme le scan en simple `findMany({ where: { spentAmount: { gte: amount * 0.8 } } })`. À fusionner avec une éventuelle remédiation cron globale (axe LOG/PRF). Pour la déduplication : remplacer le `findFirst` par un `findMany({ where: { userId: { in: memberIds }, ... } })` unique par budget (1 requête au lieu de 1).

Coût : 1 requête au lieu de M × T. **(iii) Ajouter un index `Notification(userId, type, link, createdAt)`** (déjà recommandé F-BDD-010) pour servir le `WHERE IN`. À fusionner avec F-BDD-025 dans une remédiation « optimisation des crons ».

- **Description** : Inventaire systématique des `findMany` sans `take:` sur tables à cardinalité non bornée par item parent ou par utilisateur. (1) **`/api/time-entries` GET** (`src/app/api/time-entries/route.ts:34-46`) : `findMany` filtré par `taskId?`/`userId?`/`startTime { gte, lte }?` — si l'utilisateur passe uniquement `userId`, c'est tout l'historique de pointage ; même filtré par `taskId`, une tâche très active peut accumuler plusieurs centaines d'entrées. **Aucun `take:`**. (2) **`/api/time-entries/report` GET** (`src/app/api/time-entries/report/route.ts:44-54`) : `findMany` filtré par `workspaceId?` + plage de dates — si la plage est large (« annuel »), c'est l'intégralité du pointage du workspace. **Aucun `take:`**. (3) **`/api/notes` GET** (`src/app/api/notes/route.ts:14-17`) : `findMany` filtré par `(userId, workspaceId)` + tri par `pinned/updatedAt`. **Aucun `take:`** — un utilisateur avec 1000 notes paie un scan complet. (4) **`/api/templates` GET** (`src/app/api/templates/route.ts:18-26`) : `findMany` filtré par `workspaceId`. **Aucun `take:`**. (5) **`/api/finance/transactions` GET** (`src/app/api/finance/transactions/route.ts:37-41`) : `findMany` filtré par `(userId, account.workspaceId, ...)`. **Aucun `take:`**. (6) **`/api/budget/[id]/transactions` GET** (`src/app/api/budget/[budgetId]/transactions/route.ts:91-101`) : `findMany` filtré par `budgetId` + filtres optionnels. **Aucun `take:`**. (7) **`/api/tasks/[taskId]/comments` GET** (`src/app/api/tasks/[taskId]/comments/route.ts:20-27`) : `findMany` par `taskId`. Cardinalité bornée par tâche, mais dans le cas pathologique d'une tâche-discussion à 500+ commentaires, le scan reste préoccupant. **Aucun `take:`**. (8) **`/api/tasks/[taskId]/activity` GET** (`src/app/api/tasks/[taskId]/activity/route.ts:18-25`) : idem, `findMany` par `taskId`, sans `take:`. (9) **`/api/tasks/[taskId]/attachments` GET** (`src/app/api/tasks/[taskId]/attachments/route.ts:10-14`) : idem. (10) **`/api/tasks/[taskId]/dependencies` GET** (`src/app/api/tasks/[taskId]/dependencies/route.ts:51,61`) : 2 `findMany` symétriques par `dependentTaskId`/`dependencyTaskId`, sans `take:` (déjà mentionné en F-BDD-015 pour l'index, ici flaggé pour le `take:`). (11) **`/api/tasks/[taskId]/checklists` GET** (`src/app/api/tasks/[taskId]/checklists/route.ts:23-29`) : idem. (12) **`/api/workspaces/[id]/lists` GET** (`src/app/api/workspaces/[workspaceId]/lists/route.ts:25`) : `findMany` Lists par workspace — sans `take:`. (13) **`/api/finance/goals` GET** : sans `take:`. (14) **`/api/finance/accounts` GET** : `findMany` filtré par `(workspaceId, userId)` — pas de `take:`, mais cardinalité naturellement bornée à ~10 comptes par utilisateur typiquement, donc impact mineur. (15) **`/api/finance/categories` GET** : idem, cardinalité naturellement bornée (~20 catégories), impact mineur. (16) **`/api/budget/categories` GET** : idem. (17) **`/api/budget` GET** : déjà flaggé en F-BDD-029 (cumul include + pas de take). (18) **`/api/goals` GET** : `findMany` par `workspaceId` + `include: { creator, targets }`, sans `take:`. La majorité de ces endpoints sont consommés par des hooks SWR et appelés à chaque montage de la page concernée — sur une session active, le coût aggrégé est important.
- **Preuve** : Voir références ligne par ligne ci-dessus, exhaustif sur la base de l'énumération `commande: grep -rn "take:" src/ --include="*.ts"` (9 occurrences ramenées contre 62 `findMany` au total — 53 `findMany` sont sans `take:`, dont les sites identifiés ci-dessus). Les sites non-flaggés sont ceux où le `where:` est sélectif par construction (par exemple `prisma.tag.findMany({ where: { workspaceId } })` — cardinalité naturellement bornée à quelques dizaines par workspace, hors scope).
- **Impact** : Severity_Medium par défaut (clause 7.2 + Property 7). Le coût réel dépend de la cardinalité effective de chaque table en production — invisible en analyse statique. Pour les tables historiques (`TimeEntry`, `Activity`, `Comment`, `BudgetTransaction`, `FinanceTransaction`), le risque est croissant et inévitable. Pour les autres (`Note`, `Goal`, `TaskTemplate`), l'impact reste contenu mais le pattern est uniformément absent. Dégradation UX progressive (latence d'ouverture des écrans qui augmente avec l'usage) plus risque OOM serverless en pic d'activité multi-utilisateur.
- **Recommandation** : Politique de pagination systématique. **(i) Pour les listings UX (commentaires, activités, attachments, time-entries, transactions)** : ajouter `take: 100` + `cursor`-based pagination via le champ `id` ou `createdAt`. Côté UI, implémenter un « charger plus » ou un infinite-scroll. **(ii) Pour les listings de configuration (notes, templates, goals)** : `take: 200` simple suffit comme garde-fou (au-delà, l'UI a de toute façon besoin d'un mode recherche/filtre). **(iii) Pour les rapports agrégés (`/api/budget/stats`, `/api/finance/stats`, `/api/dashboard/stats`)** : préférer `groupBy` côté Prisma plutôt que `findMany` + agrégation JS (cf. F-BDD-029 et F-BDD-026 pour les patrons). **(iv)** Documenter dans `CLAUDE.md` une règle « tout `findMany` doit avoir un `take:` ou un commentaire justifiant son absence ». À fusionner dans une remédiation « pagination généralisée des endpoints listings ».



## Tâche 8.6 — Usages de SQL brut (`$queryRaw`, `$executeRaw`)

> **Périmètre** : recherche exhaustive, en lecture seule, des usages applicatifs de SQL brut (`prisma.$queryRaw`, `prisma.$queryRawUnsafe`, `prisma.$executeRaw`, `prisma.$executeRawUnsafe`) sous `src/` et `prisma/`. Pour chaque occurrence, l'audit doit distinguer (i) tagged template (paramétrage sûr via `Prisma.sql`/template literal), (ii) variante `Unsafe` ou concaténation directe d'entrée utilisateur (interpolation non sûre — Severity_High par clause 7.5 de `requirements.md`), (iii) usage avec validation Zod préalable. La revue couvre également les helpers Prisma associés (`Prisma.sql\`...\``, `Prisma.raw(...)`, `Prisma.join(...)`, `Prisma.empty`) ainsi que les chemins d'exécution SQL alternatifs (clients `pg`/`postgres`, `neon-serverless`, `drizzle`, `kysely`, appels `pool.query(...)` ou `client.query(...)`) afin de garantir l'exhaustivité.
>
> **Méthodologie** : (i) `grep -rnE "\$queryRaw|\$queryRawUnsafe|\$executeRaw|\$executeRawUnsafe" src/ prisma/` ; (ii) `grep -rnE "Prisma\.sql|Prisma\.raw|Prisma\.empty|Prisma\.join" src/ prisma/` ; (iii) recherche large des autres clients SQL (`grep -rnE "from ['\"]pg['\"]|require\(['\"]pg['\"]\)|new Pool\(|new Client\(|\.unsafe\(" src/ prisma/`) ; (iv) recherche des chemins d'exécution alternatifs (`grep -rnE "from ['\"]@neondatabase|drizzle|postgres-js|kysely|\.query\(['\"]?SELECT|pool\.query|client\.query" src/ prisma/`) ; (v) inventaire complémentaire de toutes les occurrences `prisma.$<...>` pour confirmer le périmètre (résultat : seulement `prisma.$transaction`, qui est l'API ORM officielle de Prisma — non concerné par 7.5). Toutes les commandes sont consignées dans `notes/annexe-A-commandes.md` (Tâche 8.6).
>
> **Critères de sévérité** (rappel `requirements.md` 7.5 + Property 7) : interpolation directe d'entrée utilisateur dans une chaîne SQL → **Severity_High** (risque d'injection SQL avéré) ; usage `*Unsafe` même avec validation Zod stricte → **Severity_Medium** (le contournement de la sécurité par défaut de Prisma exige une justification documentée) ; usage `$queryRaw` en tagged template avec entrées paramétrées (placeholders `${value}` interpolés par Prisma comme paramètres SQL préparés) et validation Zod → **conforme**, pas de Finding.

### Résultat de la revue

**Aucun usage applicatif de SQL brut détecté.** Les quatre commandes ciblées (`$queryRaw`, `$queryRawUnsafe`, `$executeRaw`, `$executeRawUnsafe`) retournent **exit code 1, zéro ligne** sur l'ensemble de `src/` et `prisma/` (cf. annexe A, commande Tâche 8.6 #1). Les recherches élargies sur `Prisma.sql` / `Prisma.raw` / `Prisma.empty` / `Prisma.join` retournent également zéro occurrence (annexe A #2). L'inventaire de tous les `prisma.$<...>` ne ramène que **quatre usages de `prisma.$transaction`** (`src/app/api/tasks/reorder/route.ts:35`, `src/app/api/finance/goals/[goalId]/contribute/route.ts:19`, `src/app/api/finance/transactions/route.ts:62`, `src/app/api/invites/[token]/route.ts:87`), qui relèvent de l'API ORM officielle Prisma (transactions atomiques composées via le `Client` typé) et n'exécutent **aucun SQL textuel** : les opérations imbriquées sont toutes des appels Prisma typés (`tx.task.update`, `tx.financeAccount.update`, etc.). Cet inventaire est hors périmètre 7.5 et est documenté pour traçabilité (cf. annexe A #3).

**Présence d'un client `pg.Pool` — analyse contextuelle.** La recherche large des clients SQL alternatifs (annexe A #4) ramène **deux occurrences** dans `src/lib/prisma.ts` :
- `src/lib/prisma.ts:3` : `import { Pool } from "pg"` ;
- `src/lib/prisma.ts:11` : `const pool = new Pool({ connectionString })`.

Lecture du fichier (`src/lib/prisma.ts:1-21`) : ce `Pool` est **exclusivement** passé à `new PrismaPg(pool)` (ligne 12) puis à `new PrismaClient({ adapter })` (ligne 13). Il s'agit du **driver adapter officiel `@prisma/adapter-pg`** (cf. https://www.prisma.io/docs/orm/overview/databases/database-drivers#driver-adapters), choisi vraisemblablement pour la compatibilité avec Neon / WebSocket pooling et le runtime serverless Vercel. Toute requête SQL effective est **toujours générée par le query engine Prisma** (paramétrée par construction, contrôles de typage, échappement), puis transmise au pool `pg` qui ne fait que router les paquets binaires. **Aucun appel `pool.query(...)` ou `client.query(...)` direct** n'existe dans `src/` ni `prisma/` (annexe A #5, exit code 1, zéro ligne) — autrement dit, le `Pool` n'est jamais utilisé en dehors de l'adapter Prisma. La présence du `pg.Pool` ne constitue donc **pas** un contournement du chemin paramétré et n'ouvre pas de surface d'injection. Aucun Finding à ouvrir au titre de 7.5.

**Aucun autre chemin d'exécution SQL** (annexe A #6) : pas d'import `@neondatabase/serverless`, ni `drizzle-orm`, ni `postgres` (postgres-js), ni `kysely`, ni de littéral `pool.query("SELECT ...")` ou `client.query("SELECT ...")` dans `src/` ou `prisma/`. Le seul ORM consommateur de la base est Prisma, par le client typé.

### Verdict

**Aucun Finding pour cet axe — aucune utilisation de SQL brut détectée dans `src/` et `prisma/`.** La couche d'accès aux données est intégralement médiée par Prisma (client typé + `$transaction` composé), avec un driver adapter `@prisma/adapter-pg` qui préserve la paramétrisation native du query engine. Le risque d'injection SQL au sens de 7.5 (interpolation non sûre dans `$queryRaw`/`$executeRaw`) est, au moment de l'audit, **structurellement inexistant**.

> **Note de traçabilité** : ce constat est strict au périmètre `src/` + `prisma/` (cf. `requirements.md` 4.4). Il **ne couvre pas** un éventuel SQL embarqué dans des outillages hors périmètre (scripts ad hoc, fichiers `.sql` dans `scripts/backup-db.sh`, fonctions SQL côté Postgres telles que des triggers ou functions dans la base Neon). En particulier, si une migration future introduit une fonctionnalité de recherche full-text, de tri lexical avec `Prisma.sql\`ORDER BY ${column}\``, ou un `groupBy` polymorphe via `$queryRaw`, le présent verdict devra être réexaminé. À recouper avec F-BDD-022 (absence de migrations versionnées) si un futur déploiement introduit de tels artefacts SQL — ils devront alors être consignés dans `prisma/migrations/` et soumis à une revue paramétrisation.
